# 技术设计

## 上下文

- 当前状态：
  - `DocWeave` 已经通过 `docs/decisions/`、`docs/architecture/` 和 `docs/planning/` 明确了第一阶段技术基线、运行时拆分和 Monorepo 目录目标。
  - 仓库当前仍然只有文档与 `spec-superflow` 工作流骨架，还没有真实的工程目录、根级工作区配置或各应用/共享包的初始化入口。
  - 后续功能开发如果直接开始写某个应用，会在包管理、目录结构、共享配置、服务边界和本地基础设施约定上产生重复决策。
- 约束条件：
  - 必须遵守已确认的 phase-1 边界：`apps/web`、`apps/api`、`apps/collab`、`apps/worker` 分离；`packages/ai` 与 `packages/rag` 职责分离；RAG 只消费稳定快照。
  - 本次变更只做工程初始化，不提前实现业务页面、业务接口、协同协议细节、RAG 处理链或生产部署体系。
  - 不在本次变更中引入 `BullMQ`、`assistant-ui`、复杂导出任务、过度微服务拆分或未被文档批准的新增运行时。
  - 初始化结果必须能作为后续各应用和包的稳定落脚点，避免很快推翻根结构。
- 利益相关者：
  - 后续负责实现 `apps/web` 的前端开发者。
  - 后续负责实现 `apps/api`、`apps/collab`、`apps/worker` 的后端开发者。
  - 需要在同一仓库中维护共享包和本地依赖环境的项目维护者。

## 目标

- 建立一个与既有文档完全对齐的 Monorepo 根结构。
- 为四个 phase-1 应用和九个共享包提供明确且稳定的目录边界。
- 提供根级工作区配置、共享 TypeScript 基线和统一脚本入口，降低后续初始化成本。
- 为 PostgreSQL、Redis、Qdrant、MinIO 的本地开发入口预留统一位置。
- 让后续应用初始化可以增量推进，而不需要回头重构仓库根结构。

## 非目标

- 不在本设计中确定各应用内部的全部业务模块划分。
- 不在本设计中落地完整 CI/CD、生产部署编排或云环境适配。
- 不在本设计中实现认证、权限、文件上传、Presence、编辑器 AI、RAG 管线等专项能力。
- 不在本设计中定义所有代码规范细节或所有子包的公开 API。

## 决策

### 决策 1：采用单仓 Monorepo 根壳，先固定顶层边界再初始化子应用

- **选择**：
  - 在仓库根目录先建立 `apps/`、`packages/`、`infrastructure/`、工作区清单、根 `package.json` 与共享 `tsconfig` 等基础外壳。
  - 后续子应用和共享包基于这层根壳逐步初始化，而不是先单独创建某个 app 再反推工作区结构。
- **理由**：
  - 当前文档已经先于代码给出了完整的运行时拆分和目录目标，先固定根壳能把这些结论直接编码到仓库结构中。
  - 这样能避免 `apps/web` 或 `apps/api` 先行落地时，把自己的局部需求误写成全局约定。
  - 根壳先稳定后，后续每个子目录都能围绕统一 workspace 基线推进，实现成本更低，返工概率更小。
- **考虑的替代方案**：
  - 先初始化 `apps/web` 再补 Monorepo：短期更快，但容易让前端局部决策倒灌到整个仓库。
  - 先初始化 `apps/api` 再围绕后端组织仓库：会削弱多运行时分层，不符合当前已确认架构。

### 决策 2：目录结构严格对齐 phase-1 文档，不做额外运行时扩展

- **选择**：
  - `apps/` 仅包含 `web`、`api`、`collab`、`worker`。
  - `packages/` 仅包含已在规划中确认的 `shared`、`auth`、`database`、`editor`、`ai`、`rag`、`collaboration`、`ui`、`config`。
  - `infrastructure/` 仅为 `postgres`、`redis`、`qdrant`、`minio` 预留入口。
- **理由**：
  - 本次变更的目标是把已经拍板的边界落地，而不是继续做技术探索。
  - 严格对齐文档能让未来判断“目录是不是跑偏了”变得非常直接。
  - 过早加入未确认目录，容易把第二阶段能力伪装成第一阶段基线，增加维护噪音。
- **考虑的替代方案**：
  - 预先增加更多候选目录，如 `apps/gateway`、`packages/queue`、`packages/search`：扩展性看似更强，但会提前固化尚未批准的架构。
  - 先只建一部分目录，后续按需补：初期更省事，但会削弱“工程基线一次性定清”的价值。

### 决策 3：共享配置先上收至根工作区，子应用按继承方式消费

- **选择**：
  - 在仓库根目录提供工作区管理、共享 TypeScript 基线与公共脚本入口。
  - 各 app/package 初始化时默认继承根配置，而不是各自复制一套基础配置。
- **理由**：
  - Monorepo 的核心收益之一就是共享约定，如果初始化阶段就让每个目录各自为政，后续会很快出现漂移。
  - 共享根配置更符合本次变更“先建立可持续扩展的底座”的目标。
  - 后续无论新增前端包、后端包还是 worker 相关包，都能在同一根基线上演进。
- **考虑的替代方案**：
  - 每个 app/package 独立维护基础配置：隔离性强，但重复度高，后续统一升级困难。
  - 完全不做根级共享配置，只保留目录：初期最轻，但不能真正承担“初始化基线”的作用。

### 决策 4：本地基础设施只提供开发入口，不在本次变更承诺生产编排

- **选择**：
  - 为 `postgres`、`redis`、`qdrant`、`minio` 提供本地开发入口目录或 starter 配置位置。
  - 不在本次变更中完成完整生产部署模板、云厂商差异适配或集群级编排。
- **理由**：
  - 第一阶段当前最需要的是让开发者能在本地跑通基础依赖，而不是先把生产平台设计做满。
  - 这与现有文档“先建立最小闭环，再扩展部署与调度能力”的策略一致。
  - 把本地入口与生产编排分开，有助于减少初始化阶段的复杂度和误导。
- **考虑的替代方案**：
  - 直接把 Docker Compose、Kubernetes、云资源清单一起做满：覆盖更广，但明显超出本次初始化范围。
  - 完全不碰基础设施目录，等业务代码需要时再补：会延迟后续本地联调准备，降低工程壳价值。

## 风险与权衡

- 风险：目录和根配置虽然建立了，但某些后续子应用真实初始化时可能发现还缺少个别共享约定。
  - 缓解措施：本次设计把重点放在“稳定边界”和“共享入口”，允许在后续 app/package 初始化变更中增量补充局部配置，而不推翻根结构。
- 风险：初始化时过度抽象，生成很多空壳目录，短期看起来“代码不多但结构很大”。
  - 缓解措施：仅落地已被文档批准的 phase-1 目录，不额外扩充未知能力目录。
- 风险：根级共享配置如果设计过重，后续会拖慢不同运行时的差异化初始化。
  - 缓解措施：根工作区只承载真正共享的基础约定，把 app 级差异化配置留到各自目录中处理。
- 风险：开发者可能误以为 `infrastructure/` 目录已经代表完整部署方案。
  - 缓解措施：在目录和文档中明确这是本地开发基线入口，而不是生产级编排承诺。

## 迁移计划

- 上线步骤：
  - 创建 Monorepo 顶层目录骨架。
  - 写入根工作区配置与共享 TypeScript 基线。
  - 创建 phase-1 `apps/*` 与 `packages/*` 目录基线。
  - 创建 `infrastructure/*` 本地开发入口目录或 starter 配置。
  - 通过后续变更逐步初始化各应用和包的真实实现。
- 回滚步骤：
  - 如果本次初始化结构被判定为与规划文档冲突，则回退新增目录与根配置文件。
  - 保留 `changes/chore-monorepo-bootstrap/` 工件，基于修订后的 proposal/specs/design/tasks 重新生成初始化方案。
