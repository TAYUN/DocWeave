# 技术设计

## 上下文

- 当前状态：
  - 工作台、空间、文档元数据与 CRUD 已经形成闭环。
  - 文档页当前仍是元数据详情 + summary 编辑，没有正文编辑器。
  - 技术决策已经明确 `BlockNote` 是正文编辑的正式基线。
- 约束条件：
  - 当前 change 只做单人编辑，不引入协同、AI 或稳定快照。
  - 编辑器集成要尽量贴近 BlockNote 官方包，不先自造协议层。
  - 需要为后续协同与 AI 接入保留扩展空间，而不是把当前单人实现写死。

## 目标

- 在文档页真正渲染 BlockNote 编辑器
- 支持读取与保存单人正文内容
- 建立 `packages/editor` 的最小共享边界

## 非目标

- 不做协同
- 不做 AI
- 不做快照与索引
- 不做复杂版本管理

## 决策

### 决策 1：先落单人 BlockNote，再扩展协同

- **选择**：
  - 当前只接入 BlockNote 的单人编辑能力。
- **理由**：
  - 这和路线图里 “M2 先单人编辑，M3 再协同” 的顺序一致。

### 决策 2：正文保存先走当前 API 和 PostgreSQL 基线

- **选择**：
  - 当前正文数据继续由 `apps/api` 承接，并持久化到 PostgreSQL 相关模型或字段。
- **理由**：
  - 这能让单人编辑先成为真实闭环，不必等协同态和快照态设计完全到位。

### 决策 3：用 `packages/editor` 承接最小封装

- **选择**：
  - 不把 BlockNote 直接散落在 `apps/web` 多处，而是建立一个最小共享封装入口。
- **理由**：
  - 这样后续接协同、AI、Slash 菜单时更容易扩展。

## 风险与权衡

- 风险：当前单人内容存储方案未来可能需要为快照/协同重构。
  - 缓解措施：当前只建立最小承接位，避免过度设计正文数据模型。
- 风险：BlockNote 接入后样式和交互复杂度提升，可能暴露更多前端基础设施缺口。
  - 缓解措施：本 change 只保证编辑器能加载、编辑、保存，不追求完整产品体验。

## 迁移计划

- 上线步骤：
  - 引入 BlockNote 依赖
  - 建立 `packages/editor` 最小入口
  - 文档页渲染单人编辑器
  - 打通内容读取与保存
- 回滚步骤：
  - 如正文持久化边界不稳定，则回退 BlockNote 集成，保留当前元数据页壳
