# 技术设计

## 上下文

- 当前状态：
  - `apps/web` 已具备 overview、space、document 路由和基于 Query 的真实读取链路。
  - `apps/api` 已具备 `spaces/documents` 的真实读接口与 PostgreSQL 基线。
  - 目前最大的缺口不是“还能不能继续读数据”，而是“用户能不能真正改数据”。
- 约束条件：
  - 当前阶段仍然属于 M2 之前后的元数据基础闭环，不应提前把 BlockNote、协同、权限体系一起并入。
  - 写入流程必须复用既有 `spaces/documents` 数据模型，避免新建另一套临时表或临时 contract。
  - 表单和接口只需要最小可用，不追求复杂体验。

## 目标

- 让用户能创建 `space`
- 让用户能在某个 `space` 下创建 `document`
- 让用户能编辑 `document summary`
- 让前端能在 mutation 成功后刷新并反映数据库真实状态

## 非目标

- 不实现删除、归档、移动、排序、树拖拽
- 不实现协同、BlockNote 正文、搜索、RAG、AI 写入
- 不实现正式认证和权限校验

## 决策

### 决策 1：由 web 表单驱动最小写链路，而不是只补 API

- **选择**：
  - 同时实现写接口与对应表单。
- **理由**：
  - 当前阶段真正的价值是拿到“可交互闭环”，而不是只多几个无法在页面触达的后端接口。

### 决策 2：文档写入只开放元数据字段，不触碰正文结构

- **选择**：
  - 当前只开放 `title` 与 `summary` 级别的元数据编辑。
- **理由**：
  - 正文编辑应与 BlockNote 接入绑定，属于后续 change。

### 决策 3：Mutation 成功后使用 Query 刷新，而不是引入更重的状态管理

- **选择**：
  - 使用 TanStack Query 的 invalidation/refresh 驱动 UI 同步。
- **理由**：
  - 当前规模下这是最简单且与现有骨架最一致的方案。

## 风险与权衡

- 风险：没有认证时，所有写接口都是本地开发态能力，不能代表生产安全模型。
  - 缓解措施：明确这是本地 foundation 流程，只服务当前阶段闭环验证。
- 风险：如果表单做得过重，会让当前 change 再次膨胀。
  - 缓解措施：严格限制在 create space、create document、edit summary 三条路径。

## 迁移计划

- 上线步骤：
  - 新增写接口
  - 新增前端表单
  - 接入 mutation 刷新
  - 跑本地手动闭环验证
- 回滚步骤：
  - 如发现写入接口边界不稳定，则回退表单与 mutation，保留只读骨架
