# 20. AI 模块设计

阅读说明：

1. 本文档负责定义 DocWeave 中 `packages/ai`、编辑器 AI、聊天 / Copilot 与 RAG 问答之间的模块边界。
2. AI SDK 与交互层基线分别以 [20. AI SDK 与模型接入策略](../decisions/20.%20AI%20SDK%20与模型接入策略.md) 和 [21. 聊天与 Copilot 交互层策略](../decisions/21.%20聊天与%20Copilot%20交互层策略.md) 为准。
3. BlockNote 服务端能力接入以 [22. BlockNote 服务端能力接入设计](./22.%20BlockNote%20服务端能力接入设计.md) 为准。

## 1. 文档目的

本文档用于把 DocWeave 中所有 AI 相关能力收口成一套模块边界，避免后续出现编辑器 AI、聊天页、RAG 问答、Prompt 拼接和模型调用都各写一套实现，导致能力复用和权限边界逐渐失控。

## 2. 核心结论

DocWeave 的 AI 模块建议按以下方式分层：

1. `packages/ai`
   - 统一模型接入与流式运行时
2. `apps/api`
   - 统一业务权限、上下文组装与接口宿主
3. `@blocknote/xl-ai/server`
   - 承接编辑器 AI 官方协议层
4. `packages/rag`
   - 负责检索、上下文构建和问答流程，不负责通用模型接入

## 3. 三类 AI 场景

当前项目至少有三类 AI 场景：

1. 编辑器内 AI
2. 文档 Copilot / 聊天
3. 知识库 RAG 问答

这三类场景可以共用模型层，但不应混成同一个产品交互壳或同一条上下文构建链路。

## 4. `packages/ai` 的职责

建议 `packages/ai` 负责：

1. `Vercel AI SDK` 接入
2. provider / model 配置
3. 流式输出与错误收敛
4. Prompt helper 与 runtime helper
5. 为编辑器 AI 与聊天问答提供统一模型层

不负责：

1. 直接查数据库
2. 自己判定业务权限
3. 直接做向量检索

## 5. `apps/api` 的职责

AI 相关接口应统一由 `apps/api` 宿主承担：

1. 读取当前会话
2. 校验文档与工作区权限
3. 组装上下文
4. 选择调用编辑器 AI 或 RAG 问答链路
5. 返回流式响应

## 6. 编辑器 AI 链路

推荐链路如下：

```text
apps/web
  -> @blocknote/xl-ai
  -> apps/api hosted @blocknote/xl-ai/server
  -> permission check
  -> packages/ai runtime
```

编辑器 AI 的特点如下：

1. 上下文通常来自当前选区与当前文档
2. 输出是局部改写或局部内容生成
3. 需要支持“应用 / 放弃结果”

## 7. 聊天 / Copilot 链路

推荐链路如下：

```text
apps/web custom lightweight chat ui
  -> apps/api ai endpoints
  -> packages/ai runtime
```

聊天 / Copilot 的特点如下：

1. 更像多轮消息交互
2. 更依赖会话状态、引用和错误态展示
3. 后续才视产品稳定度评估 `assistant-ui`

## 8. RAG 问答链路

推荐链路如下：

```text
apps/web
  -> apps/api rag/chat
  -> apps/api resolve visible scope
  -> packages/rag retrieve + build context
  -> packages/ai answer
```

关键边界如下：

1. `packages/rag` 负责检索与上下文构建
2. `packages/ai` 负责模型回答
3. 两者不要互相越权吞掉对方职责

## 9. Prompt 与上下文组织

建议尽量把 Prompt 和上下文分层管理：

1. 模型运行时辅助在 `packages/ai`
2. 场景级 Prompt builder 按领域分目录
3. RAG 上下文构建在 `packages/rag`
4. 编辑器局部上下文编排由编辑器 AI 链路单独维护

## 10. 模块间共享对象

建议统一共享以下对象语义：

1. `packages/contracts/ai.ts` 负责模型调用层的 message、usage 和 model contract。
2. `packages/contracts/rag.ts` 负责 RAG endpoint 的 request / response、Citation 和 SSE event contract。
3. RAG 回答正文使用 `[citationId]` 关联 `citations[]`，流式过程中通过 `citation` event 提前发送结构化来源。
4. SSE 事件统一使用 `start -> retrieval -> text-delta/citation -> finish`，异常使用 `error` 事件，不把 provider 原始事件直接暴露给前端。
5. 业务权限和 permission-gated capability 仍由 `apps/api` 负责，不进入 `packages/ai`。

## 11. 第一阶段不建议复杂化的点

当前不建议第一阶段直接做：

1. 多 Agent 产品能力
2. 复杂工作流编排平台
3. 多模型智能路由中心
4. AI 层自带独立权限系统

## 12. 当前结论

DocWeave 的 AI 模块正确方向不是“所有 AI 功能都挤在一个接口里”，而是：

```text
apps/api own business gate
  + packages/ai own model runtime
  + packages/rag own retrieval flow
  + blocknote official ai protocol for editor ai
```

这样既能复用底层，又不会把编辑器 AI、聊天和 RAG 问答越做越乱。

## 13. M5 流式 runtime 落点

M5 使用与 `@blocknote/xl-ai@0.51.4` 兼容的 AI SDK 6：`packages/ai` 提供 provider-backed `LanguageModel`，`apps/api` 使用 `streamText` 负责场景级 system prompt、BlockNote tools 和 UI Message Data Stream。M5 暂不强行升级 AI SDK 7，避免与 BlockNote 官方包的 AI SDK 6 依赖产生类型和运行时分裂。
