# 05. 数据契约与适配层设计

阅读说明：

1. 本文档负责回答“DocWeave 如何统一设计跨边界数据 shape、适配层、前端 view-model 与 Tuyau 的分工”。
2. 若需看系统整体分层，先读 [01. 整体架构设计](./01.%20整体架构设计.md)。
3. 若需看 `apps/web` 与 `apps/api` 的具体接线方式，再读 [Frontend Adonis API Client Guide](../workflow/frontend-adonis-api-client-guide.md)。

## 1. 文档目标

本文档用于统一下面几类问题的设计口径：

1. 后端 controller / service 是否可以直接把 ORM model shape 暴露给前端。
2. 前端页面是否可以直接依赖 Tuyau 从路由推导出来的“当前响应 shape”。
3. `contracts`、`adapters`、`application`、`view-model` 分别应该承担什么职责。
4. `Tuyau` 在已有 contract 之后是否仍然有存在价值。
5. 历史兼容、第三方兼容和展示层友好字段应该落在哪里。

本文档的目标不是增加术语数量，而是让“边界到底在哪”变得清晰、可执行、可 review。

## 2. 总体结论

DocWeave 统一采用下面这条主原则：

> 跨边界先过 contract，兼容差异进 adapter，页面只吃 view-model，Tuyau 负责 transport typing，不代替业务契约。

这条原则展开后包含 5 个约束：

1. 后端 controller / service 不直接把 ORM model shape 暴露给前端。
2. 前端页面不直接依赖 Tuyau 推导出来的偶然 shape。
3. 所有跨边界数据优先先过 contract，再暴露给其他层。
4. 历史兼容和第三方兼容尽量进入 adapter，不污染 controller、page 和组件。
5. `application` 层只表达用例编排，不是把原先的 service 机械改名。

## 3. 分层职责

### 3.1 domain / service

这一层负责：

1. 业务规则。
2. 持久化协作。
3. 多对象之间的协调。

这一层不负责：

1. 对前端承诺稳定 DTO shape。
2. 暴露页面友好字段。
3. 处理 HTTP 状态码。

### 3.2 contract

`contract` 是跨边界承诺层，负责：

1. 定义 API DTO。
2. 定义写接口 input。
3. 定义 MCP payload、job payload、event payload。
4. 给 adapter、application、前端 API 层提供统一的稳定边界。

`contract` 不负责：

1. 兼容 ORM shape。
2. 兼容第三方响应。
3. 展示层友好文案。
4. 业务逻辑编排。
5. 默认 HTTP response envelope 约定（例如 `{ data }`、`{ message, data }`、`{ message, errors? }`）。

当前项目建议将这层收口在 `packages/contracts`，并按领域拆分为子入口，例如：

1. `@docweave/contracts/document`
2. `@docweave/contracts/space`
3. `@docweave/contracts/auth`

项目规范补充：

1. 新旧业务源码统一优先从领域子路径导入 contract。
2. 不再把 `@docweave/contracts` 根入口作为日常默认导入路径。
3. 根入口只保留为聚合导出层，不作为主开发路径。

### 3.3 adapter

`adapter` 是 shape 转换层，负责：

1. `Model -> DTO`
2. `raw content <-> structured content`
3. `第三方返回 -> contract`
4. `legacy payload -> current contract`

`adapter` 不负责：

1. 权限判断。
2. HTTP 状态码。
3. 大段业务编排。
4. 页面展示文案。

当前项目建议将这层收口在 `packages/adapters`。

### 3.4 application

`application` 是用例编排层，负责：

1. 表达一个明确用例，例如 `getDocument`、`updateDocument`。
2. 守住“至少更新一个字段”这类用例级业务边界。
3. 调用 service / repository / adapter 完成一次完整业务动作。

`application` 不负责：

1. 纯 shape 转换。
2. UI 友好字段拼装。
3. 把所有 service 函数原样搬运一遍。

这一层的核心要求是“按用例组织”，不是“按技术名词补层”。

### 3.5 view-model

`view-model` 是前端展示层的稳定消费入口，负责：

1. 状态值映射为页面文案。
2. 时间格式化。
3. 空摘要、空时间等展示兜底。
4. 把 DTO 中的原始内容收口成页面可直接渲染的字段。

`view-model` 不负责：

1. 发请求。
2. 调数据库。
3. 兼容后端原始 shape。

页面和组件应优先消费 `view-model`，而不是直接消费 DTO。

## 4. Tuyau 与 Contract 的关系

### 4.1 Tuyau 仍然有价值

在引入 `contract` 后，`Tuyau` 仍然保留，原因是它解决的是另一类问题：

1. 路由 params 类型联动。
2. body 类型联动。
3. 响应类型联动。
4. 前后端调用层的 transport typing。

它的价值在于“调用是否写对”，而不是“业务到底承诺什么 shape”。

### 4.2 两者的职责边界

可以把两者关系理解成：

```text
Tuyau
  -> transport typing
  -> 路由、params、body、response 的联动

Contract
  -> business contract
  -> 跨边界稳定承诺
```

因此，DocWeave 不采用“用 Tuyau 替代 contract”的做法，也不采用“有了 contract 就不需要 Tuyau”的做法。

统一做法是：

1. Controller 返回 contract DTO，而不是 ORM model。
2. Tuyau 基于 controller 返回的 DTO 继续推导调用类型。
3. 前端 API 层可以利用 Tuyau 发请求。
4. 页面层继续只消费 view-model。

### 4.3 前端 Zod / 后端 Vine 的关系

DocWeave 当前不要求“前后端必须统一使用同一个校验库”，而要求“前后端校验职责清晰，并共同围绕 contract 工作”。

当前推荐分工如下：

1. 前端如需表单校验、本地输入预校验，可使用 `zod`
2. `apps/api` 服务端请求校验继续使用 `@vinejs/vine`
3. `packages/contracts` 负责定义跨边界输入输出 shape

也就是说，DocWeave 当前统一的是“边界 contract”，不是“校验库名字”。

这样设计的原因是：

1. `zod` 更适合前端表单、本地输入体验和组件侧即时反馈
2. `vine` 与 `AdonisJS` 的 `request.validateUsing(...)` 集成更自然
3. 服务端可信边界仍应由 API 自己校验，而不是假设前端校验一定可靠

因此三者关系应理解成：

```text
zod
  -> 前端表单 / 本地输入校验
  -> 提前给用户反馈

vine
  -> 后端请求校验
  -> 服务端可信边界

contract
  -> 跨边界输入输出 shape
  -> API / MCP / job / event 的稳定承诺
```

这意味着：

1. 前端可以先用 `zod` 做一轮体验型校验
2. 请求进入 API 后，仍然必须再经过 `vine`
3. `zod` 与 `vine` 都不应各自长出与 contract 冲突的字段语义

DocWeave 当前不建议为了“看起来统一”而强制把 `Vine` 全部替换成 `Zod`。真正优先要统一的是：

1. 字段名与字段语义
2. `optional` / `nullable` 的边界
3. patch 与 create 的输入语义
4. DTO 与 input contract 的命名口径

只有在后续明确出现下面这些痛点时，再重新讨论是否统一到 `zod`：

1. 同一份输入规则需要在前后端重复维护，且维护成本明显升高
2. 前端表单 schema 数量快速增长，强依赖 schema 推导
3. 团队明确决定把“校验 schema”本身也视为共享资产，而不只是共享 contract type

## 5. 推荐数据流

以“更新文档”为例，推荐数据流如下：

```text
DocumentEditorPage
  -> 调 apps/web/src/lib/api.ts
  -> api.ts 使用 Tuyau client 发 PATCH 请求
  -> DocumentsController.validateUsing(updateDocumentValidator)
  -> application/update_document.ts 守住用例边界
  -> DocweaveCatalogService 更新模型
  -> packages/adapters 把 Model 转成 DocumentDetailDto
  -> controller 返回 DTO
  -> 前端 API 层拿到 DTO
  -> view-model 转成页面可直接消费的结构
  -> 页面渲染
```

这个数据流里每层的职责分别是：

1. `validator` 负责字段级合法性。
2. `application` 负责用例级边界。
3. `service` 负责持久化协作。
4. `adapter` 负责 shape 清洗和转换。
5. `contract` 负责跨边界承诺。
6. `Tuyau` 负责 transport typing。
7. `view-model` 负责页面友好字段。

## 6. Contract 设计规则

### 6.1 Contract 类型分类

建议至少区分 3 类 contract：

1. `Input Contract`
   - 例如 `CreateDocumentInput`
   - 例如 `UpdateDocumentInput`
2. `Output Contract`
   - 例如 `DocumentSummaryDto`
   - 例如 `DocumentDetailDto`
3. `Internal Async Contract`
   - 例如 `IndexJobPayload`
   - 例如 MCP / event payload

### 6.2 Contract 字段规则

1. 字段名表达业务语义，不表达底层存储实现。
2. 列表 DTO 和详情 DTO 分开定义，不用一个大 DTO 全场复用。
3. 枚举值要稳定，不把数据库或临时实现细节直接暴露出去。
4. `optional` 和 `nullable` 语义分开，不混用。
5. 时间、ID、富文本内容等敏感字段统一规范。

### 6.3 Patch Contract 规则

对更新接口，默认采用 patch 语义：

1. `Create*Input` 优先表达“创建时必须提供什么”。
2. `Update*Input` 优先表达“允许局部更新什么”。
3. “至少提交一个字段”这类边界，优先落到 `application`，而不是让页面、controller 和 service 各写一份。

## 7. Adapter 设计规则

### 7.1 推荐职责

adapter 推荐只做两件事：

1. shape 转换。
2. 兼容兜底。

例如：

1. `toDocumentSummaryDto(model)`
2. `toDocumentDetailDto(model)`
3. `parseDocumentContent(raw)`
4. `serializeDocumentContent(content)`
5. `extractTextPreview(content)`

### 7.2 明确禁止

adapter 不应该：

1. 偷偷查数据库。
2. 夹带权限判断。
3. 夹带 HTTP 语义。
4. 变成第二个 service。

## 8. View-Model 设计规则

前端页面默认不直接消费 DTO，而是先转成 view-model。

原因是 DTO 解决的是“边界承诺”，不是“页面是否好渲染”。

因此建议把下面这些逻辑统一收口到 view-model：

1. 状态中文文案映射。
2. 时间格式化。
3. 空时间兜底。
4. `content` 字符串转编辑器结构。

注意：`summary` 的空值兜底（从正文截取纯文本预览）当前已下沉到 `adapter` 层，由 `extractTextPreview` 在生成 DTO 时完成，不再由 view-model 独自承担。

如果页面里仍然频繁出现下面几类代码，就说明 view-model 还没收口到位：

1. `status -> 中文`
2. `new Date(updatedAt).toLocaleString(...)`
3. `summary ?? ''`
4. `JSON.parse(content)`

## 9. 落地目录建议

结合 DocWeave 当前结构，推荐采用下面的落点：

```text
packages/contracts
  -> 跨边界 DTO / Input / Payload

packages/adapters
  -> Model -> DTO
  -> content parse / serialize
  -> legacy / third-party adapters

apps/api/app/application
  -> 按用例组织
  -> get_document.ts
  -> update_document.ts

apps/api/app/controllers
  -> request.validateUsing(...)
  -> application 调用
  -> HTTP 状态码映射
  -> success response envelope 选择（`{ data }` / `{ message, data }` / `{ message }`）

apps/api/app/exceptions/handler.ts
  -> framework / auth / validation / fallback error envelope 统一
  -> `{ message, errors? }`

apps/web/src/lib/api.ts
  -> Tuyau client 的统一业务封装层
  -> `message` / `errors[]` 收口成 Query 友好的 Error

apps/web/src/features/*/lib/*view-model.ts
  -> 页面展示友好结构
```

### 10.5 当前还没有完全覆盖的部分

虽然这套设计已经开始落地，但当前仍然是“试点推进”，不是“全仓完成”。

当前尚未完全覆盖的部分包括：

1. `packages/contracts` 虽已按领域拆分，但根 `index.ts` 仍保留聚合导出，后续应避免新代码继续依赖它
2. `application` 目前主要覆盖 `documents / spaces` 主链路，其他域还未系统接入
3. 其他业务域如 `auth / rag / collab / worker payload` 还没有系统接入这套分层
4. 并非所有前端 feature 都已经有独立 view-model

因此，这份设计在当前阶段的真实状态应理解为：

1. 原则已定
2. 路径已开
3. 文档与空间主链路已有试点
4. 尚未要求一次性铺满全仓

## 11. 风险与防漂移约束

这套设计最大的风险不是“太复杂”，而是“名词越来越多，边界却还是糊的”。

因此需要明确防两件事：

### 11.1 不要只有 contracts，没有 adapter 和 view-model

如果只有 `packages/contracts`，但没有 adapter 和 view-model 的实际落点，contract 很容易退化成“类型仓库”。

结果会变成：

1. DTO 定义了一套。
2. controller 还是直接 return model。
3. 页面还是继续直接吃 DTO 原始字段。

这样收益会大幅打折。

### 11.2 不要把 application 变成“新 service 文件夹”

`application` 必须按用例组织，并只承接用例边界。

如果它只是：

1. 原有 service 改个目录。
2. 每个方法原样再包一层。
3. 不表达任何新的用例含义。

那它就不是架构分层，只是文件移动。

## 12. 当前执行建议

DocWeave 对这套设计建议采用下面的执行口径：

1. 架构原则：全盘采纳。
2. 落地节奏：分阶段推进。
3. 当前优先级：`contract -> adapter -> front view-model -> backend application`

这意味着：

1. 优先先把跨边界承诺与 shape 转换收稳。
2. 再让页面从原始 DTO 过渡到 view-model。
3. 最后按用例逐步补 application，而不是一口气铺满全仓。

## 13. 当前结论

DocWeave 当前统一采用的口径是：

1. `Tuyau` 保留，负责 transport typing。
2. `contract` 保留，负责跨边界稳定承诺。
3. `adapter` 负责兼容和 shape 转换。
4. `view-model` 负责页面可直接消费的展示结构。
5. `application` 负责用例编排，而不是代替 service 或 adapter。

只要后续继续沿着这条边界推进，项目就能在保持类型安全和开发效率的同时，避免 ORM shape、第三方 shape 和页面偶然依赖逐步污染整个代码库。
