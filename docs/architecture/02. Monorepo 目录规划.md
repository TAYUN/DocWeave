# 02. Monorepo 目录规划

阅读说明：

1. 本文档负责定义 DocWeave Monorepo 的长期目录职责边界。
2. 若需看初始化顺序和第一阶段落地节奏，继续读 [11. Monorepo 初始化与目录落地方案](../planning/11.%20Monorepo%20初始化与目录落地方案.md)。
3. 若需看总体运行时分层，继续读 [01. 整体架构设计](./01.%20整体架构设计.md)。

## 1. 文档目的

本文档用于回答以下问题：

1. 顶层 `apps/*`、`packages/*`、`infrastructure/*`、`docs/*` 分别承担什么职责。
2. 为什么某些能力要作为独立 app，而不是塞进 `apps/api`。
3. 哪些跨边界能力应该下沉到 packages。

## 2. 顶层目录

DocWeave 采用以下顶层结构：

```text
DocWeave/
├── apps/
├── packages/
├── infrastructure/
├── docs/
├── package.json
├── pnpm-workspace.yaml
└── tsconfig.base.json
```

## 3. apps 目录职责

### 3.1 apps/web

负责：

1. 工作台、文档页、搜索页、聊天页、设置页。
2. `BlockNote` 编辑器运行时。
3. `Mantine` 页面交互与状态展示。

不负责：

1. 数据库访问。
2. 权限真相判定。
3. 协同服务端同步。

### 3.2 apps/api

负责：

1. 认证、会话、权限、成员与文档元数据。
2. 文件上传元数据和签名 URL。
3. `@blocknote/xl-ai/server` 宿主接入。
4. RAG 搜索与问答编排。

不负责：

1. 协同 WebSocket。
2. 长时间索引任务执行。

### 3.3 apps/collab

负责：

1. `Hocuspocus + Yjs` 房间管理。
2. Awareness / Presence。
3. 协同 token 校验接入。

### 3.4 apps/worker

负责：

1. 快照索引任务执行。
2. 文档前处理、embedding、写入 `Qdrant`。
3. 文件后处理与导出任务。

## 4. packages 目录职责

### 4.1 packages/contracts

负责跨边界 DTO、input、job payload、event payload。

### 4.2 packages/adapters

负责 `Model -> DTO`、结构兼容、parse / serialize 与 view-model 适配。

### 4.3 packages/auth

负责角色、权限判断函数和认证相关共享 contract。

### 4.4 packages/database

负责数据库连接、查询访问层和领域数据访问封装。

### 4.5 packages/editor

负责 BlockNote 统一封装、协同接入点和前端编辑器相关共享逻辑。

### 4.6 packages/document

负责基于 `@blocknote/server-util` 的服务端快照、导出和文档前处理封装。

### 4.7 packages/ai

负责 `Vercel AI SDK` 统一模型接入、provider/runtime、Prompt 组装和流式输出封装。

### 4.8 packages/rag

负责 chunk、retrieval backend、Qdrant 实现、上下文构建、问答流程和 Citation 结构。

### 4.9 packages/collaboration

负责房间命名、协同 token contract、Awareness / Presence 结构与连接状态类型。

### 4.10 packages/ui

负责项目内共用 UI 外壳、基础组件封装和可跨页面复用的交互壳。

### 4.11 packages/config

负责 `tsconfig`、lint、test 和共享构建配置。

## 5. infrastructure 目录职责

负责本地开发与部署相关基础设施说明或编排：

1. `postgres/`
2. `redis/`
3. `qdrant/`
4. `minio/`

## 6. docs 目录职责

负责：

1. 决策文档。
2. 架构设计。
3. 阶段规划。
4. 工作流和 UI 说明。

## 7. 当前结论

Monorepo 的目标不是“看起来完整”，而是先让运行时边界和共享能力边界稳定下来。

DocWeave 当前必须坚持：

1. `apps/*` 承担运行时入口。
2. `packages/*` 承担共享领域能力。
3. `apps/collab` 和 `apps/worker` 独立存在，不塞进 `apps/api`。
