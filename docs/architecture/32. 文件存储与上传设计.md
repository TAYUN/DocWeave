# 32. 文件存储与上传设计

阅读说明：

1. 本文档负责把文件存储策略展开成运行时设计、对象命名与访问链路。
2. 文件策略拍板以 [15. 文件存储与上传策略](../decisions/15.%20文件存储与上传策略.md) 为准。
3. 文档内资源引用可继续读 [18. 文件上传与文档内资源引用设计](./18.%20文件上传与文档内资源引用设计.md)。

## 1. 文档目的

本文档用于明确 DocWeave 中文件元数据、对象存储内容、上传流程和访问解析在系统中的设计落点，避免后续“能上传”和“可维护的文件系统设计”之间断层。

## 2. 核心结论

DocWeave 的文件存储与上传设计建议如下：

1. PostgreSQL 保存文件元数据与归属关系
2. 对象存储保存真实对象内容
3. `apps/api` 管上传入口、访问校验与资源签名
4. `apps/worker` 管后处理、导出产物和文件前处理任务
5. 文件对外主标识始终使用 `fileId`

## 3. 元数据与对象内容分层

推荐分层如下：

```text
database file metadata
  -> id / owner / relation / kind / mime / size

object storage object
  -> objectKey / content bytes
```

关键原则如下：

1. 对外永远不把 `objectKey` 当业务主键
2. 删除、迁移和分享都围绕元数据关系做判断

## 4. 上传流程设计

### 4.1 第一阶段最小流程

```text
web upload file
  -> api validate
  -> api write object
  -> api create file metadata
  -> api return file dto
```

### 4.2 后续增强流程

```text
web request upload session
  -> api issue direct-upload credential
  -> web upload object
  -> api confirm metadata
```

## 5. 文件分类建议

建议至少区分：

1. 文档图片
2. 普通附件
3. 导出文件
4. 中间产物

这样做的价值在于：

1. 生命周期管理更清晰
2. 访问策略更容易分层
3. 后续 RAG / OCR / 导出任务更容易判断处理方式

## 6. 命名与路径建议

对象命名建议遵循：

1. 稳定、不可猜测
2. 允许按资源类型与工作区分层
3. 不把用户原始文件名直接当唯一对象路径

推荐方向如下：

```text
workspace/{workspaceId}/{kind}/{generatedKey}
```

## 7. 访问链路设计

推荐访问链路如下：

```text
client request file
  -> api verify current permission
  -> api issue short-lived signed url
  -> client fetch object
```

后续分享页则复用：

```text
share context
  -> api verify share access
  -> issue short-lived signed url
```

## 8. 后处理设计

建议把以下能力放到 `apps/worker`：

1. 图片压缩 / 缩略图
2. OCR 预留
3. 文件文本提取预留
4. 导出文件生成
5. 文件进入 RAG 的预处理

## 9. 与文档、导出和 RAG 的关系

文件设计必须同时兼容：

1. 编辑器内图片 / 附件引用
2. 文档导出
3. 分享访问
4. 后续文件进入 RAG

因此：

1. 文件设计不能只为上传页面服务
2. 也不能只围绕某一种文件类型定制

## 10. 不建议的做法

需要避免以下问题：

1. 上传和访问完全依赖前端拼路径
2. 不区分元数据与对象内容
3. 所有文件混放且没有类型语义
4. 直接默认公开对象存储桶

## 11. 当前结论

DocWeave 的文件系统设计，应建立在“元数据真相 + 对象内容存储 + 受控访问解析”三件事上：

```text
file metadata
  + object storage
  + api permission-gated access
  + worker post-process
```

只要这条主链稳定，后续上传直传、分享资源访问、导出和文件 RAG 才能长期收敛在同一体系里。
