# 21. BlockNote 编辑器集成设计

阅读说明：

1. 本文档负责定义 DocWeave 中 `BlockNote` 编辑器在前端页面、协同链路和业务元数据中的集成方式。
2. 文档页集成基线以 [14. 文档页协同与元数据集成设计](./14.%20文档页协同与元数据集成设计.md) 为准。
3. BlockNote 服务端能力接入以 [22. BlockNote 服务端能力接入设计](./22.%20BlockNote%20服务端能力接入设计.md) 为准。

## 1. 文档目的

本文档用于明确 BlockNote 在 DocWeave 中的前端落点、与 Yjs 的绑定方式、与业务元数据的衔接，以及页面层如何围绕编辑器组织状态，避免后续把编辑器当成一个孤立组件塞进页面。

## 2. 核心结论

DocWeave 中的 BlockNote 编辑器应按以下方式集成：

1. 作为文档页核心内容区组件存在
2. 通过 `Yjs + Hocuspocus` 绑定协同正文
3. 与文档页元数据、权限、AI 与索引状态并列集成
4. 编辑器本身不成为业务真相源，业务真相仍在 `apps/api + snapshot`

## 3. 编辑器在页面中的位置

推荐文档页结构如下：

```text
document page
  -> metadata header
  -> editor toolbar / status
  -> blocknote editor
  -> side panels (ai / presence / history / info)
```

## 4. 初始化顺序建议

建议按以下顺序初始化：

1. 获取当前用户会话
2. 获取文档页上下文 DTO
3. 计算只读 / 可编辑状态
4. 申请协同 token
5. 初始化 Yjs provider
6. 初始化 BlockNote editor
7. 绑定 Presence 与编辑器扩展

## 5. 与协同层的关系

BlockNote 编辑器与协同层的边界建议如下：

1. BlockNote 负责编辑体验与内容结构
2. Yjs 负责共享内容同步模型
3. Hocuspocus 负责 WebSocket 协同基础设施
4. 页面层负责把三者和业务状态绑在一起

## 6. 与业务元数据的关系

编辑器本体不应承担全部业务信息。

以下信息建议仍由页面或业务 DTO 管理：

1. 文档标题
2. 所属空间
3. 权限能力
4. 快照状态
5. 索引状态

## 7. 只读与可编辑状态

编辑器必须明确区分：

1. 可编辑态
2. 只读协同态
3. 无权限态

建议如下：

1. 只读态允许查看正文和 Presence
2. 只读态禁用编辑与结果应用
3. 无权限态不初始化正文协同绑定

## 8. 扩展能力接入点

建议把以下扩展能力视为编辑器周边集成，而不是编辑器本体职责：

1. 编辑器 AI
2. 图片 / 附件上传
3. Slash actions
4. 文档状态栏
5. Presence 列表

## 9. 页面状态分层

建议分为：

1. 业务页状态
2. 协同连接状态
3. 编辑器局部交互状态

不要把这些状态全部塞进单一编辑器实例对象里。

## 10. 不建议的做法

需要避免以下问题：

1. 页面一加载就先创建编辑器，再去补权限
2. 把协同 provider、业务 DTO 和编辑器 UI 紧耦合成单组件
3. 编辑器内容直接长期依赖前端临时状态
4. 标题和正文走完全不同、互不约束的协同逻辑

## 11. 当前结论

DocWeave 集成 BlockNote 的关键不是“把编辑器渲染出来”，而是把以下几层同时接稳：

```text
page context
  + capability
  + yjs collaboration
  + blocknote editor
  + side integrations
```

只要这条集成链路稳定，后续再叠 AI、文件和版本能力都会顺很多。
