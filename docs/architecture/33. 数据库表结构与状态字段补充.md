# 33. 数据库表结构与状态字段补充

阅读说明：

1. 本文档负责补充第一阶段核心表结构与关键状态字段语义。
2. 总体数据库基线以 [04. 数据库设计](./04.%20数据库设计.md) 为准。
3. 快照、索引、分享、文件等专项语义以对应专题文档为准。

## 1. 文档目的

本文档用于把第一阶段已经分散在多个专题文档中的关键字段语义再收口一次，避免后续实现时只知道“需要有这个对象”，却不知道哪些状态字段必须存在、哪些版本字段不能混用。

## 2. 适用范围

本文档主要覆盖以下对象的关键字段：

1. `workspaces`
2. `spaces`
3. `documents`
4. `document_snapshots`
5. `rag_index_jobs`
6. `files`
7. `shares`

## 3. `documents` 关键字段建议

建议重点保留：

1. `id`
2. `workspace_id`
3. `space_id`
4. `title`
5. `status`
6. `latest_snapshot_version`
7. `latest_indexed_version`
8. `latest_published_version`

状态字段说明：

1. `status` 建议至少支持 `draft / published / archived`
2. `latest_snapshot_version` 表示最新稳定正文版本
3. `latest_indexed_version` 表示当前检索可用版本
4. `latest_published_version` 表示公开阅读或正式发布版本

## 4. `document_snapshots` 关键字段建议

建议重点保留：

1. `id`
2. `document_id`
3. `version`
4. `content`
5. `created_by`
6. `created_at`

关键约束：

1. `version` 必须是单文档内稳定递增语义
2. `content` 保存稳定 BlockNote JSON 或等价服务端真相

## 5. `rag_index_jobs` 关键字段建议

建议重点保留：

1. `id`
2. `document_id`
3. `target_snapshot_version`
4. `status`
5. `stage`
6. `error_code`
7. `started_at`
8. `finished_at`

状态建议：

1. `pending`
2. `running`
3. `succeeded`
4. `failed`
5. `canceled`

阶段建议：

1. `snapshot_ready`
2. `preprocessing`
3. `chunking`
4. `embedding`
5. `upserting`
6. `publishing`

## 6. `files` 关键字段建议

建议重点保留：

1. `id`
2. `workspace_id`
3. `document_id`
4. `kind`
5. `object_key`
6. `mime_type`
7. `size`
8. `created_by`

关键原则：

1. 对外主标识是 `id`
2. `object_key` 只用于存储层定位
3. `kind` 建议至少支持 `image / attachment / export / derived`

## 7. `shares` 关键字段建议

建议重点保留：

1. `id`
2. `share_id`
3. `workspace_id`
4. `document_id`
5. `access_level`
6. `status`
7. `target_snapshot_version`
8. `expires_at`

状态建议：

1. `active`
2. `revoked`
3. `expired`

## 8. 状态字段设计原则

第一阶段状态字段建议遵循：

1. 业务状态和版本字段分开
2. 任务状态和阶段字段分开
3. 对外可见状态与内部执行态分开

## 9. 不建议的做法

需要避免：

1. 用一个 `status` 字段同时表达发布、归档、索引和任务执行
2. 用模糊字段名代替明确版本语义
3. 先省掉关键版本字段，后期再靠推导补救

## 10. 当前结论

第一阶段数据库设计最怕“对象有了，但字段语义糊”。本文档的价值就是把最关键的状态和版本字段再钉一遍，减少实现阶段反复返工。
