# 18. 文件上传与文档内资源引用设计

阅读说明：

1. 本文档负责定义文件上传、文档内图片/附件引用和资源访问的实现边界。
2. 文件存储策略以 [15. 文件存储与上传策略](../decisions/15.%20文件存储与上传策略.md) 为准。
3. 文件分享与公开资源访问可继续读 [30. 文件分享与公开资源访问设计](./30.%20文件分享与公开资源访问设计.md)。

## 1. 文档目的

本文档用于明确 DocWeave 中“文件如何进入系统、如何挂到文档里、前端如何引用、后端如何校验”的完整边界，避免后续图片、附件、导出文件和文档内资源各自采用不同引用方式。

## 2. 核心结论

DocWeave 的文件上传与文档内资源引用建议采用以下路线：

1. 文件上传由 `apps/api` 统一收口。
2. 文件元数据与对象内容分离存储。
3. 文档内资源对外主标识统一使用 `fileId`，不暴露 `objectKey`。
4. 文档内容中的图片、附件和嵌入资源应通过稳定资源引用结构挂接。

## 3. 上传链路

第一阶段推荐上传链路如下：

```text
apps/web select file
  -> apps/api validate
  -> object storage write
  -> create file metadata
  -> return file dto / fileId
```

后续可演进为：

```text
apps/web request upload session
  -> apps/api issue direct-upload credential
  -> web upload to object storage
  -> apps/api confirm metadata
```

## 4. 文档内资源引用原则

文档内资源引用建议遵循：

1. 引用结构中使用 `fileId`
2. 必要时保留 `kind`、`name`、`mimeType`
3. 不把短期 signed URL 写入长期文档真相
4. 不把对象存储路径直接写入 BlockNote 内容作为唯一来源

## 5. 资源类型建议

第一阶段建议至少区分：

1. `image`
2. `attachment`
3. `export`
4. `derived`

其中：

1. `image` 面向正文展示
2. `attachment` 面向下载与引用
3. `export` 面向系统产物
4. `derived` 面向缩略图、OCR 或其他中间资源

## 6. BlockNote 侧引用建议

在编辑器侧，建议把资源引用视为业务层扩展数据，而不是简单 URL 字符串。

推荐方向如下：

1. 上传成功后返回 `fileId` 与展示所需最小字段
2. Block / inline content 中保存 `fileId`
3. 页面渲染时再通过资源解析链路补足可访问地址

## 7. 资源解析链路

推荐资源访问链路如下：

```text
page render
  -> need file preview / download
  -> apps/api verify permission
  -> issue signed url
  -> browser fetch object
```

## 8. 与文档、快照和导出的关系

资源引用应与文档版本语义兼容：

1. 快照中保留资源引用结构
2. 导出时根据 `fileId` 解析真实资源
3. 分享页仍通过受控资源访问链路读取

## 9. `apps/api` 与 `apps/worker` 的分工

### 9.1 `apps/api`

负责：

1. 文件上传校验
2. 元数据创建与归属关系
3. 签发访问 URL
4. 删除与引用检查

### 9.2 `apps/worker`

负责：

1. 图片后处理
2. 缩略图生成
3. OCR 或文本提取预留
4. 后续文件进入 RAG 的前处理

## 10. 不建议的做法

需要避免以下问题：

1. 把 signed URL 直接写进长期文档内容
2. 前端直接拼对象地址
3. 文档内资源引用既有 `fileId` 又有大量无约束裸 URL
4. 删除文件时不检查是否仍被文档引用

## 11. 当前结论

DocWeave 的文档内资源引用，应建立在“稳定 `fileId` + 受控访问解析”这条链路上：

```text
file upload
  -> file metadata
  -> document reference by fileId
  -> api-resolved access url
```

只要这个方向先定住，后续图片、附件、导出和分享资源都会更容易统一。
