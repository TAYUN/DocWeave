# 17. 协同持久化与恢复策略

阅读说明：

1. 本文档负责定义 DocWeave 协同运行态如何持久化、恢复，以及如何与稳定快照衔接。
2. 本文档聚焦“协同服务重启后怎么办、RAG 读哪一份、第一阶段做到什么程度”。
3. 若需看版本与快照语义，继续读 [24. 版本与稳定快照设计](./24.%20版本与稳定快照设计.md)。
4. 若需看搜索与索引编排，继续读 [25. 搜索与索引任务编排设计](./25.%20搜索与索引任务编排设计.md)。

## 1. 文档目的

本文档用于明确协同正文不能永远停留在纯内存态，必须定义最小持久化与恢复路径，否则会出现服务重启即丢文、搜索与编辑状态脱节、快照真相不稳定等问题。

## 2. 核心结论

DocWeave 应明确区分两类存储对象：

1. 协同运行态
2. 稳定正文快照

推荐边界如下：

1. `apps/collab` 负责维护和恢复协同运行态。
2. 稳定快照由服务端在合适时机从协同正文导出。
3. RAG、搜索、Citation 和导出只消费稳定快照。
4. 第一阶段允许协同持久化保持最小实现，但必须从架构上预留恢复链路。

## 3. 为什么需要单独设计

如果不把持久化与恢复独立出来，后续很容易出现：

1. `apps/collab` 重启后文档内容丢失。
2. 协同实时态与业务数据库中的最新正文不一致。
3. 索引和问答读取到过旧或半完成内容。
4. 页面无法判断“当前编辑态”和“当前可检索版本”的边界。

## 4. 两层内容真相

### 4.1 协同运行态

指 `Yjs` 文档在 `apps/collab` 中的实时状态。

它的特点如下：

1. 更新频繁。
2. 适合实时编辑与 Presence。
3. 不适合作为检索与引用真相源。

### 4.2 稳定快照

指从协同正文导出的某一版稳定内容。

它的特点如下：

1. 版本可追踪。
2. 适合索引、RAG、Citation、导出和分享。
3. 不要求每次按键都立即生成。

## 5. 推荐职责拆分

建议职责如下：

1. `apps/collab`
   - 加载协同文档运行态
   - 接收 Yjs update
   - 在合适时机触发运行态存储
2. `apps/api`
   - 协调文档元数据
   - 决定稳定快照记录与索引任务触发
3. `packages/document`
   - 统一正文序列化、反序列化和稳定快照处理
4. `apps/worker`
   - 基于稳定快照继续执行切块、embedding、索引等后处理

## 6. 推荐恢复流程

当用户首次进入文档房间时，推荐流程如下：

```text
apps/collab 收到房间连接
  -> 尝试加载该文档最近一次协同持久化内容
  -> 若存在则恢复到内存中的 Yjs 文档
  -> 若不存在则从稳定正文快照或初始空文档初始化
  -> 客户端连接后继续实时同步
```

这样做的好处如下：

1. 协同层可以独立于浏览器恢复运行态。
2. 新连接进入房间时不会依赖某个浏览器重新上传整份正文。
3. 后续服务重启恢复路径更清晰。

## 7. `onLoadDocument` 与 `onStoreDocument` 接入点

`Hocuspocus` 层建议优先围绕以下钩子设计：

1. `onLoadDocument`
2. `onStoreDocument`

设计原则如下：

1. 第一阶段即使先做最小实现，也要保留这两个接入点。
2. 文档加载与存储逻辑不要散落在 WebSocket 事件回调里。
3. 后续切换底层存储实现时，尽量只替换持久化适配层。

## 8. 第一阶段建议策略

第一阶段建议采用“最小可恢复”方案：

1. 允许本地开发先以内存模式运行。
2. 但 `apps/collab` 代码结构必须支持后续接入真实存储。
3. 稳定快照的生成与索引仍围绕服务端链路设计，不依赖浏览器本地状态。

如果阶段内精力允许，推荐优先补到：

1. 协同文档可从数据库或对象存储中的上一次运行态恢复。
2. 稳定快照可在显式保存、离开空闲窗口或服务端策略点生成。

## 9. 不建议的错误做法

需要避免以下做法：

1. 把浏览器中的编辑器状态直接当作唯一正文真相源。
2. 让 RAG 直接读取协同内存态。
3. 把“协同持久化”和“稳定快照”混成同一概念。
4. 到索引阶段才发现没有稳定恢复链路。

## 10. 与快照和索引的关系

协同持久化的目标不是直接替代快照，而是保证实时层可恢复。

推荐关系如下：

```text
Yjs 运行态
  -> 协同持久化（恢复实时层）
  -> 稳定快照（服务端版本真相）
  -> 索引任务（搜索 / RAG / Citation）
```

因此：

1. 协同持久化解决“实时层别丢”。
2. 稳定快照解决“版本真相一致”。
3. 索引任务解决“可检索与可引用”。

## 11. 当前结论

DocWeave 不应把协同能力理解为“连上 WebSocket 就结束”，而应尽早建立以下链路：

1. 协同运行态可恢复。
2. 稳定正文可落地。
3. RAG 与搜索只消费稳定快照。

这样后续无论做版本、导出、分享还是索引，都不会被实时态和稳定态混淆反噬。
