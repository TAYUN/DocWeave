# 17. 协同持久化与恢复策略

阅读说明：

1. 本文档负责定义 DocWeave 协同运行态如何持久化、恢复，以及如何与稳定快照衔接。
2. 本文档聚焦“协同服务重启后怎么办、RAG 读哪一份、第一阶段做到什么程度”。
3. 若需看版本与快照语义，继续读 [24. 版本与稳定快照设计](./24.%20版本与稳定快照设计.md)。
4. 若需看搜索与索引编排，继续读 [25. 搜索与索引任务编排设计](./25.%20搜索与索引任务编排设计.md)。

## 1. 文档目的

本文档用于明确协同正文不能永远停留在纯内存态，必须定义最小持久化与恢复路径，否则会出现服务重启即丢文、搜索与编辑状态脱节、快照真相不稳定等问题。

## 2. 核心结论

DocWeave 应明确区分两类存储对象：

1. 协同运行态
2. 稳定正文快照

当前第一阶段已经落地的边界如下：

1. `apps/collab` 负责维护协同运行态，并在 `onLoadDocument` / `onStoreDocument` 上承接恢复与自动持久化。
2. 当前协同自动持久化只回写 `documents.content`，目标是保证实时层可恢复。
3. 稳定快照继续由 `apps/api` 的显式快照动作创建，不与自动持久化混用。
4. RAG、搜索、Citation 和导出只消费稳定快照，不直接消费协同运行态或 `documents.content`。

## 3. 为什么需要单独设计

如果不把持久化与恢复独立出来，后续很容易出现：

1. `apps/collab` 重启后文档内容丢失。
2. 协同实时态与业务数据库中的最新正文不一致。
3. 索引和问答读取到过旧或半完成内容。
4. 页面无法判断“当前编辑态”和“当前可检索版本”的边界。

## 4. 两层内容真相

### 4.1 协同运行态

指 `Yjs` 文档在 `apps/collab` 中的实时状态。

它的特点如下：

1. 更新频繁。
2. 适合实时编辑与 Presence。
3. 不适合作为检索与引用真相源。

### 4.2 稳定快照

指从协同正文导出的某一版稳定内容。

它的特点如下：

1. 版本可追踪。
2. 适合索引、RAG、Citation、导出和分享。
3. 不要求每次按键都立即生成。

## 5. 推荐职责拆分

当前建议职责如下：

1. `apps/collab`
   - 通过内部 API 读取 / 回写协同文档运行态
   - 接收 Yjs update
   - 在合适时机把正文自动持久化回 `documents.content`
2. `apps/api`
   - 协调文档元数据
   - 提供协同运行态内部读取 / 回写端点
   - 决定稳定快照记录与索引任务触发
3. `packages/document`
   - 统一正文序列化、反序列化和稳定快照处理
4. `packages/adapters`
   - 统一 `content string <-> BlockNote content <-> Y.Doc` 转换
5. `apps/worker`
   - 基于稳定快照继续执行切块、embedding、索引等后处理

## 6. 推荐恢复流程

当用户首次进入文档房间时，当前第一阶段推荐流程如下：

```text
apps/collab 收到房间连接
  -> 先看当前进程内是否已有该房间的 Y.Doc
  -> 若没有，则调用 apps/api 内部端点读取 documents.content
  -> 若正文存在，则恢复到内存中的 Yjs 文档
  -> 若正文为空或不存在，则退回空文档
  -> 客户端连接后继续实时同步
```

这样做的好处如下：

1. 协同层可以独立于浏览器恢复运行态。
2. 新连接进入房间时不再依赖浏览器首帧 seed 作为唯一正文来源。
3. 自动持久化和稳定快照的职责边界保持分离。

## 7. `onLoadDocument` 与 `onStoreDocument` 接入点

`Hocuspocus` 层建议优先围绕以下钩子设计：

1. `onLoadDocument`
2. `onStoreDocument`

设计原则如下：

1. 第一阶段即使先做最小实现，也要保留这两个接入点。
2. 文档加载与存储逻辑不要散落在 WebSocket 事件回调里。
3. 后续切换底层存储实现时，尽量只替换持久化适配层。

## 8. 第一阶段建议策略

第一阶段当前采用“最小可恢复”方案：

1. `apps/collab` 通过内部 API 从 `documents.content` 恢复正文。
2. `apps/collab` 自动持久化只回写 `documents.content`。
3. 稳定快照和索引仍围绕服务端显式动作设计，不依赖浏览器本地状态。

当前仍保留的后续空间如下：

1. 是否要把协同运行态从“当前正文字符串恢复”升级到更完整的运行态存储。
2. 是否要为自动持久化引入更明确的节流策略与断线恢复回归。
3. 是否要为多节点协同或更复杂恢复策略引入额外基础设施。

## 9. 不建议的错误做法

需要避免以下做法：

1. 把浏览器中的编辑器状态直接当作唯一正文真相源。
2. 让 RAG 直接读取协同内存态。
3. 把“协同持久化”和“稳定快照”混成同一概念。
4. 到索引阶段才发现没有稳定恢复链路。

## 10. 与快照和索引的关系

协同持久化的目标不是直接替代快照，而是保证实时层可恢复。

推荐关系如下：

```text
Yjs 运行态
  -> 协同持久化（恢复实时层）
  -> 稳定快照（服务端版本真相）
  -> 索引任务（搜索 / RAG / Citation）
```

因此：

1. 协同持久化解决“实时层别丢”。
2. 稳定快照解决“版本真相一致”。
3. 索引任务解决“可检索与可引用”。

## 11. 当前结论

DocWeave 不应把协同能力理解为“连上 WebSocket 就结束”，而应保持以下链路清晰：

1. 协同运行态可恢复。
2. 自动持久化与稳定快照语义分离。
3. RAG 与搜索只消费稳定快照。

这样后续无论做版本、导出、分享还是索引，都不会被实时态和稳定态混淆反噬。
