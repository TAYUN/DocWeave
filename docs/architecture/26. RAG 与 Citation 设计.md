# 26. RAG 与 Citation 设计

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段的 RAG、索引输入和 Citation 结构。
2. 总体技术基线以 [01. 最终技术方案确认](../decisions/01.%20最终技术方案确认.md) 为准。
3. 当前阶段推进顺序以 [00. 当前实施路线图](../planning/00.%20当前实施路线图.md) 为准。

## 1. 文档目的

本文档用于明确以下问题：

1. RAG 的输入来自哪里。
2. 为什么只能索引稳定快照。
3. Qdrant payload 至少要带哪些字段。
4. Citation 如何回跳到原文。

## 2. 核心结论

第一阶段 RAG 采用如下路线：

1. RAG 只索引稳定快照，不直接索引协同运行态。
2. 索引最小单位优先围绕 `BlockNote block` 及其可控 chunk 展开。
3. 检索 backend 当前固定为 `Qdrant`。
4. Citation 最小回跳结构固定为 `documentId + snapshotVersion + blockId`。

## 3. 输入来源

RAG 链路不直接读取 Yjs 文档，而是读取稳定快照中的 `BlockNote JSON`。

原因如下：

1. 协同运行态并不稳定。
2. 协同运行态不适合作为可追踪索引版本。
3. Citation 必须对应一个可回放、可校验的版本。

## 4. 索引流程

```text
document snapshot created
  -> worker 读取 BlockNote JSON
  -> 提取 block plain text
  -> 构建 chunk
  -> 调 embedding
  -> upsert 到 Qdrant
  -> 回写 indexed version
```

## 5. Chunk 原则

第一阶段建议遵循以下原则：

1. 优先保留 block 边界。
2. 不把整篇文档无上限拼成单一 chunk。
3. 若单个 block 文本过长，再在 block 内部继续切分。
4. chunk 必须能映射回原 block。

## 6. Qdrant Payload

第一阶段建议至少保留以下 payload：

1. `workspaceId`
2. `spaceId`
3. `documentId`
4. `snapshotVersion`
5. `blockId`
6. `chunkId`
7. `headingPath`
8. `plainText`

## 7. 检索流程

```text
用户提问
  -> API 先算权限可见范围
  -> packages/rag 调 Qdrant 检索
  -> 对结果做上下文组装
  -> packages/ai 生成回答
  -> 返回答案与 Citation
```

## 8. Citation 结构

第一阶段最小 Citation 建议如下：

```ts
interface Citation {
  id: string
  documentId: string
  snapshotVersion: number
  blockId: string
  chunkId?: string
  quote?: string
}
```

代码契约落点为 `packages/contracts/rag.ts` 中的 `RagCitation`。
回答正文使用 `[citationId]` 关联来源，例如 `这是结论。[c1]`，前端根据 `citations[]` 将其渲染为可点击引用。
流式回答不把 Citation 拼进 provider 原始 token 流，而是通过独立的 `citation` SSE event 和最终 `finish.citations` 传输。

约束如下：

1. 没有 `documentId` 的 Citation 无法回跳。
2. 没有 `snapshotVersion` 的 Citation 无法保证版本一致性。
3. 没有 `blockId` 的 Citation 无法精确定位到文档块。

## 9. 第一阶段不做事项

第一阶段明确不做：

1. 复杂 rerank pipeline。
2. 多后端检索混合。
3. 自动块级高亮的复杂浏览器恢复策略。
4. 全量知识图谱式关系检索。

## 10. 当前结论

第一阶段 RAG 的关键不在于“模型多强”，而在于先建立一个可控、可追踪、可回跳的闭环：

```text
稳定快照
  -> chunk
  -> qdrant
  -> search/chat
  -> citation 回原文
```
