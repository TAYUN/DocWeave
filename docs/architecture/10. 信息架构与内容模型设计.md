# 10. 信息架构与内容模型设计

阅读说明：

1. 本文档负责定义 DocWeave 的核心业务对象、层级关系与内容组织语义。
2. 总体架构边界以 [01. 整体架构设计](./01.%20整体架构设计.md) 为准。
3. 页面与导航视角可继续读 [页面总览地图](../ui/page-inventory.md)。

## 1. 文档目的

本文档用于明确 DocWeave 中“用户到底在管理什么对象、这些对象如何组织、内容如何在页面和权限中被引用”，避免后续把工作区、空间、文档、快照、文件、分享对象和检索对象混成一团。

## 2. 核心结论

DocWeave 的信息架构建议围绕以下层级组织：

1. 用户
2. 工作区 `workspace`
3. 空间 `space`
4. 文档 `document`
5. 文档内容版本 `snapshot`
6. 文件 `file`
7. 分享对象 `share`

其中：

1. `workspace` 是成员、权限和资源归属的最高业务边界。
2. `space` 是内容分类与导航容器。
3. `document` 是协同编辑、AI 与 RAG 的核心内容对象。
4. `snapshot` 是稳定内容版本，不等于实时协同态。

## 3. 为什么这样分层

采用该分层的原因如下：

1. 工作区适合承载成员、计费、设置和跨空间权限边界。
2. 空间适合承载知识主题、内容聚合和导航语义。
3. 文档适合承载实时编辑、历史版本、分享和 AI 上下文。
4. 稳定快照适合承载导出、RAG、Citation 和发布版本语义。

## 4. 核心对象关系

推荐关系如下：

```text
workspace
  -> has many spaces
  -> has many documents
  -> has many members

space
  -> belongs to workspace
  -> has many documents

document
  -> belongs to workspace
  -> belongs to space
  -> has many snapshots
  -> has many files
  -> may have many shares
```

## 5. 页面视角下的信息架构

从页面使用角度看，核心入口建议如下：

1. 工作台首页
   - 面向最近使用、空间导航和文档入口
2. 空间详情页
   - 面向某一知识空间内的文档列表和层级浏览
3. 文档编辑页
   - 面向正文、元数据、AI、索引状态和协同状态
4. 搜索 / 聊天页
   - 面向跨文档检索与问答

## 6. 内容模型建议

### 6.1 文档对象

文档对象建议至少包含：

1. `documentId`
2. `workspaceId`
3. `spaceId`
4. `title`
5. `status`
6. `latestSnapshotVersion`
7. `latestIndexedVersion`
8. `latestPublishedVersion`

### 6.2 快照对象

快照对象建议至少包含：

1. `documentId`
2. `version`
3. `content`
4. `createdBy`
5. `createdAt`

### 6.3 文件对象

文件对象建议至少包含：

1. `fileId`
2. `workspaceId`
3. `documentId` 或等价归属关系
4. `kind`
5. `objectKey`
6. `mimeType`

## 7. 文档状态语义

文档状态建议与内容版本、公开访问和索引状态解耦。

第一阶段建议至少保留以下内容状态语义：

1. `draft`
2. `published`
3. `archived`

关键原则如下：

1. `draft` 不等于不能搜索，是否进入检索由索引策略决定。
2. `published` 不等于协同已停止，它只代表某一稳定版本被明确对外或对内标记为正式版本。
3. `archived` 不应自动删除历史快照与引用关系。

## 8. 树结构与导航语义

第一阶段建议保持：

1. 空间是文档的主导航容器。
2. 文档详情路由仍以 `documentId` 为核心主键。
3. 页面面包屑和文档树应能回推出 `workspace -> space -> document` 关系。

## 9. 与协同、AI、RAG 的关系

信息架构必须支撑三条主链路：

1. 协同围绕 `document`
2. AI 围绕 `document + selection + permission`
3. RAG 围绕 `snapshot + chunk + citation`

因此：

1. 协同运行态不能直接替代文档版本对象。
2. RAG 不应直接围绕页面临时态组织。
3. AI 入口应读取业务文档上下文，而不是只依赖前端局部状态。

## 10. 第一阶段不建议复杂化的点

当前不建议第一阶段就重投入：

1. 多层嵌套文档树
2. 数据库层超复杂 polymorphic 内容模型
3. 一开始就做全局知识图谱式内容关系
4. 过细的页面对象类型分裂

## 11. 当前结论

DocWeave 的信息架构应先围绕“工作区 - 空间 - 文档 - 快照”这条主链稳定下来，再在其上叠加协同、AI、RAG、分享和文件能力。

只有主对象关系先清楚，后续权限、路由、搜索和导出才不会各说各话。
