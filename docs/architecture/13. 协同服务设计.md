# 13. 协同服务设计

阅读说明：

1. 本文档负责定义 `apps/collab` 的职责边界、技术路线与第一阶段落地要求。
2. 本文档聚焦“协同服务怎么拆、怎么接、怎么演进”。
3. 若需看文档页如何消费协同能力，继续读 [14. 文档页协同与元数据集成设计](./14.%20文档页协同与元数据集成设计.md)。
4. 若需看协同 token、Presence 和共享协议，继续读 [15. 协同领域共享协议设计](./15.%20协同领域共享协议设计.md) 与 [16. 协同 Token 与 Presence 设计](./16.%20协同%20Token%20与%20Presence%20设计.md)。

## 1. 文档目的

本文档用于明确 DocWeave 协同服务的边界与落地方式，避免后续把实时协同、业务 CRUD、权限真相和索引链路混在一起。

本文档重点回答以下问题：

1. 协同服务采用什么技术路线。
2. `apps/api`、`apps/collab`、`apps/web` 各自负责什么。
3. 第一阶段协同闭环做到什么程度。
4. 后续如何平滑扩展到持久化、快照和恢复。

## 2. 核心结论

DocWeave 的协同层采用如下路线：

1. 使用 `Yjs` 作为协同数据模型。
2. 使用 `Hocuspocus` 作为协同 WebSocket 服务框架。
3. 独立建设 `apps/collab` 进程，不把协同 WebSocket 塞进 `apps/api`。
4. `apps/api` 负责业务真相、权限判断和协同 token 签发。
5. `apps/collab` 负责房间管理、Yjs update 同步、Awareness 同步和后续持久化接入点。

## 3. 为什么采用这条路线

采用该路线的原因如下：

1. `BlockNote + Yjs` 本身就是强客户端、强实时协同形态，React SPA 与独立协同进程更契合。
2. `Hocuspocus` 已经覆盖 Yjs 同步、Awareness、连接管理等底层能力，没有必要从零实现 CRDT 协议。
3. DocWeave 仍然需要自己掌控文档房间语义、权限边界、持久化时机和与快照/RAG 的衔接。
4. 协同连接生命周期与普通 HTTP CRUD 完全不同，拆成独立进程后演进和排障都更清晰。

## 4. 运行时位置

推荐运行时关系如下：

```text
apps/web
  -> 文档页 UI、BlockNote、Yjs 客户端、Presence 展示

apps/api
  -> 认证、权限、文档元数据、协同 token 签发、快照触发

apps/collab
  -> Hocuspocus、房间连接、Yjs update、Awareness、协同持久化接入

apps/worker
  -> 稳定快照后处理、索引任务、embedding、文件后处理
```

## 5. 职责边界

### 5.1 `apps/collab` 负责什么

1. 启动 `Hocuspocus` Server。
2. 接收和管理文档房间 WebSocket 连接。
3. 同步 Yjs update。
4. 同步基础 Awareness / Presence。
5. 校验协同 token 并做最小准入判定。
6. 预留 `onLoadDocument`、`onStoreDocument` 等持久化钩子。

### 5.2 `apps/collab` 不负责什么

1. 不负责用户登录。
2. 不负责工作区、空间、文档 CRUD。
3. 不负责主业务权限真相计算。
4. 不负责 RAG 检索与问答。
5. 不负责普通 HTTP 业务接口。

### 5.3 `apps/api` 负责什么

1. 读取主业务会话。
2. 计算目标文档访问权限与协同能力。
3. 签发短期协同 token。
4. 提供文档页元数据、快照状态和索引状态。
5. 触发快照落地与后续索引任务。

## 6. 第一阶段目标

第一阶段协同闭环只要求建立最小可运行版本。

本阶段应完成：

1. `apps/collab` 可以独立启动。
2. 同一房间内两个浏览器可以协同编辑同一文档。
3. 基础 Presence 可以展示在线成员。
4. 协同连接必须经过 `apps/api` 签发的短期 token。
5. 代码结构上预留持久化与恢复接入点。

本阶段不要求：

1. 多节点横向扩展。
2. Redis pub/sub 扩展。
3. 完整历史版本功能。
4. 评论、批注、任务等更多实时对象。

## 7. 房间模型

协同房间必须与业务文档稳定映射，推荐采用：

```text
workspace:{workspaceId}:document:{documentId}
```

采用该规则的原因如下：

1. 房间语义稳定且可解析。
2. 便于协同日志排查与权限校验。
3. 便于后续扩展评论房间、子对象房间等其他前缀。

## 8. 鉴权模式

协同层不直接信任浏览器登录态，而采用如下模式：

```text
apps/web 请求 apps/api
  -> apps/api 校验业务会话与文档权限
  -> apps/api 签发短期 collab token
  -> apps/web 使用 token 连接 apps/collab
  -> apps/collab 校验 token 并绑定 room
```

关键约束如下：

1. 协同 token 必须短期有效。
2. token 必须绑定 `workspaceId`、`documentId`、`roomName` 和 `capabilities`。
3. `apps/collab` 只做最小准入判断，不自己成为权限真相源。

## 9. 与持久化和快照的关系

协同运行态与稳定正文不能混为一谈。

当前边界如下：

1. `apps/collab` 维护实时 Yjs 文档状态。
2. 稳定快照由服务端在合适时机落地。
3. `packages/rag` 只消费稳定快照，不直接消费协同运行态。
4. 索引任务由 `apps/api` 或 `apps/worker` 围绕稳定快照推进。

这意味着协同服务必须为后续持久化与恢复预留清晰钩子，但不应承担完整索引编排职责。

## 10. 与 BlockNote 的关系

推荐按如下边界理解：

1. `BlockNote` 负责编辑体验、块结构和前端扩展能力。
2. `Yjs` 负责协同数据同步模型。
3. `Hocuspocus` 负责 WebSocket 同步基础设施。
4. `apps/collab` 负责把这些底层能力与 DocWeave 的业务边界接起来。

协同服务不直接理解页面 UI 细节，它主要操作房间、token、Yjs 文档和 Awareness 协议。

## 11. 推荐目录方向

`apps/collab` 推荐先按下面结构组织：

```text
apps/collab/src
├── index.ts
├── config.ts
├── server.ts
├── auth/
├── rooms/
├── persistence/
├── awareness/
└── logging/
```

其中：

1. `auth/` 负责 token 校验与连接准入。
2. `rooms/` 负责房间命名、解析与房间级上下文。
3. `persistence/` 负责 `onLoadDocument`、`onStoreDocument` 等接入点。
4. `awareness/` 负责 Presence 结构与同步约束。

## 12. 风险与注意事项

需要避免以下高风险做法：

1. 把协同 WebSocket 混进 `apps/api`。
2. 让 `apps/collab` 直接查业务数据库并自己计算权限。
3. 直接把协同运行态当作 RAG 真相源。
4. 第一阶段为了快而省掉 token 准入链路。

## 13. 当前结论

DocWeave 协同服务的正确定位不是“又一个后端服务”，而是：

1. 独立的实时同步执行层。
2. 由 `apps/api` 提供业务真相与准入。
3. 为后续持久化、快照和索引链路提供稳定连接点。
