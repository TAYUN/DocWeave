# 28. 文件进入 RAG 的处理链路设计

阅读说明：

1. 本文档负责定义附件文件、导入文件和后续外部文档如何进入 DocWeave 的 RAG 链路。
2. 本文档聚焦“哪些文件能进 RAG、谁负责提取、什么时候触发索引”。
3. 文件存储边界以 [15. 文件存储与上传策略](../decisions/15.%20文件存储与上传策略.md) 为准。
4. RAG 索引与检索基线分别以 [25. 搜索与索引任务编排设计](./25.%20搜索与索引任务编排设计.md) 和 [26. RAG 与 Citation 设计](./26.%20RAG%20与%20Citation%20设计.md) 为准。

## 1. 文档目的

本文档用于明确 DocWeave 中“非正文文件”进入 RAG 的最小链路，避免后续把图片、PDF、Markdown、导入文件和编辑器正文混成同一种索引对象，导致版本语义、权限边界和处理成本一起失控。

## 2. 核心结论

DocWeave 中的文件进入 RAG 应采用以下路线：

1. 第一阶段 RAG 主输入仍然是稳定快照正文。
2. 文件类输入不自动默认进入 RAG。
3. 当后续支持文件进入 RAG 时，必须先经过 `apps/worker` 的受控提取与标准化。
4. 文件提取结果应先转成稳定、可追踪的文本或结构化片段，再进入 `packages/rag`。
5. 文件 RAG 与正文 RAG 必须共享权限、版本和 Citation 基线，但可以拥有不同的前处理链路。

## 3. 为什么要单独设计

如果不把文件链路单独收口，后续容易出现：

1. 用户上传一个文件就立即触发不可控索引。
2. 原始二进制文件和提取文本之间无法追踪对应关系。
3. 权限回收后仍保留旧文件片段命中结果。
4. 图片、PDF、Markdown、导入文档的处理成本和准确性完全不可控。

## 4. 输入对象分层

建议先区分以下输入类型：

1. 文档稳定快照正文
2. 文档内附件文件
3. 独立上传文件
4. 导入生成的原始文件
5. 后续 OCR / 解析中间产物

第一阶段只有第一类默认进入主 RAG 链路。

## 5. 推荐处理链路

文件进入 RAG 的后续链路建议如下：

```text
文件上传
  -> apps/api 写 file metadata
  -> apps/worker 识别文件类型
  -> 提取 text / structure / metadata
  -> 生成 file extraction artifact
  -> packages/rag chunk builder
  -> embedding
  -> upsert qdrant
  -> 回写 file index status
```

## 6. 运行时职责

### 6.1 `apps/api`

负责：

1. 校验当前用户是否允许上传、查看和索引文件。
2. 管理文件与工作区 / 空间 / 文档的归属关系。
3. 触发文件提取与索引任务。

### 6.2 `apps/worker`

负责：

1. 判断文件类型与支持能力。
2. 执行 OCR、文本提取、格式转换等重型任务。
3. 产出可供 RAG 消费的稳定提取结果。

### 6.3 `packages/document`

负责：

1. 承接 BlockNote / 快照正文的服务端处理。
2. 不强行吞掉所有文件提取逻辑，但可为 Markdown / HTML 类导入内容提供标准化入口。

### 6.4 `packages/rag`

负责：

1. 消费提取后的稳定文本或结构化片段。
2. 执行 chunk、embedding、upsert 和检索。
3. 统一 Citation 输出结构。

## 7. 文件提取结果的真相

文件不应直接以“原始对象内容”进入检索，而应先生成稳定提取结果。

建议至少保留：

1. `fileId`
2. `workspaceId`
3. `documentId` 或等价归属对象
4. `sourceType`
5. `extractionVersion`
6. `plainText`
7. `sectionPath` 或等价结构信息

这样做的原因如下：

1. 便于后续重提取、重建索引与排障。
2. 便于判断某次命中来自正文还是来自附件。
3. 便于在 Citation 中回跳到正确资源。

## 8. 第一阶段与后续阶段的边界

### 8.1 第一阶段

第一阶段明确保持：

1. 主 RAG 只吃稳定快照正文。
2. 文件上传不自动触发 OCR 或文件索引。
3. 只为后续文件进入 RAG 保留架构边界。

### 8.2 后续阶段

后续可按以下顺序扩展：

1. Markdown / HTML 文件导入后的文本进入 RAG
2. PDF 文本提取进入 RAG
3. 图片 OCR 文本进入 RAG
4. Office 文档的受控提取进入 RAG

## 9. 权限与检索边界

文件进入 RAG 后仍必须遵守业务权限计算。

推荐原则如下：

1. 权限真相仍在 PostgreSQL 与业务层。
2. 文件片段检索前仍由 `apps/api` 先算可见范围。
3. Qdrant payload 中可带 `fileId`、`documentId`、`workspaceId` 等辅助过滤字段，但不把它当最终权限真相。

## 10. Citation 建议

当命中来源于文件时，Citation 结构建议扩展为可区分资源来源。

可选方向如下：

```ts
interface Citation {
  resourceType: 'document-block' | 'file-fragment'
  documentId?: string
  snapshotVersion?: number
  blockId?: string
  fileId?: string
  fragmentId?: string
}
```

第一阶段可以先不落完整结构，但这个扩展方向应先固定。

## 11. 不建议的做法

需要避免以下问题：

1. 上传后立即无条件自动 OCR 与索引。
2. 直接把原始 PDF / 图片二进制作为 RAG 输入。
3. 文件提取结果不记录来源文件与版本。
4. 正文 chunk 与文件 chunk 混在一起却没有来源区分。

## 12. 当前结论

DocWeave 后续若要把文件真正纳入知识库，正确做法不是“上传就搜”，而是：

```text
file metadata
  -> worker extraction
  -> stable text artifact
  -> rag chunk/index
  -> source-aware citation
```

只有这样，文件 RAG 才能和正文 RAG 一样可追踪、可控、可排障。
