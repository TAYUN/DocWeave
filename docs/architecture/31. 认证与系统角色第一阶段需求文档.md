# 31. 认证与系统角色第一阶段需求文档

阅读说明：

1. 本文档负责把第一阶段认证、角色和基础权限需求收口成可实施范围。
2. 认证基线以 [14. 认证与会话实现决策](../decisions/14.%20认证与会话实现决策.md) 为准。
3. 权限模型与实现分别以 [11. 权限模型设计](./11.%20权限模型设计.md) 和 [12. 权限实现设计](./12.%20权限实现设计.md) 为准。

## 1. 文档目的

本文档用于回答第一阶段到底要把认证和角色做到什么程度，哪些属于必须落地，哪些明确后置，避免后续需求无限膨胀到 SSO、社交登录、复杂租户策略和全量后台管理。

## 2. 核心结论

第一阶段认证与系统角色建议只做到最小业务闭环：

1. 账号密码登录
2. 当前用户会话识别
3. 工作区成员边界
4. 文档基本读写能力区分
5. 协同 token、编辑器 AI、RAG 统一复用同一权限真相

## 3. 第一阶段必须包含

### 3.1 认证

1. 注册
2. 登录
3. 登出
4. 获取当前用户

### 3.2 角色

1. 工作区基础角色
2. 文档层最小读写区分

### 3.3 受保护能力

1. 文档页访问
2. 协同连接
3. 编辑器 AI
4. RAG 搜索 / 问答
5. 文件上传与访问

## 4. 第一阶段最小角色建议

建议至少支持：

1. `owner`
2. `member`
3. `viewer`

如果阶段内需要再细一层，可在文档协同视角再补：

1. `editor`
2. `commenter`

## 5. 第一阶段最小能力口径

建议至少有：

1. 能否访问工作区
2. 能否访问文档
3. 能否编辑文档
4. 能否使用编辑器 AI
5. 能否触发 RAG 搜索 / 聊天

## 6. 用户故事视角

第一阶段应至少满足以下故事：

1. 用户可以登录并进入自己的工作区
2. 成员可以打开有权限的文档
3. 只读用户可以查看但不能修改正文
4. 可编辑用户可以协同编辑并应用 AI 结果
5. 无权限用户无法获取协同 token、无法访问文档内容

## 7. 明确不做事项

第一阶段明确不做：

1. 社交登录
2. 企业 SSO
3. 多身份提供商切换
4. 复杂组织架构同步
5. 独立 API key 平台化身份体系

## 8. 验收关注点

这一主题在第一阶段至少要回答清楚：

1. 用户是否能稳定登录和维持会话
2. 页面、协同、AI、RAG 是否都围绕同一权限真相运行
3. 只读和可编辑边界是否真正生效

## 9. 当前结论

认证与系统角色第一阶段不是为了“把身份系统做完整”，而是先保证：

```text
user session
  -> role resolve
  -> capability resolve
  -> page / collab / ai / rag gate
```

只要这条主链清晰，后续扩更复杂认证体系才不会返工太重。
