# 22. BlockNote 服务端能力接入设计

阅读说明：

1. 本文档负责定义 `@blocknote/xl-ai/server` 与 `@blocknote/server-util` 在 DocWeave 中的落点与边界。
2. 总体技术路线以 [01. 最终技术方案确认](../decisions/01.%20最终技术方案确认.md) 为准。
3. 运行时边界以 [01. 整体架构设计](./01.%20整体架构设计.md) 为准。

## 1. 文档目的

本文档用于明确以下问题：

1. `@blocknote/xl-ai/server` 应落在哪个运行时。
2. `@blocknote/server-util` 应由哪个包或服务封装。
3. `AdonisJS`、`packages/ai`、`packages/document`、`apps/worker` 分别承担什么职责。
4. 哪些能力优先复用官方生态，哪些能力保留项目适配边界。

## 2. 核心结论

当前项目正式采用如下接入方式：

1. `@blocknote/xl-ai/server` 由 `apps/api` 宿主接入。
2. 模型能力统一由 `packages/ai` 提供给 `@blocknote/xl-ai/server`。
3. `@blocknote/server-util` 的服务端文档处理能力统一由 `packages/document` 封装。
4. `apps/worker` 通过 `packages/document` 使用 `@blocknote/server-util`，而不直接在每个任务里散落原始调用。

## 3. 为什么这样分层

采用该分层的原因如下：

1. `@blocknote/xl-ai/server` 本质上仍是业务 API 的一部分，因为它依赖权限、文档、上下文和模型调用编排。
2. `packages/ai` 负责统一模型接入，避免把 Vercel AI SDK 或 provider 差异散落在 `apps/api` 各处。
3. `@blocknote/server-util` 属于文档结构处理层，天然更适合收口到 `packages/document`。
4. `apps/worker` 需要复用同一套文档服务端处理能力，因此不能只把它写死在 `apps/api`。

## 4. 运行时落点

### 4.1 apps/api

负责：

1. 宿主接入 `@blocknote/xl-ai/server`。
2. 校验当前用户与目标文档权限。
3. 组装编辑器 AI 所需上下文。
4. 调用 `packages/ai` 提供模型运行时。

不负责：

1. 在控制器里散落模型 provider 细节。
2. 直接承担文档快照导出与重型转换流水线。

### 4.2 packages/ai

负责：

1. 统一接入 `Vercel AI SDK`。
2. 提供可复用的 provider/runtime 抽象。
3. 为 `@blocknote/xl-ai/server` 提供模型调用能力。
4. 为聊天、RAG、编辑器 AI 提供统一模型层。

### 4.3 packages/document

负责：

1. 基于 `@blocknote/server-util` 封装快照处理。
2. 基于 `@blocknote/server-util` 封装导出前处理。
3. 提供服务端读取和转换 `BlockNote JSON` 的统一入口。
4. 为 `packages/rag` 和 `apps/worker` 提供稳定输入。

### 4.4 apps/worker

负责：

1. 调用 `packages/document` 处理快照输入。
2. 调用 `packages/rag` 完成 chunk / embedding / upsert qdrant。
3. 处理后续导出、重建索引和文件后处理任务。

## 5. 编辑器 AI 接入链路

第一阶段建议固定如下链路：

```text
apps/web
  -> @blocknote/xl-ai
  -> apps/api hosted @blocknote/xl-ai/server
  -> apps/api permission check
  -> packages/ai runtime
  -> Vercel AI SDK
```

约束如下：

1. 前端优先贴近 `@blocknote/xl-ai` 官方交互模型。
2. 后端仍必须经过 DocWeave 的权限与上下文编排。
3. 模型 provider、base URL、key 和重试策略不直接散落在控制器中。

## 6. 文档服务端处理链路

第一阶段建议固定如下链路：

```text
stable snapshot
  -> packages/document
  -> @blocknote/server-util
  -> structured output / plain text / export pre-processing
  -> packages/rag or apps/worker consumer
```

这样做的好处：

1. 服务端对 BlockNote JSON 的解析逻辑集中管理。
2. 快照、导出、RAG 前处理共享同一套转换基础。
3. 后续更换导出格式或增加文件解析能力时，不需要在多个应用重复实现。

## 7. 与 AdonisJS 的关系

`AdonisJS` 在这里承担的是宿主业务 API 职责，而不是替代 BlockNote 官方生态。

具体来说：

1. `AdonisJS` 负责 session、auth、policy、request validation、response streaming。
2. `@blocknote/xl-ai/server` 负责编辑器 AI 的官方协议能力。
3. `packages/ai` 负责模型层。
4. `packages/document` 负责文档结构处理层。

## 8. 建议的包边界

建议在 Monorepo 中明确以下边界：

```text
apps/api
  -> route / controller / auth / permission / stream response

packages/ai
  -> model runtime / provider config / prompt helpers

packages/document
  -> blocknote server util adapters / snapshot parsers / export preprocessors

packages/rag
  -> retrieval / chunk / context build / citation
```

## 9. 第一阶段不做事项

第一阶段明确不做：

1. 为了“完全抽象”而重新发明一套与 `@blocknote/xl-ai/server` 等价的协议。
2. 让每个 worker task 直接裸调 `@blocknote/server-util`，不经统一封装。
3. 在没有真实业务需求时过早做多文档复杂导出流水线。
4. 把 BlockNote 服务端能力拆成独立微服务。

## 10. 当前结论

DocWeave 第一阶段对 BlockNote 服务端能力的正确接法不是“全都自己重写”，而是：

```text
apps/api host official AI server
  + packages/ai own model runtime
  + packages/document own server-util wrappers
  + apps/worker consume shared document pipeline
```

这样既贴近官方生态，又保留了项目自己的业务边界和后续可替换空间。
