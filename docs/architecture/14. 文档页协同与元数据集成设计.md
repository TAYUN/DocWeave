# 14. 文档页协同与元数据集成设计

阅读说明：

1. 本文档负责定义文档页如何整合元数据、协同正文、权限、Presence、AI 与索引状态。
2. 本文档聚焦“文档页初始化顺序、数据来源拆分和页面级集成边界”。
3. 若需看协同 token 与 Presence 结构，继续读 [16. 协同 Token 与 Presence 设计](./16.%20协同%20Token%20与%20Presence%20设计.md)。
4. 若需看最终技术总基线，优先读 [01. 最终技术方案确认](../decisions/01.%20最终技术方案确认.md)。

## 1. 文档目的

本文档用于明确文档页如何把业务元数据、协同正文、权限状态、Presence、索引状态和编辑器能力整合到同一个页面中。

## 2. 核心结论

DocWeave 的文档页不应被理解成“一个只负责渲染编辑器的页面”，而应被理解成：

```text
文档业务上下文
  + 协同正文
  + 页面级权限状态
  + 编辑器扩展能力
  + AI / 索引状态入口
```

推荐页面组织原则如下：

1. 文档元数据由 `apps/api` 提供。
2. 协同正文由 `apps/collab` 同步。
3. 页面先拿业务上下文，再建立协同连接。
4. 页面不把协同正文当作唯一业务真相源。

## 3. 页面初始化流程

推荐初始化顺序如下：

```text
进入文档页
  -> 获取当前用户会话
  -> 获取文档页面上下文
  -> 判定权限与 editable 状态
  -> 获取协同 token
  -> 初始化 editor
  -> 建立协同连接
  -> 绑定 presence / metadata / title
```

## 4. 推荐数据来源拆分

### 4.1 业务元数据

来源于主业务 API，例如：

1. 文档标题。
2. 所属空间。
3. 当前用户角色。
4. 权限能力。
5. 快照与索引状态。

### 4.2 协同正文

来源于协同服务，例如：

1. 正文 block 内容。
2. 标题协同态。
3. awareness / presence。

### 4.3 页面派生状态

页面本地计算或聚合，例如：

1. 当前是否只读。
2. 当前协同是否已连接。
3. 当前是否允许 AI 应用结果。

## 5. 只读与可编辑状态

文档页必须显式区分：

1. 可访问但只读。
2. 可访问且可编辑。
3. 无权限访问。

页面行为建议如下：

1. 只读时仍允许加载正文和 Presence。
2. 只读时禁用 AI 应用、编辑和插入行为。
3. 无权限时不初始化编辑器协同正文。

## 6. 当前结论

文档页的关键不是“把编辑器渲染出来”，而是把以下几个来源正确整合：

1. 业务元数据。
2. 协同正文。
3. 页面权限状态。
4. 编辑器扩展能力。
