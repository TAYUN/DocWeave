# 16. 协同 Token 与 Presence 设计

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段协同 token、房间语义和 Presence 数据结构。
2. 认证与会话基线以 [14. 认证与会话实现决策](../decisions/14.%20认证与会话实现决策.md) 为准。
3. 总体架构边界以 [01. 整体架构设计](./01.%20整体架构设计.md) 为准。

## 1. 文档目的

本文档用于明确以下问题：

1. 协同 token 由谁签发、谁校验、携带什么字段。
2. 房间命名如何保持稳定和可解析。
3. Presence 要同步哪些最小信息。
4. 只读、可编辑、AI 可用等能力如何映射到协同能力字段。

## 2. 核心结论

第一阶段协同链路采用如下路线：

1. `apps/api` 负责签发短期协同 token。
2. `apps/collab` 负责校验 token 并建立房间连接。
3. 房间主语义围绕 `workspace + document` 组合构建。
4. Presence 只同步最小必要在线信息，不把业务真相全量塞进协同层。
5. 协同 token 里要明确 `capabilities`，避免 `apps/collab` 再去猜当前连接是否可编辑。
6. 第一阶段 `workspaceId` 与 `documentId` 沿用当前业务层的字符串 ID 语义，不强行改成数字。

## 3. 目标与非目标

### 3.1 目标

1. 让协同连接建立过程可被独立验证。
2. 让 `apps/collab` 不依赖业务数据库也能做最小准入判定。
3. 让 Presence 能支撑在线状态、只读/可编辑提示和编辑器光标广播控制。

### 3.2 非目标

第一阶段不追求：

1. 复杂房间级 ACL 动态热更新。
2. 多端设备会话聚合。
3. 复杂协同审计。
4. 评论、批注、任务等更多实时能力的统一协议。

## 4. 协同房间命名

第一阶段建议采用稳定、可解析的房间命名：

```text
workspace:{workspaceId}:document:{documentId}
```

约束如下：

1. 房间名必须只表达协同正文对象边界，不夹带 UI 语义。
2. 房间名必须能被 `apps/collab` 解析回 `workspaceId` 和 `documentId`。
3. 后续如需支持评论房间、Presence 专房间或子对象房间，应新增独立前缀，不复用当前主文档房间格式。

## 5. 协同 Token 结构

第一阶段建议统一为如下结构：

```ts
interface CollaborationTokenPayload {
  version: 1
  workspaceId: string
  documentId: string
  roomName: string
  capabilities: {
    canRead: boolean
    canEdit: boolean
    canComment: boolean
    canUseEditorAI: boolean
  }
  user: {
    id: number
    name: string
    avatarUrl: string | null
    color: string
    role: 'owner' | 'editor' | 'commenter' | 'viewer'
  }
  issuedAt: string
  expiresAt: string
}
```

## 6. 为什么 token 要带 capabilities

推荐在 token 中显式带 `capabilities`，原因如下：

1. `apps/collab` 需要快速判断当前连接是否允许编辑。
2. 前端需要基于同一份可信协同会话数据决定是否广播光标和选区。
3. 后续如果出现 `viewer` 可读但不可编辑、`commenter` 可评论但不可写正文等场景，显式能力字段比只传角色更稳。

## 7. Token 生命周期

第一阶段建议采用短期 token：

1. token 必须带 `issuedAt` 和 `expiresAt`。
2. token 应视为房间连接凭据，而不是长期登录态。
3. 过期后由 `apps/web` 重新向 `apps/api` 申请新的协同 token。

## 8. apps/api 的职责

`apps/api` 负责：

1. 读取当前业务会话。
2. 校验用户是否有目标文档访问权限。
3. 计算协同能力字段。
4. 签发短期协同 token。

`apps/api` 不负责：

1. 持续维持协同连接。
2. 直接转发 Yjs update。
3. 在每次 update 时重新做正文级权限判定。

## 9. apps/collab 的职责

`apps/collab` 负责：

1. 校验协同 token。
2. 解析 `roomName`。
3. 基于 token 决定是否接受连接。
4. 基于 `capabilities.canEdit` 控制是否允许编辑态广播。

`apps/collab` 不负责：

1. 查询主业务数据库。
2. 重新计算空间或文档成员关系。
3. 判定编辑器 AI 或 RAG 能否使用。

## 10. Presence 数据结构

第一阶段建议 Presence 只同步最小可展示信息：

```ts
interface CollaborationAwarenessState {
  user: {
    id: number
    name: string
    avatarUrl: string | null
    color: string
    role: 'owner' | 'editor' | 'commenter' | 'viewer'
  }
  canEdit: boolean
}
```

展示侧可转换为：

```ts
interface CollaborationPresenceEntry {
  id: number
  name: string
  avatarUrl: string | null
  color: string
  role: 'owner' | 'editor' | 'commenter' | 'viewer'
  canEdit: boolean
  isCurrentUser: boolean
}
```

## 11. Presence 使用原则

Presence 仅用于：

1. 在线成员展示。
2. 当前连接是否可编辑的 UI 提示。
3. 编辑器光标、选区等实时协作体验的最小支撑。

Presence 不用于：

1. 作为权限真相源。
2. 作为成员列表真相。
3. 存储复杂业务状态。

## 12. 前端行为建议

`apps/web` 应按如下方式使用协同会话：

1. 有 `canEdit=true` 时，允许挂载编辑器协同绑定与光标广播。
2. 有 `canRead=true` 但 `canEdit=false` 时，允许进入只读协同浏览态。
3. `canUseEditorAI` 由业务 API 决定，但可同步给前端用于提前禁用入口。

## 13. 第一阶段不做事项

第一阶段明确不做：

1. token 自动刷新协议标准化。
2. 多房间联合 presence 聚合。
3. 评论、任务、批注等二级实时对象统一协议。
4. 更复杂的 cursor telemetry 和性能采样。

## 14. 当前结论

DocWeave 第一阶段协同协议的关键不是“字段多不多”，而是先建立一条稳定、可验证、可扩展的协同准入链路：

```text
apps/api session
  -> collaboration token
  -> apps/collab verify
  -> presence / canEdit / room binding
```

只要这条链路清晰，后续再扩展更复杂的房间能力和实时对象都会更顺。
