# 04. 数据库设计

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段的核心表与关系。
2. 若需看权限语义，继续读 [14. 认证与会话实现决策](../decisions/14.%20认证与会话实现决策.md)。
3. 若需看索引状态与快照版本关系，继续读 [25. 搜索与索引任务编排设计](./25.%20搜索与索引任务编排设计.md)。

## 1. 文档目的

本文档用于明确第一阶段应先落哪些表，以及它们之间的核心关系和职责边界。

## 2. 核心结论

第一阶段数据库真相源采用 `PostgreSQL`，至少应覆盖以下领域：

1. 用户与会话。
2. 工作区、空间、文档与成员。
3. 稳定快照。
4. 文件元数据。
5. AI / RAG 配置。
6. 索引任务与索引状态。

## 3. 第一阶段建议表

建议至少包含：

1. `users`
2. `workspaces`
3. `workspace_members`
4. `spaces`
5. `space_members`
6. `documents`
7. `document_members`
8. `document_snapshots`
9. `files`
10. `ai_settings`
11. `rag_settings`
12. `rag_index_jobs`

## 4. 文档与层级模型

建议内容模型如下：

```text
workspace
  -> spaces
    -> documents
```

文档表至少应保留：

1. `workspace_id`
2. `space_id`
3. `parent_document_id`
4. `title`
5. `latest_snapshot_version`
6. `latest_indexed_version`

## 5. 快照模型

`document_snapshots` 至少应保留：

1. `document_id`
2. `version`
3. `content`
4. `created_by`
5. `created_at`

约束如下：

1. 快照是 RAG 与导出的稳定输入。
2. `latest_snapshot_version` 与 `latest_indexed_version` 不能混用。

## 6. 索引任务模型

`rag_index_jobs` 至少应保留：

1. `document_id`
2. `target_snapshot_version`
3. `status`
4. `error_code`
5. `started_at`
6. `finished_at`

## 7. 文件元数据模型

`files` 至少应保留：

1. `file_id`
2. `workspace_id`
3. `uploader_user_id`
4. `object_key`
5. `mime_type`
6. `size`
7. `kind`

约束如下：

1. 对外主标识使用 `file_id`。
2. `object_key` 仅作为存储层内部定位字段。

## 8. 配置模型

第一阶段建议保留：

1. `ai_settings`
2. `rag_settings`

用于承接：

1. 模型提供商与模型名。
2. embedding 配置。
3. topK 等检索参数。

## 9. 第一阶段不做事项

第一阶段不急于落：

1. 评论线程完整模型。
2. 收藏与通知模型。
3. 历史会话持久化模型。
4. 复杂组织 / 组权限模型。

## 10. 当前结论

DocWeave 第一阶段数据库设计的关键不在于“表够不够多”，而在于先支撑以下真相链路：

```text
workspace / space / document
  + snapshot version
  + index job status
  + file metadata
```

只要这条链路成立，协同、AI、RAG 和文件能力就有稳定的业务真相源。
