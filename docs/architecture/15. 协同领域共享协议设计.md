# 15. 协同领域共享协议设计

阅读说明：

1. 本文档负责定义协同领域需要跨 `apps/web`、`apps/api`、`apps/collab` 共享的协议。
2. 本文档聚焦“哪些协同语义必须统一、放在哪一层统一”。
3. 若需看 token 和 Presence 的第一阶段具体结构，继续读 [16. 协同 Token 与 Presence 设计](./16.%20协同%20Token%20与%20Presence%20设计.md)。
4. 若需看通用数据契约与适配层边界，继续读 [05. 数据契约与适配层设计](./05.%20数据契约与适配层设计.md)。

## 1. 文档目的

本文档用于把协同领域的“共同语言”提前收口，避免前端、API 和协同服务各写一套房间命名、token payload、Presence 和连接状态语义。

## 2. 核心结论

DocWeave 应把协同领域共享协议沉淀到独立共享包中，推荐由 `packages/contracts` 承载协议定义，由 `packages/adapters` 和前端 view-model 层消费。

第一阶段至少统一以下几类协议：

1. 房间命名与解析规则。
2. 协同 token payload。
3. 协同能力字段 `capabilities`。
4. Presence / Awareness 最小结构。
5. 协同连接状态枚举。

## 3. 为什么不能各写一套

如果不统一，后续很容易出现：

1. 前端和协同服务对 `roomName` 的理解不一致。
2. API 签发的 token 字段与协同服务校验字段不一致。
3. Presence 字段命名和可展示语义逐步漂移。
4. 页面连接状态、权限状态和协同运行态无法稳定映射。

## 4. 推荐分层方式

协同共享协议建议按下面方式落位：

```text
packages/contracts
  -> 定义协同 DTO、枚举和 schema

packages/adapters
  -> 负责把后端/协同返回值转换为前端可消费结构

apps/api
  -> 生产 token payload 与页面上下文

apps/collab
  -> 校验 token payload 与房间语义

apps/web
  -> 消费连接状态、Presence、capabilities 与 view-model
```

关键边界如下：

1. `contracts` 负责定义“原始协议”。
2. `adapters` 负责兼容与转换。
3. 页面层优先消费整理后的 `view-model`，而不是散落解析协议细节。

## 5. 必须统一的协议主题

第一阶段建议统一以下主题：

1. `room`
2. `collaborationToken`
3. `collaborationCapabilities`
4. `collaborationUser`
5. `collaborationAwareness`
6. `collaborationConnectionStatus`

## 6. 房间命名协议

房间命名不应由多个 app 各自手拼字符串，推荐固定为：

```text
workspace:{workspaceId}:document:{documentId}
```

建议至少有以下共享能力：

1. `buildDocumentRoomName(workspaceId, documentId)`
2. `parseDocumentRoomName(roomName)`

## 7. 协同用户协议

协同用户结构应与主业务用户 DTO 区分，但保留最小映射信息。

建议至少包含：

1. `id`
2. `name`
3. `avatarUrl`
4. `color`
5. `role`

这样做的原因如下：

1. 协同展示只需要最小展示态。
2. 不应把后台完整用户对象直接塞进协同层。
3. `color`、`role` 等协同展示字段应有稳定口径。

## 8. Capability 协议

`capabilities` 用于表达当前连接在当前文档中的协同能力。

第一阶段建议至少包含：

1. `canRead`
2. `canEdit`
3. `canComment`
4. `canUseEditorAI`

关键原则如下：

1. 协同服务只消费能力结果，不重新发明业务权限规则。
2. 前端只依据可信能力字段决定是否允许编辑、评论或触发编辑器 AI。

## 9. Token Payload 协议

协同 token payload 必须是明确可解析的稳定结构，而不是模糊字符串。

建议至少包含：

1. `version`
2. `workspaceId`
3. `documentId`
4. `roomName`
5. `user`
6. `capabilities`
7. `issuedAt`
8. `expiresAt`

## 10. Presence / Awareness 协议

Presence 最容易在多个层级逐渐长歪，因此第一阶段就应统一最小结构。

建议最小字段如下：

1. `user`
2. `canEdit`

后续可扩展但第一阶段不强制的字段包括：

1. `cursor`
2. `selection`
3. `lastActiveAt`

## 11. 连接状态协议

页面和协同运行时建议统一连接状态枚举，避免每个页面自造一套文案和值。

推荐至少包含：

1. `idle`
2. `connecting`
3. `connected`
4. `disconnected`
5. `unauthorized`
6. `error`

## 12. 与业务 API 的边界

协同共享协议不负责定义权限真相，也不负责承载完整页面业务模型。

具体边界如下：

1. 权限真相源仍在 `apps/api`。
2. 协同共享协议只承载被计算后的会话能力。
3. 页面完整业务上下文仍由业务 API DTO 提供。
4. 协同协议只处理“进入实时协同所需的共同语言”。

## 13. 与 `contracts -> adapters -> view-model` 的关系

协同领域也必须遵守当前项目已确认的跨边界规则：

```text
contracts
  -> 定义 DTO / schema / enum
adapters
  -> 兼容后端返回和运行时差异
view-model
  -> 页面最终消费形态
```

这意味着：

1. 不让 React 页面直接散落解析 token 或 Presence 原始结构。
2. 不把协同服务的原始运行时对象直接上抛给 UI。
3. 当后续协议扩展时，优先在 adapter 和 view-model 层收敛影响面。

## 14. 第一阶段最小落地项

第一阶段建议至少完成：

1. 房间命名函数定稿。
2. token payload 结构定稿。
3. capabilities 字段定稿。
4. 连接状态枚举定稿。
5. Presence 展示最小结构定稿。

## 15. 当前结论

协同领域共享协议的价值不在于“多造一个抽象层”，而在于让三端说同一种语言：

1. `apps/api` 签发什么。
2. `apps/collab` 校验什么。
3. `apps/web` 消费什么。

只要这三件事先统一，后续协同能力继续扩展时就不会越做越散。
