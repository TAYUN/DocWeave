# 12. 权限实现设计

阅读说明：

1. 本文档负责把权限模型落实到 `apps/api`、`apps/web`、`apps/collab` 的实现边界。
2. 权限角色与能力语义以 [11. 权限模型设计](./11.%20权限模型设计.md) 为准。
3. 会话与协同 token 基线以 [14. 认证与会话实现决策](../decisions/14.%20认证与会话实现决策.md) 与 [16. 协同 Token 与 Presence 设计](./16.%20协同%20Token%20与%20Presence%20设计.md) 为准。

## 1. 文档目的

本文档用于明确权限在运行时中到底如何被计算、缓存、透传和消费，避免后续把业务权限判断散落在控制器、React 组件、协同服务和 RAG 检索层里。

## 2. 核心结论

DocWeave 的权限实现应遵循以下原则：

1. 权限真相源在 `apps/api`。
2. 页面层只消费 capability 结果，不自行成为权限判定中心。
3. `apps/collab` 只消费短期 token 中的协同能力结果。
4. `packages/rag` 只消费由业务层预先计算的可见范围，不直接判权限。

## 3. 推荐实现分层

推荐按以下层次实现：

```text
session
  -> current user
  -> membership / role resolve
  -> capability resolve
  -> dto / token / response exposure
```

## 4. `apps/api` 的职责

`apps/api` 负责：

1. 读取当前会话。
2. 查询工作区 / 空间 / 文档成员关系。
3. 计算角色与 capability。
4. 在接口层做准入控制。
5. 把结果透传给页面 DTO、协同 token、AI 接口和 RAG 检索入口。

## 5. `apps/web` 的职责

`apps/web` 负责：

1. 根据 capability 渲染可交互状态。
2. 根据 capability 控制按钮可见或 disabled 状态。
3. 根据 capability 控制本地是否尝试发起某些操作。

`apps/web` 不负责：

1. 重新定义权限真相。
2. 只凭前端状态决定是否越权调用接口。

## 6. `apps/collab` 的职责

`apps/collab` 应仅消费协同 token 中的最小能力结果，例如：

1. `canRead`
2. `canEdit`
3. `canComment`
4. `canUseEditorAI`

它不应：

1. 重新查数据库算成员关系
2. 自己定义业务角色语义

## 7. `packages/rag` 与 `packages/ai` 的职责

### 7.1 `packages/rag`

负责：

1. 在已授权的文档范围内执行检索
2. 构建检索上下文

不负责：

1. 自己判断当前用户能看哪些文档

### 7.2 `packages/ai`

负责：

1. 模型调用与流式返回

不负责：

1. 自己判断用户能否触发文档级 AI 能力

## 8. Capability 暴露位置

第一阶段建议至少在以下位置暴露 capability：

1. 文档页元数据 DTO
2. 协同 token payload
3. 搜索 / 聊天接口前置鉴权结果
4. 文件上传与下载接口行为

## 9. 推荐实现流程

### 9.1 页面接口

```text
session
  -> resolve workspace / space / document role
  -> resolve capabilities
  -> return page dto
```

### 9.2 协同 token

```text
session
  -> resolve document capabilities
  -> sign short-lived collaboration token
```

### 9.3 RAG 搜索 / 问答

```text
session
  -> resolve visible documents
  -> pass allowed range to packages/rag
```

## 10. 错误态建议

建议统一以下错误语义：

1. `unauthenticated`
2. `forbidden`
3. `not_found`

重点如下：

1. 页面级无权限与资源不存在要尽量区分清楚。
2. 协同 token 获取失败应是明确可定位的拒绝，而不是模糊网络错误。
3. 搜索与问答无权限时，不应把不可见文档误暴露成检索结果。

## 11. 不建议的做法

需要避免以下问题：

1. 把权限逻辑分散在多个控制器手写判断里
2. 前端根据 role 名称自己推导复杂能力
3. 协同服务或 RAG 层直接变成权限真相源
4. 页面仅靠隐藏按钮替代真实后端鉴权

## 12. 当前结论

DocWeave 的权限实现关键不是“在哪一层都做一点”，而是：

```text
apps/api resolve truth
  -> expose stable capabilities
  -> web / collab / rag consume result
```

只要这个方向稳定，后续扩分享、公开访问和更复杂角色矩阵都会容易很多。
