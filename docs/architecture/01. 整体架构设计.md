# 01. 整体架构设计

阅读说明：

1. 本文档负责回答“系统整体怎么分层、模块边界怎么划”。
2. 最终技术拍板以 [01. 最终技术方案确认](../decisions/01.%20最终技术方案确认.md) 为准。
3. 若需看当前阶段推进顺序，继续读 [00. 当前实施路线图](../planning/00.%20当前实施路线图.md)。

## 1. 整体架构目标

DocWeave 的整体架构需要同时满足以下要求：

1. 支持在线富文本文档编辑。
2. 支持多人实时协同。
3. 支持权限、成员、文件和会话等业务系统能力。
4. 支持编辑器内 AI 与知识库问答并存。
5. 支持 RAG 检索和 Citation 回跳。
6. 支持后续私有化部署和运行时拆分。

## 2. 总体分层

整体采用 Monorepo + 多进程分层架构：

```text
Web 前端应用
  -> 调用主业务 API
  -> 连接协同服务

主业务后端 API
  -> 负责认证、权限、文档元数据、AI/RAG 编排、文件签名 URL

独立协同服务
  -> 负责 Yjs 同步、Awareness、协同 token 接入点

独立 Worker
  -> 负责快照索引、embedding、文件后处理和导出任务

共享包
  -> 提供 editor、ai、rag、auth、database 等领域能力
```

## 3. 应用边界

### 3.1 apps/web

Web 应用负责：

1. 工作台页面。
2. 空间导航与文档树。
3. 文档编辑页。
4. 搜索页与聊天页。
5. 设置页。
6. 编辑器内 AI 触发和结果应用。

Web 应用不负责：

1. 真实业务权限判定。
2. 数据库访问。
3. 协同服务端同步。
4. RAG 检索可信边界判定。

### 3.2 apps/api

API 应用负责：

1. 用户与会话认证。
2. 工作区、空间、文档、成员和权限管理。
3. 文档元数据、树结构与稳定快照语义管理。
4. 编辑器 AI 请求编排与 `@blocknote/xl-ai/server` 宿主接入。
5. RAG 搜索与问答编排。
6. 文件上传元数据和签名 URL。
7. 索引任务记录创建。

API 应用不负责：

1. 承载实时协同 WebSocket。
2. 直接执行长时间 embedding / indexing 任务。
3. 直接渲染聊天和编辑器 UI。

### 3.3 apps/collab

协同服务负责：

1. 文档房间连接管理。
2. Yjs update 同步。
3. Awareness 在线状态同步。
4. 协同 token 的接入与校验扩展点。

协同服务不负责：

1. 普通业务 CRUD。
2. 权限真相源管理。
3. RAG 检索与 AI 问答。

### 3.4 apps/worker

Worker 负责：

1. 从稳定快照读取索引输入。
2. 执行 chunk 切分。
3. 执行 embedding。
4. 向 `Qdrant` 写入或重建索引。
5. 处理文件后处理、导出和重建任务。

Worker 不负责：

1. 对外暴露主业务 HTTP API。
2. 承载实时协同连接。
3. 直接决定业务权限。

## 4. 包边界

### 4.1 packages/ai

负责：

1. `Vercel AI SDK` 统一模型接入。
2. Prompt 组装。
3. 流式输出封装。
4. 为 `@blocknote/xl-ai/server` 提供模型调用能力。
5. 为编辑器 AI 与聊天问答提供可复用的模型调用能力。

不负责：

1. 检索。
2. 权限过滤。
3. 向量库访问。

### 4.2 packages/rag

负责：

1. chunk 策略。
2. 检索 backend 抽象。
3. `Qdrant` 实现。
4. 上下文构建。
5. 问答流程。
6. Citation 元数据结构。

不负责：

1. 通用模型 provider 接入。
2. 主业务权限真相源。
3. 协同运行态同步。

### 4.3 packages/document

负责：

1. 基于 `@blocknote/server-util` 封装快照处理。
2. 基于 `@blocknote/server-util` 封装导出能力。
3. 为 RAG 提供稳定快照的前处理输入。
4. 统一文档 JSON 的服务端解析与转换。

不负责：

1. 模型调用。
2. 向量检索。
3. 权限真相源管理。

## 5. 三态分离原则

系统必须明确区分三种状态：

```text
协同运行态
  -> Yjs document
  -> 用于多人实时编辑

稳定快照态
  -> BlockNote JSON snapshot
  -> 用于导出、版本、索引输入
  -> 服务端优先通过 @blocknote/server-util 处理

检索索引态
  -> chunk + embedding + qdrant payload
  -> 用于 search / chat / citation
```

约束如下：

1. RAG 只吃稳定快照，不直接消费协同运行态。
2. Citation 必须能回到 `documentId + snapshotVersion + blockId`。
3. 编辑器 AI 可以围绕当前选区和局部上下文工作，但不应直接绕过稳定快照语义污染索引链路。

## 6. 核心数据流

### 6.1 文档编辑流

```text
用户打开文档页
  -> Web 获取文档元数据
  -> Web 请求协同 token
  -> Web 连接 collab 服务
  -> BlockNote 绑定 Yjs 文档
  -> 用户进行编辑
  -> 更新通过 Yjs 在房间内同步
```

### 6.2 稳定快照与索引流

```text
用户保存或触发稳定快照
  -> API 创建 snapshotVersion
  -> API 调度 enqueueDocumentIndex(documentId, snapshotVersion)
  -> Worker 领取索引任务
  -> 读取 BlockNote JSON 快照
  -> packages/document 基于 @blocknote/server-util 做前处理
  -> chunk
  -> embedding
  -> upsert 到 Qdrant
  -> 回写 latestIndexedVersion
```

### 6.3 编辑器 AI 流

```text
用户选中文本或通过 Slash 触发 AI
  -> Web 通过 @blocknote/xl-ai 发起 AI 动作
  -> Web 调用 API 中承载的 @blocknote/xl-ai/server 接口
  -> API 校验文档访问权限
  -> packages/ai 调用 Vercel AI SDK
  -> Web 接收流式结果
  -> 用户决定是否应用到编辑器
```

### 6.4 RAG 搜索与问答流

```text
用户在搜索页或聊天页提问
  -> API 先计算可访问文档范围
  -> packages/rag 调 Qdrant 检索
  -> 构建受控上下文
  -> packages/ai 调用模型生成回答
  -> 返回答案与 Citation
  -> 前端根据 documentId + snapshotVersion + blockId 回跳原文
```

## 7. 第一阶段调度策略

第一阶段采用“抽象先行、实现从简”的调度路线。

### 7.1 抽象层

统一保留如下接口：

```ts
interface IndexJobScheduler {
  enqueueDocumentIndex(documentId: string, snapshotVersion: number): Promise<void>
}
```

### 7.2 第一阶段实现

采用数据库 job 记录 + worker 轮询：

1. API 写入 `rag_index_jobs` 记录。
2. Worker 轮询领取 `pending` 任务。
3. Worker 执行索引并回写状态。

### 7.3 第二阶段实现

待主链路稳定后，再替换为：

1. `BullMQ` enqueue 写入 `Redis` queue。
2. Worker 作为 `BullMQ consumer` 消费任务。

这样可以保证：

1. 上层业务不依赖具体队列实现。
2. 第一阶段不被基础设施复杂度拖慢。

## 8. 部署形态

当前推荐部署如下：

```text
apps/web
  -> 静态站点 / Nginx

apps/api
  -> 单独容器

apps/collab
  -> 单独容器

apps/worker
  -> 单独容器

PostgreSQL / Redis / Qdrant / MinIO
  -> 独立服务
```

当前部署策略：

1. 生产优先采用 `Docker Compose` 起步。
2. 后续再评估迁移到 `Kubernetes`。

## 9. 当前结论

DocWeave 当前采用的是“React SPA + AdonisJS + BlockNote 官方 AI / 文档服务端生态 + 独立协同服务 + 独立 Worker + Qdrant + PostgreSQL”的分层架构。

这套架构的核心价值不在于堆更多技术，而在于：

1. 运行时边界清晰。
2. 协同、AI、RAG 三条链路互不混淆。
3. 第一阶段可以按最小闭环推进。
4. 后续替换调度、聊天 UI、对象存储提供商时，不需要推翻主架构。
