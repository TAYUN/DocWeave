# 34. 设计系统与 Token 工作流

阅读说明：

1. 本文档负责定义 DocWeave 中设计规范、主题 token、实现入口和生成流程之间的关系。
2. 设计规范总源头以根目录 [DESIGN.md](D:/code-my/DocWeave/DESIGN.md) 为准。
3. 前端实现规范以 [frontend-mantine-implementation-guide](D:/code-my/DocWeave/docs/workflow/frontend-mantine-implementation-guide.md) 为准。

## 1. 文档目的

本文档用于把 DocWeave 现有的设计系统相关事实再收口一次，明确：

1. 哪份文档是设计真相源
2. 哪些文件承载 theme / token / 组件实现
3. 设计变更应该按什么顺序流转
4. 如何避免 `DESIGN.md`、`theme.ts`、页面实现逐渐分叉

## 2. 核心结论

DocWeave 当前设计系统与 token 工作流建议按以下方式理解：

1. `DESIGN.md` 是视觉语言和设计判断的单一来源。
2. `apps/web/src/theme.ts` 是当前 `Mantine` 默认主题值的显式事实入口。
3. 页面与组件实现应消费 `Mantine` 主题语义，而不是各自散写私有视觉常量。
4. 设计变更必须先改设计规范，再改实现入口，最后回到页面核对。

## 3. 为什么需要单独写这份文档

如果不把这条链路单独讲清楚，后续很容易出现：

1. `DESIGN.md` 说一套，`theme.ts` 写一套
2. 页面实现直接绕过主题层硬编码样式
3. 团队把“当前 Mantine 默认值”和“项目自定义 token”混成一个概念
4. 设计讨论停留在抽象描述，没有落到具体实现入口

## 4. 设计系统的层次划分

当前建议分成四层：

```text
DESIGN.md
  -> 定义视觉气质、颜色角色、层级与状态语义

apps/web/src/theme.ts
  -> 显式承载当前 Mantine 默认主题值与必要扩展

组件 / 页面实现
  -> 消费主题语义、组件样式和布局约束

UI 文档与收口清单
  -> 核对页面规格、实现和设计规范是否一致
```

## 5. 各层职责

### 5.1 `DESIGN.md`

负责：

1. 定义产品视觉气质
2. 定义颜色角色与状态语义
3. 定义字体、留白、层级和交互状态原则

不负责：

1. 指定组件 props 怎么传
2. 指定某个页面写哪段 CSS
3. 代替 `theme.ts` 承载代码可读的默认值

### 5.2 `apps/web/src/theme.ts`

负责：

1. 让当前主题默认值显式可查
2. 承载必要的 `Mantine` 主题配置入口
3. 为页面和组件实现提供统一主题消费点

不负责：

1. 变成第二份设计规范文档
2. 随意发明与 `DESIGN.md` 不一致的视觉体系

### 5.3 页面与组件实现

负责：

1. 消费设计语义
2. 落地布局、状态和组件层级
3. 避免硬编码破坏主题一致性

## 6. Token 的当前定义方式

DocWeave 当前并不是“完全自定义 token 系统”，而是：

1. 视觉判断优先继承 `Mantine` 官方默认主题
2. 当前安装版本的默认值允许在 `theme.ts` 中显式展开
3. 项目自定义只应发生在有明确业务设计理由的地方

换句话说：

1. 现在的 token 更偏“官方默认值显式化”
2. 而不是“一开始就重造一整套品牌 token 体系”

## 7. 设计变更工作流

当需要调整设计方向或 token 时，建议按以下顺序执行：

```text
先确认设计目标
  -> 更新 DESIGN.md
  -> 更新 theme.ts 中对应主题入口
  -> 回看组件和页面实现
  -> 用 UI 收口清单核对一致性
```

这条顺序的意义在于：

1. 先有设计判断，再改代码
2. 避免直接在实现层偷偷变更视觉语义
3. 保证设计层和实现层有可追踪关系

## 8. 与 Mantine 的关系

当前设计系统与 `Mantine` 的关系建议理解为：

1. `Mantine` 是基础主题与组件语义承载层
2. DocWeave 不是要完全绕过 `Mantine`
3. DocWeave 也不应把 `Mantine` 默认主题机械照搬到毫无产品判断

因此，正确做法是：

1. 以 `Mantine` 默认语义为基线
2. 通过 `DESIGN.md` 补足产品层判断
3. 通过 `theme.ts` 提供实现层统一入口

## 9. 与 Tailwind 的关系

当前项目同时使用 `Tailwind CSS v4`，但它不应变成第二套平行主题系统。

建议原则如下：

1. Tailwind 主要承担布局、间距和局部样式辅助
2. 主题语义仍应围绕 `Mantine` 与设计规范统一
3. 不要在 Tailwind 类名里再硬编码另一套品牌色体系

## 10. 与 UI 文档的关系

以下文档负责从不同角度保证设计系统落地：

1. `DESIGN.md`
   - 定义视觉语言
2. `frontend-mantine-implementation-guide`
   - 约束实现方式
3. `ui/page-inventory.md` 与单页规格
   - 约束页面语义与结构
4. `planning/46. UI 页面规格与实现一致性收口清单`
   - 约束最终一致性核对

## 11. 不建议的做法

需要避免以下问题：

1. 改页面样式时只改组件，不回写设计规范
2. `DESIGN.md` 长期不更新，但 `theme.ts` 和页面实现已经大量偏离
3. Mantine、Tailwind、页面私有 CSS 三套视觉语义并行生长
4. 一开始就过度抽象出复杂 token pipeline，但没有真实维护能力

## 12. 当前结论

DocWeave 当前最合理的设计系统工作流不是“先造一套庞大 token 工厂”，而是先把这条链路跑顺：

```text
DESIGN.md
  -> theme.ts
  -> component / page implementation
  -> ui consistency review
```

只要这条链稳定，后续无论要不要扩成更完整的 token 系统，成本都会低很多。
