# 30. 文件分享与公开资源访问设计

阅读说明：

1. 本文档负责把分享访问策略展开为文件与公开资源的实际访问链路设计。
2. 分享策略以 [26. 分享访问与公开链接策略](../decisions/26.%20分享访问与公开链接策略.md) 为准。
3. 文件访问基线以 [15. 文件存储与上传策略](../decisions/15.%20文件存储与上传策略.md) 为准。

## 1. 文档目的

本文档用于明确公开分享页中的图片、附件、导出文件等资源如何被安全访问，避免后续实现分享页时为了省事直接把对象存储变成公开读，破坏整个文件权限基线。

## 2. 核心结论

DocWeave 的公开分享资源访问建议采用以下路线：

1. 分享页本身是公开只读页面，但资源访问仍通过受控后端链路。
2. `apps/api` 必须根据 `shareId`、分享状态和资源归属关系决定是否允许访问。
3. 即使资源来源于公开分享文档，也优先签发短期 URL，而不是暴露长期公开对象地址。
4. 资源访问与文档分享应共享同一分享对象语义，而不是各自发明另一套匿名访问规则。

## 3. 访问链路

推荐访问链路如下：

```text
用户打开 share page
  -> apps/api 校验 shareId
  -> 返回公开页面 DTO
  -> 页面请求图片 / 附件 / 导出文件
  -> apps/api 再基于 share context 校验资源可见性
  -> 签发短期 URL
  -> 浏览器拉取对象内容
```

## 4. 为什么不直接公开对象存储

不建议直接公开对象存储的原因如下：

1. 分享撤销后资源难以立即失效。
2. 很难区分哪些资源属于公开分享，哪些仍属私有。
3. 未来若引入密码分享、成员限定分享，会立即和公开桶模式冲突。
4. `objectKey` 一旦泄露，后续安全边界会被持续侵蚀。

## 5. 资源对象分层

建议至少区分以下资源：

1. 文档正文中的图片
2. 文档附件
3. 导出文件
4. 后续分享页专用预生成缩略图或预览资源

不同资源可复用同一访问基线，但不应假设生命周期完全一致。

## 6. `apps/api` 的职责

在公开资源访问场景中，`apps/api` 负责：

1. 校验分享链接是否有效
2. 判断资源是否属于该分享文档或该分享版本
3. 决定资源是否允许在公开上下文下被访问
4. 返回短期可访问 URL

## 7. 分享与版本的关系

分享资源访问不应脱离版本语义。

推荐原则如下：

1. 如果分享页读取的是已发布版本，则资源访问也应围绕该版本关联的资源集合。
2. 不应因为正文是旧发布版本，就误把新草稿里新增的私有附件也公开出来。
3. 导出文件如果绑定特定快照或发布版本，也应沿同一版本链路访问。

## 8. 前端页面行为建议

分享页前端建议遵循：

1. 页面只消费公开 DTO 和受控资源 URL
2. 不自己拼对象存储路径
3. 资源失效时展示可理解的缺失态或权限态
4. 不把分享页上的资源访问缓存成长效真相

## 9. 后续可扩展方向

在当前基线成立后，后续可以扩展：

1. 密码保护分享资源访问
2. 成员限定分享资源访问
3. 资源级访问次数统计
4. 分享撤销后的资源即时失效策略

## 10. 不建议的做法

需要避免以下做法：

1. 公开分享页直接返回原始对象地址
2. 分享页和登录态页面共用一套“猜测式”资源访问逻辑
3. 只校验文档分享，不校验具体资源是否属于当前分享上下文
4. 公开分享时顺手把整桶资源改成公开读

## 11. 当前结论

DocWeave 中的“公开分享”不应理解为“所有相关资源都变公开”，而应理解为：

```text
share context
  -> api verify
  -> resource-level allow
  -> short-lived access url
```

只有这样，公开阅读体验和文件权限基线才不会彼此打架。
