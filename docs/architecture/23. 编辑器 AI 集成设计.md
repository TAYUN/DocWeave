# 23. 编辑器 AI 集成设计

阅读说明：

1. 本文档负责定义编辑器 AI 在前端交互、后端协议和结果应用层的集成设计。
2. AI 模块基线以 [20. AI 模块设计](./20.%20AI%20模块设计.md) 为准。
3. 编辑器与 BlockNote 集成以 [21. BlockNote 编辑器集成设计](./21.%20BlockNote%20编辑器集成设计.md) 为准。

## 1. 文档目的

本文档用于把“编辑器 AI”从通用 AI 模块里单独展开，明确前端入口、后端协议、局部上下文、结果应用和权限态处理，避免后续把聊天 AI 和编辑器局部动作混成同一套交互。

## 2. 核心结论

DocWeave 的编辑器 AI 建议采用以下路线：

1. 前端优先贴近 `@blocknote/xl-ai` 的官方交互模型
2. 后端由 `apps/api` 宿主 `@blocknote/xl-ai/server`
3. 模型能力统一通过 `packages/ai`
4. 编辑器 AI 结果必须经过“应用 / 放弃”分叉，而不是直接静默写回正文

## 3. 交互入口

第一阶段建议至少支持以下入口：

1. 选区入口
2. Slash 入口

后续可扩展：

1. 工具栏入口
2. 右键菜单入口
3. 侧边 Copilot 面板入口

## 4. 动作集合建议

第一阶段建议至少覆盖：

1. 改写
2. 扩写
3. 缩写
4. 翻译
5. 总结

这些动作的共同特点如下：

1. 目标都围绕当前局部内容
2. 输出都需要用户确认后再回写

## 5. 上下文构建建议

编辑器 AI 上下文建议由以下部分组成：

1. 当前选区内容
2. 当前文档必要上下文
3. 当前用户能力边界
4. 当前动作类型

关键原则如下：

1. 不把整篇文档无上限塞进 Prompt
2. 不直接依赖浏览器任意状态作为可信业务上下文
3. 文档权限和可编辑能力必须由后端再次确认

## 6. 返回结果与应用逻辑

编辑器 AI 返回结果建议遵循：

1. 结果先以预览态呈现
2. 用户显式选择应用或放弃
3. 应用后再进入正文协同链路

这样做的原因如下：

1. 减少误写正文
2. 保持用户可控
3. 便于在协同状态下处理 AI 结果的落位

## 7. 权限与只读态

编辑器 AI 建议遵循：

1. 无文档读取权限则不能调用
2. 只读用户可按产品策略决定是否允许“生成建议”，但第一阶段不允许应用结果
3. 可编辑用户才允许真正回写正文

## 8. 前后端链路

推荐链路如下：

```text
editor selection / slash action
  -> @blocknote/xl-ai
  -> apps/api hosted @blocknote/xl-ai/server
  -> permission + context build
  -> packages/ai runtime
  -> stream result back
  -> user apply or discard
```

## 8.1 模型工具调用兼容约束

编辑器 AI 依赖 `@blocknote/xl-ai` 的 `toolChoice: required`，模型运行时必须保证该约束与供应商参数兼容。

当前 `qwen3.6-plus` 在 DashScope 思考模式下不接受 `tool_choice: required`，因此编辑器 AI runtime 通过 DashScope OpenAI-compatible transport 注入 `enable_thinking: false`。该设置只作用于编辑器 AI，不代表所有聊天或 RAG 场景都必须关闭思考模式。

## 9. 与协同的关系

编辑器 AI 结果应用后，本质上仍然是一种正文修改，因此：

1. 应用结果后要进入正常协同同步链路
2. AI 结果不应绕过协同层直接制造另一套正文真相
3. 只读协同态不应允许回写结果

## 10. 错误态建议

建议至少区分：

1. 权限失败
2. 模型调用失败
3. 流式中断
4. 应用结果失败

用户可感知层面应做到：

1. 错误可理解
2. 放弃结果后编辑器仍可继续工作
3. 不把失败态卡死在页面里

## 11. 不建议的做法

需要避免以下问题：

1. 编辑器 AI 直接自动写回正文
2. 把聊天多轮状态和局部 AI 动作混成同一 UI
3. 后端不校验权限只信任前端动作类型
4. 失败后破坏当前编辑器内容或交互状态

## 12. 当前结论

DocWeave 的编辑器 AI 应被理解为一条局部内容增强链路，而不是“把聊天 UI 塞进编辑器”：

```text
local context
  -> ai action
  -> streamed suggestion
  -> user-controlled apply
```

只要这条链路先做稳，后续再扩更复杂的编辑器 AI 能力才有意义。

## 13. 协同应用安全门禁

官方 AIExtension 负责 block 操作、review、accept、reject 和 abort；DocWeave 在应用前负责再次确认当前 editor 实例未销毁、documentId 未变化、用户仍具备编辑能力、协同连接允许应用且目标 block 仍存在。任何检查失败都只能拒绝应用或重新生成，不能通过 API 直接写入 `documents.content`。
