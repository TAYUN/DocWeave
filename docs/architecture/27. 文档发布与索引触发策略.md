# 27. 文档发布与索引触发策略

阅读说明：

1. 本文档负责定义 DocWeave 中“保存快照”“发布内容”“触发索引”三者之间的关系。
2. 本文档聚焦“什么时候触发索引、发布读哪版、哪些动作不应混为一谈”。
3. 若需看稳定快照语义，继续读 [24. 版本与稳定快照设计](./24.%20版本与稳定快照设计.md)。
4. 若需看索引任务编排，继续读 [25. 搜索与索引任务编排设计](./25.%20搜索与索引任务编排设计.md)。

## 1. 文档目的

本文档用于明确文档编辑、稳定快照、发布状态和索引触发之间的边界，避免后续把“用户点保存”“知识库索引更新”“公开分享读取哪版内容”混成一个不清晰的动作。

## 2. 核心结论

DocWeave 应明确区分以下动作：

1. 保存协同内容为稳定快照
2. 触发索引任务
3. 将某一版内容标记为已发布

推荐原则如下：

1. 保存不等于发布。
2. 发布不一定立即索引，但应具备明确触发策略。
3. 索引必须围绕稳定快照版本触发，而不是围绕浏览器实时态触发。
4. 后续公开分享与对外阅读优先围绕已发布版本或显式分享版本。

## 3. 为什么要拆开

如果不拆开，后续会出现：

1. 用户只是暂存修改，却误以为外部搜索已更新。
2. 未正式发布的内部草稿进入公开分享或知识库问答结果。
3. 索引任务被每次编辑频繁触发，成本和状态都失控。
4. 搜索命中的内容与公开页显示版本不一致。

## 4. 推荐状态语义

文档层建议至少存在以下版本相关概念：

1. `latestSnapshotVersion`
2. `latestIndexedVersion`
3. `latestPublishedVersion`

如果当前阶段先不全部实现成真实字段，也应至少统一这三层语义。

## 5. 三类动作的推荐含义

### 5.1 保存快照

表示：

1. 将当前协同内容沉淀为稳定版本。
2. 可供后续索引、导出和回溯。
3. 不自动代表内容对外可见。

### 5.2 触发索引

表示：

1. 选择某个稳定快照版本进入检索链路。
2. 最终更新 `latestIndexedVersion`。
3. 只影响搜索 / RAG 可见版本，不直接等于公开发布。

### 5.3 发布内容

表示：

1. 选定某一版稳定内容作为外部阅读或明确展示版本。
2. 后续公开分享、只读公开页、部分导出能力可优先读取该版本。
3. 发布后的版本通常应要求具备稳定快照，必要时再要求已完成索引。

## 6. 第一阶段推荐触发方式

第一阶段建议先采用简单、可解释的触发策略：

1. 用户点击“保存并更新知识库”时：
   - 先生成稳定快照
   - 再创建索引任务
2. 普通协同编辑不自动触发索引
3. “发布”动作后续阶段再明确接入，但文档模型需为其预留语义

这样做的原因如下：

1. 最小闭环更容易验证。
2. 可以避免索引任务过于频繁。
3. 有利于先把快照和索引一致性做稳。

## 7. 后续阶段推荐触发方式

当后续进入发布与分享阶段后，可扩展为：

1. 手动仅保存快照
2. 手动保存并索引
3. 手动发布某一快照版本
4. 发布时自动触发索引
5. 定时或空闲窗口重建索引

但这些扩展都应建立在“保存 / 索引 / 发布”语义已拆开的前提上。

## 8. 与分享访问的关系

分享访问后续推荐优先读取：

1. `latestPublishedVersion`
2. 或分享对象显式绑定的 `targetSnapshotVersion`

不建议默认读取：

1. 协同实时态
2. 最新但未发布的草稿快照

## 9. 与 RAG 的关系

RAG 与搜索只应读取当前已成功索引版本。

推荐关系如下：

```text
latestSnapshotVersion
  -> 可用于后续索引输入

latestIndexedVersion
  -> 可用于检索与问答

latestPublishedVersion
  -> 可用于公开阅读与分享
```

必要时三者可以相同，但语义上不要强行等同。

## 10. API 语义建议

后续建议 API 至少支持以下语义：

1. `POST /api/documents/:documentId/snapshots`
   - 显式生成稳定快照
2. `POST /api/documents/:documentId/index`
   - 基于某个快照触发索引
3. `POST /api/documents/:documentId/publish`
   - 发布某个稳定快照版本
4. `GET /api/documents/:documentId/status`
   - 返回快照、索引、发布三个维度的当前状态

## 11. 不建议的做法

需要避免以下错误方向：

1. 每次按键或每次协同更新都自动触发索引。
2. 让前端自行决定某版内容是否已发布。
3. 公开分享直接读取最新协同内容。
4. 把“保存并索引”当成永远唯一动作，导致后续无法拆分。

## 12. 当前结论

DocWeave 后续若要同时支撑协同编辑、稳定知识库、公开阅读和分享访问，就必须先把这条链路拆清楚：

```text
live collaboration
  -> stable snapshot
  -> index job
  -> published version
```

只有这样，保存、搜索、分享和导出才不会彼此串味。
