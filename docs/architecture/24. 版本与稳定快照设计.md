# 24. 版本与稳定快照设计

阅读说明：

1. 本文档负责定义协同实时态、稳定快照和索引版本之间的关系。
2. 本文档聚焦“内容什么时候算稳定、RAG 应该读哪一版、版本字段怎么表达”。
3. 若需看索引任务链路，继续读 [25. 搜索与索引任务编排设计](./25.%20搜索与索引任务编排设计.md)。
4. 若需看最终技术总基线，优先读 [01. 最终技术方案确认](../decisions/01.%20最终技术方案确认.md)。

## 1. 文档目的

本文档用于明确协同编辑态、稳定正文快照、历史版本与 RAG 索引之间的关系，避免后续系统出现“用户正在看的正文、搜索结果、AI 回答来源彼此不一致”的问题。

## 2. 核心结论

当前项目必须明确区分三种状态：

1. 协同编辑态正文。
2. 稳定快照版本。
3. 已完成索引版本。

推荐关系如下：

```text
Yjs 协同实时内容
  -> 生成稳定快照 snapshotVersion
  -> 基于稳定快照触发索引
  -> 生成 latestIndexedVersion
```

## 3. 为什么必须分层

如果不区分这三层，会出现：

1. 用户刚输入的内容还没稳定落地，RAG 已经读不到。
2. 搜索返回的内容和当前编辑器看到的内容不一致。
3. AI 回答引用的版本无法追踪。

## 4. 三层内容状态

### 4.1 协同编辑态

指当前 `Yjs` 文档在协同服务中的实时状态。

### 4.2 稳定快照

指从协同正文中提取出的某一时刻稳定内容版本。

### 4.3 已索引版本

指已经完成 chunk、embedding、写入检索后端的版本。

## 5. 推荐版本语义

建议使用以下概念：

1. `liveVersion`
2. `snapshotVersion`
3. `latestIndexedVersion`

第一阶段不一定全部落成最终字段名，但语义必须先统一。

## 6. 谁来生成快照

推荐由系统服务端生成快照，而不是浏览器直接决定最终快照真相。

建议职责分工：

1. `apps/collab`
   - 维护实时协同内容
2. `apps/api`
   - 触发快照记录与索引任务
3. `packages/document`
   - 统一处理稳定快照内容
4. `packages/rag`
   - 只消费稳定快照内容

## 7. 当前结论

版本系统必须从一开始就不仅仅被看成“历史功能”，而要被当成：

1. 协同内容稳定落地机制。
2. RAG 一致性基础。
3. 分享与导出基础。
4. 后续历史版本能力的基石。
