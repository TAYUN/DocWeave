# 29. 导入导出与格式转换设计

阅读说明：

1. 本文档负责把导入导出策略展开为可执行的系统边界与运行时分工。
2. 导入导出策略基线以 [27. 文件导入与导出策略](../decisions/27.%20文件导入与导出策略.md) 为准。
3. BlockNote 服务端处理边界以 [22. BlockNote 服务端能力接入设计](./22.%20BlockNote%20服务端能力接入设计.md) 为准。

## 1. 文档目的

本文档用于明确 DocWeave 中导入、导出和格式转换链路的系统分层、任务化边界和版本关系，避免后续把“导出一个文件”“导入一个 Markdown”“生成 PDF”都塞进控制器里临时处理。

## 2. 核心结论

DocWeave 的导入导出与格式转换链路应按以下方式组织：

1. `apps/api` 负责权限、入口和任务记录。
2. `packages/document` 负责围绕 BlockNote / 稳定快照的标准化转换能力。
3. `apps/worker` 负责重型导入解析、导出生成和中间产物清理。
4. 导入与导出都应围绕“稳定可追踪输入”运行，而不是围绕浏览器临时态运行。

## 3. 两条主链路

推荐分为两条独立主链路：

### 3.1 导入链路

```text
原始文件上传
  -> apps/api 记录 file metadata
  -> apps/worker 解析 / 标准化
  -> packages/document 转成 BlockNote 可管理结构
  -> 生成新文档初始内容或新稳定快照
```

### 3.2 导出链路

```text
用户请求导出
  -> apps/api 校验权限与目标版本
  -> packages/document 读取稳定快照
  -> apps/worker 执行格式转换
  -> 导出文件写入对象存储
  -> apps/api 返回下载入口
```

## 4. 推荐输入真相

### 4.1 导入输入

导入输入是真实外部文件，但转换后的业务真相不应继续停留在原始格式。

推荐原则如下：

1. 原始文件只作为来源材料。
2. 转换后的结构化内容才是 DocWeave 管理对象。
3. 必要时保留原始文件与转换结果的映射关系。

### 4.2 导出输入

导出输入不应直接读取协同运行态，而应读取：

1. 稳定快照
2. 或后续的已发布版本

## 5. `packages/document` 的职责

在导入导出链路中，`packages/document` 建议承担：

1. BlockNote JSON 标准化读取
2. Markdown / HTML 等近结构化格式转换适配
3. 导出前的文档结构预处理
4. 统一输出给 worker 使用的中间表示

这层的价值在于：

1. 避免转换逻辑分散在 `apps/api` 和 `apps/worker`
2. 让快照、RAG、导出前处理共用同一套文档服务端能力

## 6. `apps/worker` 的职责

建议把以下能力明确放进 `apps/worker`：

1. 大文件导入解析
2. PDF 导出
3. 批量导出
4. 需要图片、样式或附件预处理的格式转换
5. 中间文件清理与失败重试

## 7. 任务模型建议

当链路开始任务化后，建议至少区分：

1. `document_import_jobs`
2. `document_export_jobs`

建议最小状态包括：

1. `pending`
2. `running`
3. `succeeded`
4. `failed`
5. `canceled`

## 8. 导入输出建议

导入结果建议至少沉淀以下信息：

1. `sourceFileId`
2. `targetDocumentId`
3. `conversionStatus`
4. `conversionFormat`
5. `snapshotVersion` 或等价内容版本引用

## 9. 导出输出建议

导出结果建议至少沉淀以下信息：

1. `exportFileId`
2. `documentId`
3. `snapshotVersion` 或 `publishedVersion`
4. `exportFormat`
5. `generatedAt`

## 10. 不建议的做法

需要避免以下问题：

1. API 同步阻塞执行重型导入导出
2. 浏览器直接负责复杂格式转换
3. 导出产物不记录来源版本
4. 导入成功后无法追踪原始文件与转换结果关系

## 11. 当前结论

DocWeave 的导入导出与格式转换，应被看成一条受控的文档服务端处理链路：

```text
raw file / stable snapshot
  -> packages/document normalize
  -> worker convert
  -> file metadata + version relation
```

只要这条链路稳定，后续再扩格式、补批量任务或接入更复杂转换都会容易很多。
