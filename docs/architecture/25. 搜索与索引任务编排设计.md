# 25. 搜索与索引任务编排设计

阅读说明：

1. 本文档负责定义 DocWeave 从稳定快照到可检索内容之间的任务编排和状态链路。
2. 本文档聚焦“谁触发索引、谁执行索引、谁回写状态、如何避免版本覆盖”。
3. 若需看 RAG 输入与 Citation 结构，继续读 [26. RAG 与 Citation 设计](./26.%20RAG%20与%20Citation%20设计.md)。
4. 若需看最终技术总基线，优先读 [01. 最终技术方案确认](../decisions/01.%20最终技术方案确认.md)。

## 1. 文档目的

本文档用于明确 DocWeave 从协同正文到知识库可检索内容之间的任务链路、状态语义、失败处理与模块边界。

本文档重点回答以下问题：

1. 文档内容从哪里进入索引流程。
2. 为什么必须基于稳定快照而不是实时协同态直接索引。
3. 索引任务由谁触发、谁执行、谁更新状态。
4. 如何避免旧任务覆盖新版本。
5. 搜索、问答和索引状态如何保持一致。

## 2. 核心结论

DocWeave 中的搜索与索引流程应围绕“稳定快照版本”编排，而不是围绕浏览器当前编辑态编排。

推荐主链路如下：

```text
协同编辑态
  -> 生成稳定快照 snapshotVersion
  -> 创建索引任务
  -> packages/document 基于 @blocknote/server-util 做前处理
  -> chunk
  -> embedding
  -> 写入 Qdrant
  -> 原子切换 latestIndexedVersion
```

## 3. 为什么要单独设计任务编排

如果没有清晰的索引编排层，后续容易出现以下问题：

1. 前端不知道“保存”和“建立索引”分别完成到了哪一步。
2. 同一文档连续触发多次索引时，旧任务可能回写新状态。
3. 检索结果无法追踪对应正文版本。
4. 回答引用和当前编辑器内容出现版本错位。
5. 出错后难以判断失败发生在快照、切块、向量化还是写库阶段。

因此，这份文档的重点不是底层向量算法，而是任务生命周期设计。

## 4. 任务链路总览

### 4.1 推荐分层

推荐按如下层次组织：

```text
apps/api
  -> 接收“保存并更新知识库”等业务触发
  -> 生成快照与索引任务记录

packages/document
  -> 承担稳定快照的文档服务端前处理

packages/rag
  -> 承担切块、向量化、写入后端、检索运行时

apps/worker
  -> 领取任务并执行索引链路

Qdrant
  -> 承担向量检索副本
```

### 4.2 第一阶段推荐执行模式

第一阶段采用“数据库任务记录 + 独立 worker 轮询”模式：

1. `apps/api` 创建任务。
2. `apps/worker` 轮询领取任务。
3. 任务状态先落 PostgreSQL。

后续再演进到：

1. `BullMQ`
2. 重试与调度中心
3. 更复杂的后台任务治理

## 5. 输入与输出边界

### 5.1 输入

索引任务输入不应是“前端提交的一段随意正文”，而应是：

1. `documentId`
2. `targetSnapshotVersion`
3. 对应快照内容或其稳定引用

### 5.2 输出

索引任务输出至少应包括：

1. 已写入的 chunk 集合
2. 对应 embedding 数据
3. 最新成功的 `latestIndexedVersion`
4. 任务执行状态
5. 失败原因或错误码

## 6. 触发方式

### 6.1 第一阶段推荐触发方式

第一阶段建议明确以手动触发为主：

1. 用户点击“保存并更新知识库”。
2. 服务端先生成稳定快照。
3. 再创建索引任务。

### 6.2 后续可扩展方式

后续可增加：

1. 单独“仅保存快照”。
2. 空闲期自动索引。
3. 定时批量重建索引。
4. 文档发布时自动触发索引。

第一阶段不建议把触发策略做得过于复杂。

## 7. 推荐任务状态模型

建议 `rag_index_jobs` 至少包含以下状态：

1. `pending`
2. `running`
3. `succeeded`
4. `failed`
5. `canceled`

建议同时记录以下阶段语义：

1. `snapshot_ready`
2. `preprocessing`
3. `chunking`
4. `embedding`
5. `upserting`
6. `publishing`

第一阶段即使不把阶段拆成独立字段，也建议至少在日志和错误结构里保留这些语义。

## 8. 版本一致性设计

### 8.1 关键原则

`latestIndexedVersion` 只能在整条链路成功完成后更新。

也就是说：

1. 不能在 chunk 切分完成时就更新 `latestIndexedVersion`。
2. 不能在 embedding 成功但向量未完整写入时提前发布。
3. 必须在“可检索副本已完整可用”后再切换。

### 8.2 原子发布原则

推荐语义如下：

```text
任务执行成功
  -> 所有 chunk 与向量写入完成
  -> 文档 latestIndexedVersion 更新为 targetSnapshotVersion
  -> 任务标记 succeeded
```

### 8.3 防止旧任务覆盖新状态

如果同一文档先后触发多个任务：

1. 每个任务都必须绑定 `targetSnapshotVersion`。
2. 旧任务成功时必须判断是否仍允许发布。
3. 若发现当前文档已有更高版本任务成功，可将旧任务标记为过期完成而不回写最新状态。

## 9. 前处理、切块与元数据要求

索引不应只保存纯文本，还应尽量保留结构化元数据。

第一阶段推荐链路如下：

```text
BlockNote JSON snapshot
  -> packages/document
  -> @blocknote/server-util
  -> plain text / heading / block structure
  -> packages/rag chunk builder
```

推荐 chunk 元数据至少包含：

1. `documentId`
2. `workspaceId`
3. `spaceId`
4. `snapshotVersion`
5. `chunkIndex`
6. `blockId`
7. `headingPath`

这些字段的价值在于：

1. 检索结果可追溯。
2. Citation 可定位。
3. 权限过滤和页面跳转更稳定。

## 10. 与检索阶段的关系

搜索和问答读取的必须是“当前活跃已索引版本”。

推荐检索前提如下：

1. 文档存在。
2. 用户有访问权限。
3. 文档存在有效 `latestIndexedVersion`。
4. 检索候选仅来自当前激活版本的数据。

这样可以避免：

1. 旧快照残留结果泄漏。
2. 同一文档多个版本混查。
3. 回答引用到未发布版本。

## 11. 与权限系统的关系

索引任务本身不负责定义权限，但必须携带足够的业务字段以支撑检索前过滤。

推荐原则：

1. 权限真相源仍在 PostgreSQL。
2. 检索前先由业务层计算 `allowedDocumentIds` 或等价可见范围。
3. Qdrant payload 只作为辅助过滤字段，不作为最终权限真相。

## 12. 第一阶段调度实现

### 12.1 调度抽象

第一阶段保持如下抽象：

```ts
interface IndexJobScheduler {
  enqueueDocumentIndex(documentId: string, snapshotVersion: number): Promise<void>
}
```

### 12.2 当前实现

当前实现建议如下：

1. `apps/api` 写入 `rag_index_jobs` 数据库记录。
2. `apps/worker` 轮询读取 `pending` 任务。
3. `apps/worker` 加锁或占用任务后执行。
4. `apps/worker` 回写成功、失败和版本切换状态。

### 12.3 后续实现

第二阶段再替换为：

1. `BullMQ enqueue -> Redis queue`
2. `BullMQ consumer -> apps/worker`

上层业务不应因为替换调度方式而变化。

## 13. 失败处理建议

### 13.1 可预期失败类型

至少要区分以下失败：

1. 快照不存在。
2. 文档不存在。
3. 前处理失败。
4. 切块失败。
5. embedding 调用失败。
6. 向量写入失败。
7. 发布状态回写失败。

### 13.2 失败后的状态

失败时应满足：

1. 当前 `latestIndexedVersion` 不回退。
2. 当前线上可检索版本继续保持可用。
3. 任务记录明确标记失败原因。

也就是说，失败不应破坏上一版已可用索引。

## 14. API 语义建议

围绕索引任务建议保留以下接口语义：

1. `POST /api/documents/:documentId/index`
   - 基于当前稳定快照触发索引
2. `GET /api/documents/:documentId/index-status`
   - 返回当前索引状态与最近任务
3. `GET /api/documents/:documentId/snapshots`
   - 返回稳定快照列表

必要时也可扩展：

1. `POST /api/documents/:documentId/snapshots`
   - 显式生成稳定快照

## 15. 推荐数据库字段联动

建议以下表之间保持明确关系：

1. `documents`
   - `latestSnapshotVersion`
   - `latestIndexedVersion`
2. `document_snapshots`
   - `documentId`
   - `version`
   - `content`
3. `rag_index_jobs`
   - `documentId`
   - `targetSnapshotVersion`
   - `status`
   - `errorCode`
   - `startedAt`
   - `finishedAt`

## 16. 第一阶段实施建议

第一阶段建议至少做到：

1. 建立“快照版本 -> 索引任务 -> latestIndexedVersion”主语义。
2. 索引任务记录进入数据库设计与 API 设计基线。
3. `packages/document` 中预留前处理边界。
4. `packages/rag` 中预留切块、向量化、发布三个主步骤边界。
5. 检索只消费当前有效索引版本。
6. 失败不破坏上一版可用检索结果。

第一阶段不强求：

1. 完整分布式队列。
2. 自动重试中心。
3. 多租户批量调度控制台。
4. 索引任务可视化后台。

## 17. 当前结论

如果 DocWeave 希望后续真正支撑“协同编辑 + 稳定检索 + 可追溯 AI 回答”，那么搜索与索引一定不能只是一个“调 embedding 写向量库”的脚本，而必须是围绕快照版本组织的正式任务编排链路。
