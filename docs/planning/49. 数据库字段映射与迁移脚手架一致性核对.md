# 49. 数据库字段映射与迁移脚手架一致性核对

阅读说明：

1. 本文档负责核对数据库专题文档中的目标字段语义，与当前 `apps/api` 中 migration / schema / model / seed 脚手架之间的一致性。
2. 数据库目标基线以 [04. 数据库设计](../architecture/04.%20数据库设计.md) 和 [33. 数据库表结构与状态字段补充](../architecture/33.%20数据库表结构与状态字段补充.md) 为准。
3. 若需继续推进版本、快照与索引落地，继续读 [24. 版本与稳定快照设计](../architecture/24.%20版本与稳定快照设计.md) 与 [25. 搜索与索引任务编排设计](../architecture/25.%20搜索与索引任务编排设计.md)。

## 1. 文档目的

本文档用于回答一个非常实际的问题：

当前文档里已经定义的数据库对象、字段和状态语义，和 `apps/api` 里已经存在的 migration / Lucid model / seed 脚手架，究竟对齐到了哪一步，还有哪些明显缺口。

这样做的价值有两点：

1. 避免后续实现时误以为“文档已经设计过，所以数据库已经有了”。
2. 给后续 schema 演进提供明确的补齐顺序，而不是边做接口边临时补表。

## 2. 本次核对范围

本次核对基于以下实际实现：

1. `apps/api/database/migrations`
2. `apps/api/database/schema.ts`
3. `apps/api/app/models`
4. `apps/api/database/seeders/docweave_baseline_seeder.ts`

截至当前，实际已存在的数据库表只有：

1. `users`
2. `auth_access_tokens`
3. `spaces`
4. `documents`

## 3. 当前实现现状

### 3.1 migration 已落表

当前 migration 已落的表和字段如下：

1. `users`
   - `id`
   - `full_name`
   - `email`
   - `password`
   - `created_at`
   - `updated_at`
2. `auth_access_tokens`
   - `id`
   - `tokenable_id`
   - `type`
   - `name`
   - `hash`
   - `abilities`
   - `created_at`
   - `updated_at`
   - `last_used_at`
   - `expires_at`
3. `spaces`
   - `id`
   - `name`
   - `summary`
   - `created_at`
   - `updated_at`
4. `documents`
   - `id`
   - `space_id`
   - `title`
   - `summary`
   - `status`
   - `content`
   - `created_at`
   - `updated_at`

### 3.2 model 已暴露对象

当前 Lucid model 只覆盖：

1. `User`
2. `Space`
3. `Document`

其中：

1. `Document` 当前暴露 `spaceId / title / summary / content / status`
2. `Document.status` 当前在业务上已被收窄为 `draft | review | ready`
3. `Space` 当前没有 `workspaceId`
4. `User` 通过 `schema.ts` 自动生成字段映射，再叠加认证能力

### 3.3 seed 当前隐含的业务现实

当前 seed 体现的其实是一个早期简化模型：

1. `Space` 被当成接近“工作台一级容器”的对象使用
2. `Document` 直接挂在 `Space` 下
3. 文档正文直接存在 `documents.content`
4. 文档状态仍是编辑流程状态语义，不是信息架构文档里的发布状态语义

## 4. 已对齐项

虽然整体仍是早期脚手架，但以下几项已经和文档方向部分对齐：

### 4.1 用户与会话基线已存在

已落地：

1. `users`
2. `auth_access_tokens`

这和 [14. 认证与会话实现决策](../decisions/14.%20认证与会话实现决策.md) 的第一阶段账号密码登录基线是一致的。

### 4.2 空间与文档主对象已存在最小闭环

已落地：

1. `spaces`
2. `documents`
3. `documents.space_id -> spaces.id`

这支撑了当前工作台、空间详情页、文档页的最小内容闭环。

### 4.3 文档正文已有单表存储基线

`documents.content` 已存在，说明当前实现已经能支撑：

1. 文档创建
2. 文档读取
3. 文档更新

这和当前单人编辑器基线是匹配的，只是还没有演进到“实时态 / 快照态 / 已索引态”三层分离。

## 5. 未对齐项

下面这些缺口，是当前文档基线与数据库脚手架之间最核心的不一致来源。

### 5.1 `workspace` 领域尚未落表

文档基线中，`workspace` 是：

1. 成员边界
2. 权限边界
3. 文件归属边界
4. 空间和文档的上层归属边界

但当前实现里：

1. 没有 `workspaces`
2. 没有 `workspace_members`
3. `spaces` 没有 `workspace_id`
4. `documents` 没有 `workspace_id`

这意味着目前数据库现实仍然是：

```text
space -> document
```

而不是文档基线里的：

```text
workspace -> space -> document
```

### 5.2 文档关键版本字段尚未落地

文档基线中反复强调的字段：

1. `latest_snapshot_version`
2. `latest_indexed_version`
3. `latest_published_version`

当前 `documents` 表中都不存在。

这会直接影响：

1. 稳定快照主链路
2. RAG 可检索版本切换
3. 分享与公开阅读的目标版本表达
4. Citation 追溯

### 5.3 文档状态语义发生冲突

当前代码里的 `documents.status` 实际使用值是：

1. `draft`
2. `review`
3. `ready`

而数据库与信息架构文档中建议的内容状态语义是：

1. `draft`
2. `published`
3. `archived`

这说明当前 `status` 更像“编辑流程阶段”，而不是“内容生命周期状态”。

如果后续不收口，容易出现两个问题：

1. 同一个字段同时表达审核流程和发布语义
2. API、页面、索引、分享各自对 `status` 做不同解释

### 5.4 快照表尚未存在

文档基线中的 `document_snapshots` 当前完全未落地。

缺失内容包括：

1. `document_id`
2. `version`
3. `content`
4. `created_by`
5. `created_at`

这意味着当前还没有真正可复用的：

1. 稳定版本真相源
2. 历史版本主表
3. RAG 的稳定输入表

### 5.5 索引任务表尚未存在

文档基线中的 `rag_index_jobs` 当前完全未落地。

缺失内容包括：

1. `target_snapshot_version`
2. `status`
3. `stage`
4. `error_code`
5. `started_at`
6. `finished_at`

这意味着 [25. 搜索与索引任务编排设计](../architecture/25.%20搜索与索引任务编排设计.md) 中定义的任务生命周期，目前还没有数据库真相源承接。

### 5.6 文件表尚未存在

文档基线中的 `files` 当前完全未落地。

缺失内容包括：

1. `workspace_id`
2. `document_id`
3. `kind`
4. `object_key`
5. `mime_type`
6. `size`
7. `created_by`

这会影响：

1. 文档内资源引用
2. 上传后资源复用
3. 导出产物管理
4. 分享时的资源访问

### 5.7 分享表尚未存在

文档基线中的 `shares` 当前完全未落地。

缺失内容包括：

1. `share_id`
2. `access_level`
3. `status`
4. `target_snapshot_version`
5. `expires_at`

这意味着分享与公开链接能力还没有数据库模型基础。

### 5.8 成员关系表尚未存在

当前实现没有：

1. `workspace_members`
2. `space_members`
3. `document_members`

这说明权限能力目前还处于“认证基线先行，资源授权未落表”的阶段。

## 6. 字段层一致性判断

从字段映射角度，可以把当前状态简单分成三类：

### 6.1 已一致

1. `users.email / password`
2. `spaces.id / name / summary`
3. `documents.id / space_id / title / summary`
4. `documents.content`

### 6.2 已有对象但字段明显不足

1. `spaces`
   - 缺 `workspace_id`
2. `documents`
   - 缺 `workspace_id`
   - 缺 `parent_document_id`
   - 缺 `latest_snapshot_version`
   - 缺 `latest_indexed_version`
   - 缺 `latest_published_version`
3. `documents.status`
   - 当前值域与文档目标语义不一致

### 6.3 文档已定义但数据库尚未起表

1. `workspaces`
2. `workspace_members`
3. `space_members`
4. `document_members`
5. `document_snapshots`
6. `files`
7. `shares`
8. `ai_settings`
9. `rag_settings`
10. `rag_index_jobs`

## 7. 当前最大风险

如果后续继续在现有简化 schema 上直接叠功能，而不先做字段和表的收口，最容易出现以下风险：

1. 页面和 API 已经开始消费 `workspace` 语义，但数据库仍只有 `space`
2. 快照、索引、分享各自临时找地方存状态，后续很难收口
3. `documents.status` 被不同模块赋予不同含义
4. RAG 先以 `documents.content` 直读推进，后续再切快照表时需要二次迁移

## 8. 建议补齐顺序

建议不要一次性把所有设计表全补完，而是按主链路影响排序推进。

### 8.1 第一优先级

先补最影响主链路的一组：

1. `workspaces`
2. `workspace_members`
3. `spaces.workspace_id`
4. `documents.workspace_id`
5. `documents.latest_snapshot_version`
6. `documents.latest_indexed_version`
7. `documents.latest_published_version`
8. `document_snapshots`
9. `rag_index_jobs`

这组补完后，`workspace -> space -> document -> snapshot -> index job` 主链才真正有数据库真相源。

### 8.2 第二优先级

再补资源与访问相关对象：

1. `files`
2. `shares`
3. `space_members`
4. `document_members`

这组主要服务：

1. 上传与资源引用
2. 分享访问
3. 更细粒度权限

### 8.3 第三优先级

最后再补配置与优化类对象：

1. `ai_settings`
2. `rag_settings`

如果第一阶段前期模型配置仍以环境变量和服务端配置文件为主，这组可以延后。

## 9. 对当前 migration 演进的建议

这次核对后的建议不是“立刻大改所有代码”，而是：

1. 先把数据库目标基线和当前实现差异明确写死
2. 后续按里程碑拆 migration，而不是一次性补一个超大迁移
3. 在开始 M4 稳定快照与索引闭环前，优先把 `document_snapshots + rag_index_jobs + documents` 版本字段补齐
4. 在开始更完整权限与分享能力前，优先把 `workspace` 及成员关系落表

## 10. 当前结论

当前 `apps/api` 的数据库脚手架，仍然是支撑单人文档与基础工作台的早期简化模型；而文档体系已经进入面向 `workspace / snapshot / indexing / share / file` 的正式平台模型。

这不是坏事，但必须明确承认两层现实：

1. 文档基线已经定义了目标结构。
2. 数据库实现目前只落到了其中最小的一段。

因此，后续最值得优先推进的，不是继续在旧字段上堆更多业务语义，而是先把 `workspace`、`snapshot`、`index job` 这三块数据库真相源补上。
