# 51. ID 统一策略规划说明

## 1. 背景

DocWeave 当前仓库里的 ID 形态并不统一：

1. `users.id` 使用自增整数。
2. `auth_access_tokens.tokenable_id` 跟随 `users.id`，也是整数。
3. `spaces.id`、`documents.id` 对外是字符串，但当前实现并不是纯 UUID，而是带可读前缀的字符串 ID。
4. M4 新增的 `document_snapshots.id`、`rag_index_jobs.id` 已经使用 UUID。
5. Qdrant point id 在实际联调中也暴露出格式约束，稳定 UUID 形态比自定义字符串更稳。

这说明“长期希望统一”是合理方向，但“现在立刻全仓一次性统一”并不等于最小成本。

## 2. 当前判断

当前阶段**不建议**在普通功能 change 中顺手推动“全项目 ID 一次性改成 UUID”。

原因如下：

1. `users.id` 已经接入 Adonis Auth 与 `auth_access_tokens`，改动会穿透认证、token、测试、历史数据与 migration。
2. 现有 `spaces` / `documents` 已经在 API、MCP、测试和种子数据中广泛被引用，迁移成本高于“新表直接统一”。
3. 这类改动的收益主要是中长期一致性，而不是当前某一条功能链路的直接阻塞项。
4. 一次性改全仓容易把“ID 统一专项”误混进别的主线 change，增加回归面。

## 3. 推荐策略

推荐采用“**旧链路保守，新增资源统一**”的渐进策略。

### 3.1 新增资源

从现在开始，新的业务资源表默认统一使用 UUID 作为主键。

包括但不限于：

1. 新增业务实体表。
2. 新增后台任务表。
3. 新增快照、索引、发布、审计、引用等资源表。

这样做的收益是：

1. 避免继续制造新的 ID 形态分叉。
2. 对外 contract 可以统一把资源 id 当作 opaque string。
3. 更适合后续多进程、异步任务、外部存储与分布式场景。

### 3.2 旧认证链路

`users.id` 与 `auth_access_tokens.tokenable_id` 暂时保留整数。

原因不是它们“应该永远如此”，而是：

1. 它们已经落进认证基座。
2. 它们的迁移必须独立规划、独立回归、独立验收。

### 3.3 旧业务字符串 ID

`spaces.id`、`documents.id` 当前维持现状，但不再推荐复制现有“可读前缀 + 随机尾巴”的生成方式到新资源。

后续若要统一，优先级应为：

1. 先定项目级 ID policy。
2. 再单独决定是否把 `spaces` / `documents` 迁移为纯 UUID。
3. 最后才讨论是否把 `users.id` 也迁移为 UUID。

## 4. 项目级建议口径

建议后续项目统一采用下面这套口径：

1. **资源主键**：优先 UUID。
2. **对外 API contract**：统一视作 `string`，不对调用方承诺可排序或可枚举语义。
3. **内部顺序号**：继续使用整数，例如 `snapshotVersion`、任务重试次数、分页偏移等。
4. **第三方存储中的点位 ID / 外部对象 ID**：优先使用标准 UUID 或第三方允许的标准稳定格式，不再使用随意拼接字符串。

## 5. 不在本说明内立即推进的范围

本说明只做策略规划，不直接触发下面这些迁移：

1. `users.id` 改 UUID。
2. `auth_access_tokens` 跟随迁移。
3. `spaces.id` / `documents.id` 数据回填与兼容层。
4. 旧 API / MCP / 测试数据的批量改写。

这些内容如果要做，应单独开 change，例如：

- `feat-id-unification-baseline`

## 6. 后续建议

如果后续准备正式推进 ID 统一专项，建议该 change 至少回答下面几个问题：

1. `spaces` / `documents` 是否迁移为纯 UUID，还是继续保留现状并只冻结未来策略。
2. `users.id` 是否保持整数，还是纳入二阶段迁移。
3. API / MCP / 前端路由是否需要兼容旧 ID。
4. 种子数据、测试数据、fixtures、Qdrant payload、日志字段如何同步调整。
5. 是否需要引入“外部可见 ID”与“内部主键”分层。

## 7. 当前结论

当前最务实的结论是：

1. **不要在普通功能 change 中顺手全仓改 UUID。**
2. **从现在开始，新增资源主键统一 UUID。**
3. **历史认证链路和历史资源 ID 的统一迁移单独规划。**

这条策略既能阻止仓库继续扩散新的 ID 形态，又不会把高风险迁移夹带进当前功能主线。
