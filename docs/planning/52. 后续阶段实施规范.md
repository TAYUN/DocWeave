# 52. 后续阶段实施规范

本文档规定 M6 及之后阶段如何拆分范围、确认前置条件和判断是否进入代码实施。

## 1. 阶段划分原则

每个阶段必须只有一个主要用户结果，并满足以下约束：

1. 阶段目标能够用一句话验收。
2. 阶段内的任务共同服务于同一条用户链路。
3. 非前置依赖的横向能力不得顺手并入当前阶段。
4. 基础设施升级不得与用户功能默认绑定。
5. 每个阶段都必须明确不做什么。

## 2. 能力线拆分

DocWeave 后续按四条能力线推进：

### 2.1 核心知识产品线

```text
编辑 -> 协同 -> 快照 -> 索引 -> 搜索 -> 问答 -> 稳定化
```

对应 M2 至 M7，优先级最高。

### 2.2 文档输入线

```text
文件/外部内容 -> 解析 -> BlockNote 内容 -> 文档快照
```

对应 M8，首批优先支持 Markdown / HTML 等近结构化格式。

### 2.3 文档输出与访问线

```text
稳定快照/已发布版本 -> 导出 -> 对象存储 -> 下载/分享
```

对应 M9，依赖稳定版本和发布语义，不提前塞入 M6。

### 2.4 基础设施与体验线

```text
数据库轮询 -> BullMQ
轻量聊天 UI -> assistant-ui 或其他成熟交互层
```

对应 M10 及之后，必须由真实规模或维护成本驱动。

## 3. M6 实施规范

M6 的主结果是“已索引文档可搜索、可问答、可回到 Citation 原文”。

M6 允许补充 RAG 必需的 block 元数据预处理，但不得实现完整导入导出、文件 RAG、发布分享或队列升级。

M6 的验收必须覆盖：

1. 权限范围过滤。
2. `latestIndexedVersion` 版本过滤。
3. Qdrant 检索。
4. 流式回答。
5. Citation 校验和回跳。
6. 无索引、无权限、旧版本和失败任务状态。

## 4. M7 实施规范

M7 不以增加新用户入口为目标，重点验证核心链路长期可用：

1. 版本语义统一。
2. 协同、快照、索引、AI 和 RAG 的跨链路回归。
3. 权限回收和旧索引隔离。
4. 失败恢复、错误码和日志字段收口。
5. 仍保留数据库轮询 worker，除非已有性能或可靠性证据证明需要升级。

## 5. M8/M9 实施规范

### M8 文档输入

1. 先定义原始文件、导入任务、目标文档和转换结果的关系。
2. 首批只实现一种或两种近结构化格式。
3. 导入必须可追踪来源、版本和失败原因。
4. 文件导入不自动扩大为 OCR、Office 和复杂附件 RAG。

### M9 文档输出与分享

1. 导出必须绑定稳定快照或已发布版本。
2. 导出结果必须记录来源版本和格式。
3. 重型导出进入 worker，不在 API 请求内同步执行。
4. 下载和分享必须复用文档权限与对象存储访问策略。
5. PDF、批量导出和公开访问可以继续拆成独立 change。

## 6. 进入下一阶段的门槛

只有满足以下条件，才进入下一阶段：

1. 当前阶段主链路验收通过。
2. 当前阶段的回归清单已执行。
3. 未完成事项已明确归属到后续阶段，不留在当前 change 的隐含范围内。
4. 前置版本、权限和数据契约已经稳定。
5. 新阶段的用户结果、非目标和验收标准已经确认。

## 7. 明确的延期原则

以下能力默认不应提前进入核心主线：

1. 完整导入导出。
2. OCR、PDF、Office 和复杂文件后处理。
3. 发布、分享和公开访问。
4. BullMQ 和复杂后台调度。
5. assistant-ui 和完整聊天产品化壳。

只有当它们成为当前用户闭环的实际阻塞项，或已有明确的规模/可靠性证据时，才单独建立 change 推进。
