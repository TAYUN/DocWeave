# 50. 数据库演进迁移拆分方案

阅读说明：

1. 本文档负责把 [49. 数据库字段映射与迁移脚手架一致性核对](./49.%20数据库字段映射与迁移脚手架一致性核对.md) 中识别出的缺口，拆成可执行的 migration 演进批次。
2. 数据库目标基线以 [04. 数据库设计](../architecture/04.%20数据库设计.md) 与 [33. 数据库表结构与状态字段补充](../architecture/33.%20数据库表结构与状态字段补充.md) 为准。
3. 实施顺序需与 [00. 当前实施路线图](./00.%20当前实施路线图.md) 和 [22. M4 稳定快照与索引实施文档](./22.%20M4%20稳定快照与索引实施文档.md) 保持一致。

## 1. 文档目的

本文档用于回答：

数据库从当前的早期简化模型，演进到支撑 `workspace / snapshot / index job / file / share` 的正式模型时，migration 应该怎么拆批次，才能既不打乱现有闭环，又不给后续主链路埋兼容坑。

## 2. 当前起点

截至当前，实际数据库脚手架只有：

1. `users`
2. `auth_access_tokens`
3. `spaces`
4. `documents`

其中当前最关键的现实是：

1. `spaces` 还没有 `workspace_id`
2. `documents` 还没有版本字段
3. `document_snapshots` 尚未落表
4. `rag_index_jobs` 尚未落表
5. 文件、分享、成员关系都还没有正式表结构

## 3. 拆批原则

后续 migration 建议遵循以下原则：

1. 先补主链路真相源，再补扩展能力对象。
2. 先新增表和新增列，再逐步把业务读写切过去。
3. 避免在一个 migration 批次里同时做“结构重组 + 大量语义改名 + 现有接口行为改变”。
4. 允许短期保留兼容字段或兼容状态，但必须明确最终收口目标。

## 4. 推荐迁移批次

### 4.1 Batch A：补齐 `workspace` 主边界

目标：

让数据库正式进入 `workspace -> space -> document` 分层，而不是继续停留在 `space -> document`。

建议新增或调整：

1. 新建 `workspaces`
2. 新建 `workspace_members`
3. `spaces` 新增 `workspace_id`
4. `documents` 新增 `workspace_id`

建议字段最小集：

1. `workspaces`
   - `id`
   - `name`
   - `slug` 或等价唯一标识
   - `created_by`
   - `created_at`
   - `updated_at`
2. `workspace_members`
   - `id`
   - `workspace_id`
   - `user_id`
   - `role`
   - `created_at`
   - `updated_at`

兼容策略：

1. 先允许 `spaces.workspace_id`、`documents.workspace_id` 在过渡期可为空。
2. 先通过 seed 或一次性脚本补一条默认 `workspace`。
3. 再把现有 `spaces` 和 `documents` 回填到默认 `workspace`。
4. 回填完成后再把这两个字段改成 `not null`。

这一批完成后，后续权限、文件和分享才有稳定归属边界。

### 4.2 Batch B：补齐 `documents` 版本与状态字段

目标：

先把文档主表上的关键版本语义补上，为快照和索引链路做准备。

建议新增或调整：

1. `documents` 新增 `latest_snapshot_version`
2. `documents` 新增 `latest_indexed_version`
3. `documents` 新增 `latest_published_version`
4. 评估 `documents.status` 的目标值域

关于 `status` 的建议：

1. 不建议直接沿用当前 `draft / review / ready` 作为长期内容状态
2. 推荐收口到 `draft / published / archived`
3. 如果短期内还需要保留“编辑流程态”，应新增独立字段，而不是继续复用 `status`

兼容策略：

1. 新增版本字段时先允许为空
2. 在还没正式接入快照前，`latest_snapshot_version` 可先保持 `null`
3. `status` 改造优先放在业务层和 seed 同步切换后再落最终约束

### 4.3 Batch C：落地 `document_snapshots`

目标：

让稳定快照成为正式数据库对象，而不是继续只把正文放在 `documents.content`。

建议新增：

1. `document_snapshots`

建议字段最小集：

1. `id`
2. `document_id`
3. `version`
4. `content`
5. `created_by`
6. `created_at`

关键约束建议：

1. `(document_id, version)` 唯一
2. `version` 按单文档递增
3. `content` 存稳定 BlockNote JSON 或服务端等价真相

兼容策略：

1. 短期保留 `documents.content`
2. 第一阶段可以采用“双写但以旧读为主”的过渡方式
3. 当快照读取链路稳定后，再把索引、分享、导出逐步切到 `document_snapshots`

### 4.4 Batch D：落地 `rag_index_jobs`

目标：

把快照到索引的任务生命周期正式落到数据库，而不是只存在于代码抽象里。

建议新增：

1. `rag_index_jobs`

建议字段最小集：

1. `id`
2. `document_id`
3. `target_snapshot_version`
4. `status`
5. `stage`
6. `error_code`
7. `started_at`
8. `finished_at`
9. `created_at`
10. `updated_at`

状态建议：

1. `pending`
2. `running`
3. `succeeded`
4. `failed`
5. `canceled`

这一批完成后，`apps/worker` 才有正式的数据库任务真相源可消费。

### 4.5 Batch E：把索引版本切换闭环接到 `documents`

目标：

不是只把任务表建出来，而是把 `document_snapshots -> rag_index_jobs -> documents.latest_indexed_version` 真正串起来。

建议配套改动：

1. API 创建快照时回写 `documents.latest_snapshot_version`
2. API 创建索引任务时写入 `rag_index_jobs`
3. Worker 成功完成后原子更新 `documents.latest_indexed_version`

这一批不一定需要新表，但需要和前面几个 migration 批次一起规划，避免“表先有了，行为长期没切过去”。

### 4.6 Batch F：补齐文件与分享对象

目标：

让上传、资源引用和分享访问有正式表结构，不再依赖临时字段或后续补洞。

建议新增：

1. `files`
2. `shares`

建议字段最小集：

1. `files`
   - `id`
   - `workspace_id`
   - `document_id`
   - `kind`
   - `object_key`
   - `mime_type`
   - `size`
   - `created_by`
   - `created_at`
   - `updated_at`
2. `shares`
   - `id`
   - `share_id`
   - `workspace_id`
   - `document_id`
   - `access_level`
   - `status`
   - `target_snapshot_version`
   - `expires_at`
   - `created_by`
   - `created_at`
   - `updated_at`

### 4.7 Batch G：补齐更细粒度成员关系

目标：

在 `workspace_members` 已稳定的前提下，再决定是否需要更细粒度资源授权。

建议新增：

1. `space_members`
2. `document_members`

说明：

1. 如果第一阶段权限主要收口在 `workspace`，这批可以延后
2. 不建议在 `workspace` 还没落稳时，就提前起更细粒度成员表

### 4.8 Batch H：补齐配置类对象

目标：

为模型、embedding 和检索参数提供正式数据库承载。

建议新增：

1. `ai_settings`
2. `rag_settings`

说明：

1. 若第一阶段仍以环境变量为主，这批可以最后做
2. 不应阻塞 M3/M4/M5 主链路

## 5. 建议实施顺序

按当前项目节奏，推荐顺序如下：

1. Batch A：`workspace`
2. Batch B：`documents` 版本字段
3. Batch C：`document_snapshots`
4. Batch D：`rag_index_jobs`
5. Batch E：索引版本闭环接线
6. Batch F：`files + shares`
7. Batch G：更细粒度成员关系
8. Batch H：配置类对象

## 6. 对现有代码的兼容建议

为了降低迁移风险，建议迁移期遵循以下做法：

1. 先避免一次性删除 `documents.content`
2. 先避免一次性强制所有旧数据拥有完整 `workspace` 归属
3. 先避免在一个版本里同时改路由 contract、DTO 和 schema
4. 先通过 seed / 脚本回填，再收紧数据库约束

## 7. 每批完成后的验收点

### 7.1 Batch A 验收点

1. 现有 `spaces` 与 `documents` 都能追溯到 `workspace`
2. 登录后工作台读取链路不被破坏

### 7.2 Batch B/C/D/E 验收点

1. 可以为文档生成稳定快照
2. 可以创建索引任务
3. Worker 可以消费任务
4. 成功后 `latestIndexedVersion` 正确回写

### 7.3 Batch F/G/H 验收点

1. 文件上传有正式元数据记录
2. 分享链接有明确状态与目标版本
3. 配置对象不再散落于环境变量和临时代码分支

## 8. 当前结论

数据库演进最怕两个极端：

1. 一直停留在早期简化表结构上硬堆能力
2. 一次性做一个超大 schema 重构，把现有最小闭环一起打碎

当前更合适的做法，是按本文档把 migration 拆成多批，先稳住 `workspace`、`snapshot`、`index job` 三条真相源，再逐步补齐文件、分享和配置对象。
