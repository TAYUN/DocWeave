# 52. 工具链版本与 TypeScript 配置收敛计划

## 1. 背景

M5 编辑器 AI 落地后，AI provider 适配优化暴露出 Monorepo 工具链的历史漂移：不同应用和 package 分别在不同时间引入了 TypeScript、`@types/node` 与 tsconfig 配置。

当前差异不会立刻阻塞业务功能，但会造成以下问题：

1. VS Code 使用 package 自身 tsconfig 检查时，可能找不到某个类型库。
2. 同一段代码在应用级 typecheck 通过、在 package 级 typecheck 报错。
3. 升级某个三方库时，不同 TypeScript 版本对类型语法的支持不一致。
4. Node、浏览器 React 与通用 package 的运行环境边界不够清楚。

本规划是 M5 之后的工程治理事项，不属于编辑器 AI 功能主线。

## 2. 当前事实

截至本文件创建时，仓库存在以下版本与配置差异：

1. `packages/ai`、`packages/document`、`packages/rag`、`apps/worker` 使用 TypeScript 5.9 与 `@types/node` 24。
2. `apps/web` 使用 TypeScript 6 与 `@types/node` 24。
3. `apps/api` 使用 TypeScript 6 与 `@types/node` 25。
4. `apps/collab` 使用 TypeScript 7 与 `@types/node` 26。
5. `packages/editor` 使用浏览器 React 配置：`DOM`、`react`、`react-dom` 与 Bundler 模块解析。
6. Node 运行时 package 使用 `NodeNext` 模块与 Node 类型。
7. 通用 adapter package 使用 Bundler 模块解析；若使用标准 Fetch API，应显式声明 `DOM` lib。若其测试位于源码目录且使用 `node:test`，则还需要本 package 的 Node 类型。

## 3. 目标与边界

目标：

1. 统一 TypeScript 主版本与 `@types/node` 主版本。
2. 让每个 package 的独立 typecheck 可重复、可通过。
3. 将 tsconfig 收敛为少量明确的运行环境 profile。
4. 保持应用与 package 的模块解析策略可解释，不依赖上级目录偶然安装的类型包。

不做：

1. 不把所有 package 强行改成同一份 tsconfig。
2. 不在此变更中升级 BlockNote、AdonisJS、React 或 AI SDK。
3. 不顺手重写业务代码或 M5 已验证的编辑器 AI 链路。

## 4. 建议的配置 Profile

| Profile               | 适用范围                                           | 核心配置                                                                             |
| --------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------ |
| Node library          | `packages/ai`、`packages/document`、`packages/rag` | `module/moduleResolution: NodeNext`、`types: ["node"]`                               |
| Browser React library | `packages/editor`                                  | `lib: ["ES2023", "DOM"]`、`types: ["react", "react-dom"]`、`jsx: react-jsx`、Bundler |
| Isomorphic adapter    | `packages/adapters`、`packages/contracts`          | Bundler；按实际 API 声明 `DOM` 或 Node 类型，避免默认混入全部运行时类型              |
| Node application      | `apps/api`、`apps/collab`、`apps/worker`           | 由框架或 Node 运行时决定；Node 类型版本与仓库基线一致                                |
| Browser application   | `apps/web`                                         | 由 Vite/React 决定；DOM 与 React 类型版本与仓库基线一致                              |

`DOM` 只表示 TypeScript 提供标准 Web API 类型，并不意味着代码一定在浏览器执行。Node 18+ 的原生 Fetch 也可使用这组类型；但应在 package tsconfig 中显式声明，不能依赖其他项目间接提供。

## 5. 推荐执行顺序

### 5.1 先建立版本基线

1. 选择一个已验证的 TypeScript 主版本，优先确认 AdonisJS、Vite、BlockNote、Tuyau 的兼容范围。
2. 选择一个统一的 `@types/node` 主版本，并确认 Node 运行时版本支持。
3. 将版本通过 pnpm workspace catalog 或根级约束统一声明，避免每个 package 手写不同版本。
4. 更新 lockfile 后执行一次全量安装，确保 VS Code 与 CI 使用相同依赖树。

### 5.2 再收敛 tsconfig

1. 在根目录建立可复用的 Node、Browser React、Isomorphic 三类 tsconfig 基础文件。
2. 各 package 仅保留与自身运行环境有关的少量覆盖项。
3. 保持 `packages/editor` 与 Node package 的 module resolution 分离。
4. 对使用 `node:test` 的 package，要么声明 Node 类型并保留测试在 `src`，要么将测试移至不参与生产源码 typecheck 的目录；两种方式选择其一并统一。

### 5.3 最后补齐验证入口

1. 为所有 `packages/*` 增加可独立执行的 `typecheck` script。
2. 在根 `package.json` 增加 workspace 级 typecheck 命令。
3. 在 CI 中执行安装、全量 typecheck、关键 lint 与关键测试。
4. 明确 package 级 typecheck 是 VS Code 报错的最终判断来源之一，不能只依赖 `apps/api` 或 `apps/web` 的间接检查。

## 6. 验收标准

完成该独立工程变更时，应满足：

1. `pnpm install --frozen-lockfile` 可成功执行。
2. 每个 `apps/*` 与 `packages/*` 的 typecheck 命令可独立通过。
3. VS Code 打开任一 package 时，不出现“找不到 node/react/DOM 类型定义”的环境性报错。
4. `pnpm why typescript` 与 `pnpm why @types/node` 的顶层版本符合选定基线；框架必须保留的嵌套版本需有记录。
5. M5 编辑器 AI、协同服务与 worker 的现有回归验证继续通过。

## 7. 风险提醒

1. TypeScript 主版本升级可能暴露三方库声明、装饰器、module resolution 与 JSX 类型的兼容问题。
2. `@types/node` 升级可能改变 Fetch、Web Streams、定时器等全局类型，影响 API、协同和 worker。
3. 不应为了“版本一致”删除浏览器包的 `DOM` 或 React 类型，也不应把 Node 类型默认注入全部前端 package。
4. 本项应以独立 `chore(toolchain)` 变更推进，单独验证和提交，避免与业务功能变更互相干扰。
