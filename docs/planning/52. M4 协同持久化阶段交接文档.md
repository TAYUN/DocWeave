# 52. M4 协同持久化阶段交接文档

阅读说明：

1. 本文档用于把 `feat-m4-collaboration-persistence-baseline` 这条独立阶段交接到新会话继续推进。
2. 当前主线顺序仍以 [00. 当前实施路线图](./00.%20当前实施路线图.md) 为准；本阶段属于 M4 的协同持久化子线，不是 M5。
3. 本文档只描述“下一段要做什么、边界是什么、验证什么”，不重新展开整套 M4 稳定快照与索引设计。

## 1. 当前状态

当前项目已经完成或接近完成的相关上下文：

1. M3 协同最小闭环已存在。
2. M4 稳定快照与索引基线已经单独推进过，相关 change 和工件已经形成。
3. 当前仍保留一条独立的协同持久化阶段：`feat-m4-collaboration-persistence-baseline`。
4. 现阶段协同仍以“前端 seed + 手动保存快照”作为过渡路径，服务端恢复与持久化语义还没有真正补齐。

## 2. 本阶段目标

本阶段只解决一件事：

> 让协同正文不再只依赖前端 seed 或临时会话态，而是能够从服务端稳定真相恢复，并在协同编辑过程中回写到稳定持久化路径。

拆开来说，就是三件具体事：

1. `onLoadDocument` 从数据库正文或等价稳定真相恢复到 `Yjs`
2. `onStoreDocument` 或节流后的持久化链路把 `Yjs` 回写到稳定快照
3. 明确“手动保存”和“自动持久化”各自负责什么，避免职责重叠

## 4. 输入边界

新会话进入后，只处理以下范围：

1. `apps/collab` 的 `onLoadDocument` / `onStoreDocument` 或等价钩子
2. `apps/api` 中与协同正文恢复、快照保存有关的最小承接点
3. `apps/web` 文档编辑页里与协同恢复/保存相关的最小调用链
4. 需要时补最小回归测试与协同回归清单

不要扩展的范围：

1. 不在这里扩成 M5 编辑器 AI
2. 不在这里重做 M4 索引基线
3. 不在这里引入复杂多节点协同、离线回放或队列系统
4. 不把协同持久化顺手扩成完整导出 / RAG / AI 方案

不要改动的内容：

1. 不破坏已完成的 M4 稳定快照与索引基线
2. 不重写已有的 `packages/contracts` 统一响应契约
3. 不混入已单独提交的文档编辑器 lint 修复

## 5. 这阶段最值得验证的点

### 5.1 服务端恢复

1. 刷新或重新进入文档后，协同内容应从服务端稳定真相恢复
2. 不应继续依赖页面初始化时的临时 seed 作为唯一内容来源
3. 房间重新连接后，正文与房间必须仍然对应同一篇文档

### 5.2 持久化职责

1. 手动保存按钮负责什么
2. 自动持久化负责什么
3. 两者是否会重复写同一份数据
4. 两者是否会在同一时刻互相覆盖

### 5.3 恢复一致性

1. 浏览器重连后正文是否仍然一致
2. 协同状态切换后是否出现内容回滚
3. 退出再进入后是否能继续从稳定真相恢复

## 6. 期望产出

新会话至少应产出以下内容之一：

1. 服务端恢复与持久化的明确实现方案
2. 清晰的职责边界说明
3. 针对恢复 / 持久化的最小回归测试
4. 需要回写到 `design.md` / `tasks.md` / `specs/` 的结论

## 7. 当前已知结论

1. 这不是 M5。
2. 这是一条独立的 M4 协同持久化子线。
3. 当前最核心的风险不是“协同能不能跑”，而是“正文真相到底由谁负责”。
4. 如果职责边界不先定清，后续会继续出现 seed、手动保存和自动持久化互相覆盖的问题。

## 8. 开会话时的最短提示词

你在新会话里可以直接这样开：

> 请继续 `feat-m4-collaboration-persistence-baseline`，先补齐协同正文的服务端恢复与持久化职责边界，再决定是否需要最小回归测试。
