# 45. 权限矩阵与 Capability 对照表

阅读说明：

1. 本文档负责把角色语义和 capability 结果整理成最小对照口径。
2. 权限模型与实现分别以 [11. 权限模型设计](../architecture/11.%20权限模型设计.md) 和 [12. 权限实现设计](../architecture/12.%20权限实现设计.md) 为准。

## 1. 文档目的

本文档用于给实现和联调阶段提供一份最小权限对照表，避免大家只记得角色名，却不知道最终页面、协同、AI、RAG 应该消费哪组能力结果。

## 2. 最小 capability 建议

建议统一至少包含：

1. `canReadDocument`
2. `canEditDocument`
3. `canCommentDocument`
4. `canShareDocument`
5. `canUseEditorAI`
6. `canUseRagSearch`
7. `canUseRagChat`

## 3. 最小对照原则

建议按如下原则理解：

1. `viewer` 至少应可读，不可编辑
2. `editor` 应可读、可编辑、可应用 AI 结果
3. `commenter` 若第一阶段启用，则可读、可评论、不可改正文
4. 分享能力不应默认对所有可读用户开放

## 4. 协同视角能力

协同 token 至少建议映射：

1. `canRead`
2. `canEdit`
3. `canComment`
4. `canUseEditorAI`

说明如下：

1. 页面层 capability 可以更完整
2. 协同 token 只保留进入协同链路所需最小能力

## 5. 当前结论

第一阶段最重要的不是做一张巨大的权限表，而是先保证角色到 capability 的核心映射稳定，这样页面、协同、AI 和 RAG 才不会各自解释角色。
