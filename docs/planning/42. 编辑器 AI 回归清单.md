# 42. 编辑器 AI 回归清单

阅读说明：

1. 本文档负责定义编辑器 AI 最小闭环的回归清单。
2. 编辑器 AI 路线与实现分别以 [21. 聊天与 Copilot 交互层策略](../decisions/21.%20聊天与%20Copilot%20交互层策略.md)、[22. BlockNote 服务端能力接入设计](../architecture/22.%20BlockNote%20服务端能力接入设计.md) 和 [21. M5 编辑器 AI 实施文档](./21.%20M5%20编辑器%20AI%20实施文档.md) 为准。

## 1. 文档目的

本文档用于在编辑器 AI 入口、流式返回、结果应用和权限态发生变更后，提供统一回归清单，避免出现“能调接口但编辑器里用起来不稳”的情况。

## 2. 适用场景

以下改动后建议至少跑一轮本文档：

1. `@blocknote/xl-ai` 接入方式调整
2. `@blocknote/xl-ai/server` 宿主逻辑调整
3. 编辑器 AI Prompt、动作集合或返回结构调整
4. 只读 / 可编辑权限逻辑调整
5. 流式协议或错误处理逻辑调整

## 3. 必跑场景

### 3.1 入口

1. 选区入口可以正常出现。
2. Slash 入口可以正常触发。
3. 只读状态下入口展示与可用性符合预期。

### 3.2 动作

第一阶段最小动作建议至少验证：

1. 改写
2. 扩写
3. 缩写
4. 翻译
5. 总结

### 3.3 流式体验

1. AI 结果可以流式返回。
2. 用户可以在生成过程中看到状态变化。
3. 中断或失败时 UI 不应卡死。

### 3.4 结果处理

1. 用户可以应用 AI 结果。
2. 用户可以放弃 AI 结果。
3. 应用结果后正文结构不应明显损坏。

### 3.5 权限态

1. 只读用户不能应用 AI 结果。
2. 无权限用户不能调用编辑器 AI。
3. 权限失败时错误提示可理解。

## 4. 重点观察项

回归时重点观察：

1. 是否出现结果流回来了但不能正确落入编辑器
2. 是否出现 AI 输出破坏 block 结构
3. 是否出现只读态仍能应用结果
4. 是否出现失败后编辑器卡在不可继续交互状态

## 5. 最小交接信息

每次较大编辑器 AI 改动交接时，建议至少记录：

1. 本次改动涉及哪些入口或动作
2. Prompt 或模型配置是否变化
3. 返回结构或应用逻辑是否变化
4. 已跑过哪些动作场景
5. 已知不稳定点是什么

## 6. 通过口径

编辑器 AI 回归通过的最小标准如下：

1. 两个主要入口可用。
2. 五个基础动作可跑通。
3. 流式返回、应用、放弃三条链路都成立。
4. 权限态正确，失败态不致命。

## 7. 当前结论

编辑器 AI 最怕“演示能跑、真实编辑就出问题”。本文档的价值就是确保每次改动后，都至少把入口、动作、流式、应用和权限这五层再走一遍。

## 8. M5 实施补充

1. API 返回必须保留 UI Message Data Stream，而不是统一 JSON envelope。
2. 未认证和不存在文档的请求必须在 provider 调用前失败。
3. 只读用户、协同断线、editor 已销毁或目标 block 已变化时不得应用 AI 结果。
4. `packages/document` 的上下文 builder 必须限制标题、当前 block 和邻近 block 长度，不得触发 RAG。
5. 当前 M5 依赖必须保持 `ai@6` 与 `@blocknote/xl-ai@0.51.4` 对齐。
