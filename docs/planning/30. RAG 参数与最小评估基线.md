# 30. RAG 参数与最小评估基线

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段 RAG 默认参数、调参边界和最小评估方法。
2. 本文档聚焦“参数先怎么定、改动后怎么回归、最小验收怎么看”。
3. RAG 底座与索引链路分别以 [23. RAG 检索方案选型](../decisions/23.%20RAG%20检索方案选型.md) 和 [25. 搜索与索引任务编排设计](../architecture/25.%20搜索与索引任务编排设计.md) 为准。
4. 当前实施路线与验收口径以 [00. 当前实施路线图](./00.%20当前实施路线图.md) 和 [10. 第一阶段验收标准与测试策略](./10.%20第一阶段验收标准与测试策略.md) 为准。

## 1. 文档目的

本文档用于把第一阶段 RAG 的默认参数和最小评估方法收口成一套可执行基线，避免后续每次效果不理想时都同时改 chunk、topK、embedding、Prompt 和模型，最后无法判断到底哪里退化。

## 2. 核心结论

第一阶段不追求一次性把 RAG 调到“最优”，而是先建立：

1. 一套默认参数
2. 一套最小样例集
3. 一套改动后可回归的检查方法

## 3. 第一阶段参数基线目标

第一阶段参数基线至少要保证：

1. 文档可以稳定进入索引。
2. 搜索能命中核心文档。
3. 问答能明显依赖知识库内容而不是大面积脱离上下文。
4. Citation 能回到原文块或原文段落范围。
5. 改动参数后可以判断是否变差。

## 4. 第一阶段建议固定的参数项

第一阶段建议优先固定以下参数类别：

1. chunk 策略
2. retrieval `topK`
3. 回答上下文总预算
4. 单 chunk 截断上限
5. embedding 模型配置
6. 回答模型配置

## 5. chunk 策略基线

### 5.1 结论

第一阶段建议采用“结构优先、长度兜底”的 chunk 策略，而不是纯字符数暴力切分。

### 5.2 推荐原则

1. 尽量沿 BlockNote block 边界切分。
2. 尽量保留标题层级与块定位信息。
3. 避免切得过碎，导致召回结果噪声化。
4. 避免单 chunk 过大，导致上下文浪费。

### 5.3 第一版建议

第一版建议先采用以下保守方向：

1. 以 block 组合作为基础 chunk 单元。
2. 保留 `blockId`、`headingPath`、`snapshotVersion` 等定位元数据。
3. 允许少量 overlap，但不要过大。
4. 单 chunk 保持中等长度，必要时再按回归结果增减。

## 6. retrieval `topK` 基线

### 6.1 结论

第一阶段建议使用较小、可控的 `topK`，不要一开始拉得很大。

### 6.2 原因

如果 `topK` 过大，通常会带来：

1. 上下文噪声增加。
2. 引用来源过多。
3. 成本与延迟上升。
4. 回答更容易被无关内容干扰。

### 6.3 第一版建议

第一版可以先按场景区分：

1. 搜索页 `topK` 偏小
2. 问答场景 `topK` 略高但仍受控

重点不是先把数字调大，而是先确认：

1. 正确文档能否进入前几位。
2. 前几条 chunk 是否足以支撑回答。

## 7. 上下文预算基线

第一阶段必须控制上下文总预算，不能把所有检索结果无上限拼给模型。

建议至少同时限制：

1. 总上下文预算
2. 单 chunk 最大长度
3. 多轮问答时保留的历史消息数量

推荐原则如下：

1. 按检索排序依次装入上下文。
2. 超出预算即停止追加。
3. 单 chunk 过长时先截断，再决定是否继续拼接其他 chunk。

## 8. 第一阶段暂不复杂化的点

第一阶段建议先不同时引入以下复杂变量：

1. hybrid retrieval
2. BM25 兜底排序
3. rerank
4. query rewrite
5. HyDE
6. 多 embedding 并行
7. 多回答模型自动路由

原因很简单：

1. 变量过多时很难判断问题出在哪。
2. 最小闭环还没稳定时，复杂增强只会放大排查成本。

## 9. 最小评估分层

第一阶段最小评估建议分成三层：

1. 索引与检索评估
2. 问答与引用评估
3. 改动后回归稳定性评估

## 10. 索引与检索评估

至少要验证：

1. 文档更新后能进入新的索引版本。
2. 搜索能命中正确文档或正确主题。
3. 检索结果带正确的文档与块定位信息。
4. 无权限文档不会进入当前用户可见结果。

## 11. 问答与引用评估

至少要验证：

1. 回答明显依赖检索内容。
2. 无答案时模型能说明不确定或拒答，而不是强编。
3. Citation / 来源可以跳到原文。
4. 回跳后能尽量落到正确块或正确片段附近。

## 12. 最小样例集建议

第一阶段建议维护一批非常小但稳定的样例集，不求多，但要覆盖关键场景。

建议至少包含：

1. 定义说明类问题
2. 权限规则类问题
3. 配置说明类问题
4. 长文档摘要类问题
5. 精确术语检索类问题

每条样例建议记录：

1. 问题
2. 期望命中文档
3. 期望命中块或主题
4. 是否允许模糊回答
5. 备注

## 13. 第一阶段最小验收口径

第一阶段 RAG 不要求“非常聪明”，但至少应满足：

1. 关键问题能稳定命中正确文档。
2. 回答不会大面积脱离知识库瞎编。
3. 来源和引用可以追溯。
4. 文档更新后，新版本内容可以重新进入检索。

## 14. 调参顺序建议

第一阶段调参建议遵循以下顺序：

1. 先确认索引是否正确。
2. 再确认检索是否命中正确文档。
3. 再调 chunk 大小与 `topK`。
4. 再观察回答效果。
5. 最后才考虑换 embedding、Prompt 或更复杂策略。

不要一开始就同时更改：

1. embedding
2. chunk
3. `topK`
4. Prompt
5. 回答模型

否则很难判断到底是哪项导致效果变化。

## 15. AI / 代理协作建议

如果让 AI 编码代理参与 RAG 调整，最合适的输入不是一句“帮我优化效果”，而是给它：

1. 当前参数
2. 当前样例集
3. 当前失败现象
4. 允许调整的变量范围

这样才能让改动更可回归、可解释。

## 16. 当前结论

DocWeave 第一阶段 RAG 参数与评估的关键，不是追求最优，而是先建立一条稳定的调优闭环：

```text
默认参数
  -> 小样例集
  -> 改动后回归
  -> 再决定是否继续增强
```

只要这条闭环成立，后续无论人工调优还是 AI 辅助调优，都会稳很多。
