# 00. 当前实施路线图

阅读说明：

1. 本文档用于回答“现在应该先做什么、后做什么”。
2. 本文档只保留稳定的阶段顺序和最小闭环，不展开到全部实现细节。
3. 技术基线以 [01. 最终技术方案确认](../decisions/01.%20最终技术方案确认.md) 为准。
4. 架构边界以 [01. 整体架构设计](../architecture/01.%20整体架构设计.md) 为准。

## 1. 当前目标

当前项目的第一阶段目标不是一次做完整平台，而是先建立一个可持续演进的最小闭环。

第一阶段应优先打通以下主链路：

1. 账号与权限基础。
2. 文档编辑与协同。
3. 稳定快照与索引。
4. 编辑器 AI 最小闭环。
5. RAG 搜索与问答最小闭环。

## 2. 执行原则

后续阶段按以下原则推进：

1. 先补一个可稳定跑通的最小闭环。
2. 再把相关入口收口成一致行为。
3. 再补最小回归测试。
4. 最后才扩展体验和基础设施复杂度。

当前前端技术基线补充说明：

1. 产品 UI 组件库统一采用 `Mantine`。
2. `Tailwind CSS v4` 继续保留，用于布局与工具类样式。
3. BlockNote 编辑器 UI 适配层统一采用 `@blocknote/mantine`。

明确延后事项：

1. 第一阶段不急于引入 `BullMQ`。
2. 第一阶段不急于接入 `assistant-ui`。
3. 第一阶段不急于扩展复杂文件后处理和导出能力。

跨阶段规划提醒：

1. ID 统一不是当前任一功能主线里的顺手改造项，应按独立规划推进。
2. 新增资源主键优先统一采用 UUID；历史认证链路与历史资源 ID 的迁移单独开 change。
3. 详细说明见 [51. ID 统一策略规划说明](./51.%20ID%20%E7%BB%9F%E4%B8%80%E7%AD%96%E7%95%A5%E8%A7%84%E5%88%92%E8%AF%B4%E6%98%8E.md)。

## 3. 阶段路线

### 3.1 M1 工程骨架与基础设施联通

目标：把 Monorepo、四端应用和基础依赖跑起来。

最小任务：

1. 初始化 `apps/web`、`apps/api`、`apps/collab`、`apps/worker`。
2. 初始化 `packages/shared`、`packages/auth`、`packages/database`、`packages/editor`、`packages/ai`、`packages/rag`、`packages/collaboration`。
3. 本地联通 `PostgreSQL / Redis / Qdrant / MinIO`。
4. 建立最小环境变量模板和启动方式。

### 3.2 M2 主业务元数据与单人编辑

目标：用户可以登录、进入工作台、打开真实文档页并进行单人编辑。

最小任务：

1. 完成认证与会话。
2. 完成工作区、空间、文档基础 API。
3. 前端展示空间树和文档页。
4. 接入 `BlockNote` 单人编辑器。
5. 产品页面交互统一收口到 `Mantine` 组件体系。

### 3.3 M3 协同最小闭环

目标：两个浏览器可以同时编辑同一篇文档，并看到基础在线状态。

最小任务：

1. `apps/collab` 独立启动并接入文档房间。
2. 前端接入 `Yjs + Hocuspocus`。
3. 接入协同 token 签发与校验语义。
4. 同步正文和基础 presence。

### 3.4 M4 稳定快照与索引基线

目标：协同内容可以进入稳定快照，再触发最小索引任务。

最小任务：

1. 定义 `snapshotVersion` 语义。
2. 落地 `document_snapshots`。
3. 落地 `rag_index_jobs`。
4. Worker 通过数据库轮询执行索引任务。
5. 打通 `snapshot -> chunk -> embedding -> qdrant`。

### 3.5 M5 编辑器 AI 最小闭环

目标：文档页支持最小的改写、扩写、缩写、翻译、总结动作。

最小任务：

1. 前端接入 `@blocknote/xl-ai`。
2. `apps/api` 宿主接入 `@blocknote/xl-ai/server`。
3. `packages/ai` 通过 `Vercel AI SDK` 提供模型能力。
4. 至少支持选区入口和 Slash 入口。
5. 支持流式返回与应用 / 放弃结果。

### 3.6 M6 稳定快照、导出与 RAG 前处理基线

目标：文档稳定快照、导出与索引前处理统一收口到 `BlockNote` 官方服务端工具链。

最小任务：

1. 新增 `packages/document` 或等价模块，封装 `@blocknote/server-util`。
2. 用统一入口处理快照解析、导出和 RAG 前处理。
3. 明确 `snapshot -> server-util preprocess -> chunk -> embedding -> qdrant` 链路。
4. 为后续导出任务预留统一接口。

### 3.7 M7 RAG 搜索与问答最小闭环

目标：用户可以搜索知识库、发起问答，并回到引用原文。

最小任务：

1. 接通 `packages/rag` 的 Qdrant backend。
2. 接通 `POST /api/rag/search`。
3. 接通 `POST /api/rag/chat`。
4. 返回最小 Citation：`documentId + snapshotVersion + blockId`。
5. 前端实现轻量搜索页和聊天页。

### 3.8 M8 队列与体验增强

目标：在主链路稳定后，再升级调度实现和聊天交互层。

最小任务：

1. 将数据库轮询调度替换为 `BullMQ`。
2. 视需要为聊天 / Copilot 引入 `assistant-ui`。
3. 增强文件后处理、导出和重建索引任务。

## 4. 当前优先建议

如果现在开始真正搭建工程，建议按下面顺序启动：

1. 先完成 M1。
2. 再完成 M2，让文档页能真正打开。
3. 然后优先推进 M3 和 M4，先把协同、快照和索引输入链路建立起来。
4. 最后补 M5 和 M6，把 AI 和 RAG 做成产品闭环。

## 5. 当前结论

DocWeave 的第一阶段不追求“技术栈全装齐”，而是先按下面顺序建立最小闭环：

```text
工程骨架
  -> 单人编辑
  -> 协同
  -> 稳定快照
  -> 索引
  -> 编辑器 AI
  -> RAG 搜索与问答
```

只要这个顺序不乱，后续再接 `BullMQ`、`assistant-ui`、导出和更复杂的检索能力都会顺很多。
