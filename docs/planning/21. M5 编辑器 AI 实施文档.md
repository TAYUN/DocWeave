# 21. M5 编辑器 AI 实施文档

阅读说明：

1. 本文档负责把 DocWeave 编辑器 AI 的第一阶段能力写成可实施、可验收的范围与落地边界。
2. 本文档聚焦“支持哪些动作、从哪里触发、结果怎么应用、第一阶段不做什么”。
3. 若需看 BlockNote 服务端接入边界，继续读 [22. BlockNote 服务端能力接入设计](../architecture/22.%20BlockNote%20服务端能力接入设计.md)。
4. 若需看最终技术总基线，优先读 [01. 最终技术方案确认](../decisions/01.%20最终技术方案确认.md)。

## 1. 背景

在当前阶段文档中，项目已经明确了以下基础共识：

1. 编辑器采用 `@blocknote/core + @blocknote/react + @blocknote/mantine`。
2. 编辑器 AI 前端优先采用 `@blocknote/xl-ai`。
3. 编辑器 AI 后端由 `apps/api` 宿主 `@blocknote/xl-ai/server`。
4. 模型能力统一由 `packages/ai` 基于 `Vercel AI SDK` 提供。
5. 文档服务端处理和上下文前处理优先复用 `@blocknote/server-util` 与 `packages/document`。
6. 编辑器 AI、文档 Copilot、知识库问答是三类不同场景，不应混成一套模糊交互壳。

当前已经明确但仍需收口成需求文档的部分包括：

1. 第一阶段到底支持哪些编辑器 AI 动作。
2. 编辑器中从哪些入口触发 AI。
3. `@blocknote/xl-ai` 在 DocWeave 中采用哪些能力，哪些能力暂不启用。
4. AI 返回结果如何与当前 BlockNote 文档结合。
5. 哪些能力本阶段不做。
6. 如何验收第一阶段编辑器 AI 已经闭环。

因此，本阶段需要补齐第一版编辑器 AI 实施文档，让后续人工开发或 AI 编码代理都能按统一边界推进。

## 2. 目标

### 2.1 业务目标

让用户在文档编辑器中直接调用 AI，对当前文档内容进行基础写作辅助，而不必先进入独立聊天页。

第一阶段完成后，用户进入文档编辑页应能完成：

1. 选中文本后触发 AI 动作。
2. 在无选区情况下触发基础写作动作。
3. 对当前内容执行改写、扩写、缩写、翻译、总结等操作。
4. 查看 AI 返回结果。
5. 选择将结果插入文档或替换当前选区。

### 2.2 实施目标

1. 在 `apps/web` 中完成 `@blocknote/xl-ai` 的最小可用接入。
2. 在 `apps/api` 中完成 `@blocknote/xl-ai/server` 的宿主接入。
3. 在 `packages/ai` 中建立编辑器 AI 对应的模型运行时与配置边界。
4. 在 `packages/document` 中建立必要的文档上下文前处理能力。
5. 支持流式返回 AI 结果。
6. 支持将 AI 结果安全地应用回当前编辑器。

## 3. 范围

### 3.1 本阶段包含

1. 编辑器内 AI 最小可用交互入口。
2. 选区类 AI 动作。
3. 无选区基础写作类 AI 动作。
4. 当前文档上下文提取与适配。
5. `@blocknote/xl-ai/server` 接口接入。
6. 流式 AI 结果返回。
7. 结果插入或替换当前内容。
8. 基础错误提示与取消交互。

### 3.2 本阶段不包含

1. 不实现独立右侧聊天面板。
2. 不实现完整文档 Copilot。
3. 不实现知识库 RAG 检索增强写作。
4. 不实现 Citation 插入。
5. 不实现多人协同状态下的 AI 冲突消解。
6. 不实现 AI 调用审计、额度、计费。
7. 不实现复杂工具调用工作流。
8. 不实现自动改写整篇文档的长流程 Agent。

本阶段重点是：

```text
BlockNote 内 AI 触发
  -> 组装当前文档局部上下文
  -> 调用 @blocknote/xl-ai/server
  -> packages/ai 提供模型能力
  -> 返回结果
  -> 用户确认后应用回编辑器
```

## 4. 用户与使用场景

### 4.1 目标用户

本阶段主要面向：

1. 文档编写者。
2. 产品和技术文档整理者。
3. 项目协作成员。

### 4.2 核心场景

#### 场景一：选区改写

1. 用户在文档中选中一段文字。
2. 用户通过编辑器 AI 入口发起动作。
3. 用户选择“改写”。
4. 系统将选区内容作为主上下文提交。
5. AI 返回改写结果。
6. 用户确认后替换原选区。

#### 场景二：选区翻译

1. 用户选中一段内容。
2. 用户选择“翻译”。
3. AI 返回翻译结果。
4. 用户确认后替换原选区或插入到后方。

#### 场景三：无选区扩写

1. 用户光标位于某段落或空行。
2. 用户触发“扩写”。
3. 系统提取当前块和附近上下文。
4. AI 返回扩写结果。
5. 用户确认后插入到当前块后方。

#### 场景四：总结当前内容

1. 用户选中一段较长内容，或在当前文档中触发总结动作。
2. AI 返回简短总结。
3. 用户可将结果插入为新段落，而不是强制覆盖原文。

## 5. 实施定位

### 5.1 模块职责

推荐边界如下：

```text
apps/web
  -> @blocknote/xl-ai 前端交互
  -> 编辑器内 AI 入口
  -> 展示结果与应用结果

apps/api
  -> 宿主 @blocknote/xl-ai/server
  -> 做权限与请求校验

packages/ai
  -> 模型运行时
  -> provider 配置
  -> 流式输出与 AI runtime 边界

packages/document
  -> 文档上下文前处理

packages/rag
  -> 本阶段不直接参与编辑器 AI 主链路
```

### 5.2 与聊天场景的关系

编辑器 AI 不等于知识库问答。

区别如下：

1. 编辑器 AI 主要围绕当前文档或当前选区。
2. 知识库问答面向多文档检索。
3. 编辑器 AI 更偏局部写作动作。
4. 知识库问答更偏多轮对话和来源追溯。

因此，本阶段编辑器 AI 不强依赖 `assistant-ui`。

## 6. 推荐交互入口

### 6.1 第一阶段推荐入口

第一阶段建议至少支持以下两个入口：

1. 选区入口。
2. Slash Menu 入口。

### 6.2 为什么不先做复杂侧边面板

因为第一阶段最重要的是验证：

1. 触发链路是否通。
2. 上下文是否正确。
3. `@blocknote/xl-ai/server` 与业务权限链路是否顺畅。
4. 结果是否能安全写回编辑器。

侧边面板、历史记录、复杂会话态都不是第一阶段阻塞项。

## 7. 第一阶段建议支持的 AI 动作

建议至少支持以下动作：

1. `rewrite`
2. `expand`
3. `shorten`
4. `translate`
5. `summarize`

如 `@blocknote/xl-ai` 的现成能力模型支持更贴近官方命名，可在实现层做动作映射，但产品语义仍以上述五类基础写作动作为准。

动作语义建议如下：

1. `rewrite`
   - 对原意基本不变地优化表达
2. `expand`
   - 在当前内容基础上扩写
3. `shorten`
   - 保留核心信息并压缩篇幅
4. `translate`
   - 翻译为指定目标语言
5. `summarize`
   - 输出简要总结

## 8. 上下文组装规则

### 8.1 必要上下文

第一阶段建议编辑器 AI 请求至少携带：

1. `documentId`
2. `documentTitle`
3. `action`
4. `selectedText`
5. `currentBlockText`
6. `surroundingText`
7. 可选用户补充 prompt

### 8.2 组装原则

1. 有选区时，`selectedText` 优先级最高。
2. 无选区时，以当前块和附近内容为主。
3. 不把整篇文档无上限拼给模型。
4. 本阶段不引入 RAG 检索上下文。
5. 若需要服务端解析块级上下文，优先经由 `packages/document` 与 `@blocknote/server-util` 处理，而不是随意手写文档结构解析。

## 9. API 设计要求

### 9.1 接口

本阶段固定采用：

```text
apps/api 宿主的 @blocknote/xl-ai/server endpoint
```

即：

1. 前端协议优先贴近 `@blocknote/xl-ai` 官方接法。
2. 业务侧不额外先发明一套等价接口协议。

### 9.2 接口职责

接口负责：

1. 校验当前用户是否有目标文档访问权限。
2. 校验动作是否合法。
3. 组装编辑器 AI 所需上下文。
4. 调用 `packages/ai`。
5. 返回流式结果。

接口不负责：

1. 知识库检索。
2. 协同正文同步。
3. 直接修改数据库正文。

## 10. 结果应用策略

### 10.1 第一阶段建议支持两种应用方式

1. 替换选区。
2. 插入到当前块后方。

### 10.2 建议规则

1. `rewrite`、`shorten`、`translate`
   - 默认适合替换选区
2. `expand`
   - 默认适合插入到当前块后方
3. `summarize`
   - 默认适合插入新段落，而不是覆盖原文

### 10.3 为什么不直接自动落文档

因为第一阶段应优先保证用户可控。

至少应让用户能够：

1. 查看结果。
2. 决定是否应用。
3. 放弃本次结果。

## 11. 错误与取消处理

### 11.1 至少需要处理的错误

1. 网络错误。
2. 模型调用失败。
3. provider 配置缺失。
4. 权限不足。
5. 请求超时。

### 11.2 至少需要支持的取消行为

1. 用户在生成中主动停止。
2. 用户关闭当前 AI 菜单。
3. 用户不应用本次结果。

## 12. 权限与安全边界

本阶段需要明确以下边界：

1. 编辑器 AI 请求必须基于当前登录用户。
2. 只有可访问该文档的用户才能调用对应文档的编辑器 AI。
3. 本阶段不要求 AI 自己读取未授权知识库。
4. 模型密钥与 provider 配置不应暴露给前端为长期可信真相。
5. 若当前协同会话为只读状态，前端应禁用“应用到文档”行为。

## 13. 验收标准

第一阶段完成后，至少应满足：

1. 用户可以在文档页触发编辑器 AI。
2. 至少两个入口可用：
   - 选区入口
   - Slash Menu 入口
3. 至少五个基础动作可用：
   - 改写
   - 扩写
   - 缩写
   - 翻译
   - 总结
4. AI 响应支持流式返回。
5. 用户可以选择应用或放弃结果。
6. 文档页原有编辑和协同基础能力不被破坏。

## 14. 第一阶段不做的明确事项

为了避免范围失控，本阶段明确不做：

1. 独立聊天面板。
2. 文档 Copilot。
3. RAG 增强编辑。
4. Citation 插入。
5. 评论联动。
6. 批量改写整篇文档。
7. 多步 Agent 工作流。

## 15. 当前结论

DocWeave 编辑器 AI 第一阶段实施的核心不是“把所有 AI 能力都塞进编辑器”，而是先建立一个清晰、可控、可验证的最小闭环：

```text
@blocknote/xl-ai 入口
  -> 局部上下文组装
  -> @blocknote/xl-ai/server
  -> packages/ai runtime
  -> 流式返回结果
  -> 用户确认后应用回编辑器
```

只要这条链路建立起来，后续再扩展文档 Copilot、RAG 增强写作和引用插入，都会顺很多。
