# 43. 协同与索引失败排障手册

阅读说明：

1. 本文档负责提供协同连接、稳定快照、索引任务和搜索结果异常时的最小排障路径。
2. 可观测性基线以 [17. 可观测性基线](../decisions/17.%20可观测性基线.md) 为准。
3. 协同与索引设计分别以 [13. 协同服务设计](../architecture/13.%20协同服务设计.md)、[24. 版本与稳定快照设计](../architecture/24.%20版本与稳定快照设计.md)、[25. 搜索与索引任务编排设计](../architecture/25.%20搜索与索引任务编排设计.md) 为准。

## 1. 文档目的

本文档用于在协同与索引链路出问题时，提供一条可重复执行的排障顺序，避免大家看到“搜不到”“不同步”“回答是旧内容”时，直接从浏览器或向量库某一头乱猜。

## 2. 先判断属于哪一类问题

先把问题归到下面四类之一：

1. 协同连接建立失败
2. 协同连接已建立但内容不同步
3. 快照 / 索引任务失败
4. 索引成功但搜索 / 问答结果异常

## 3. 协同连接建立失败

排查顺序建议如下：

1. 看 `apps/api` 是否成功签发协同 token
2. 看 `apps/collab` 是否收到连接请求
3. 看 token 校验是否失败
4. 看房间名是否与目标 `workspaceId + documentId` 匹配

重点日志字段：

1. `event`
2. `roomName`
3. `workspaceId`
4. `documentId`
5. `userId`
6. `errorCode`

## 4. 协同连接已建立但不同步

排查顺序建议如下：

1. 确认两个浏览器是否连入同一房间
2. 确认可编辑用户是否真的拥有 `canEdit`
3. 确认 Yjs 绑定是否建立
4. 确认标题和正文是否走了同一协同链路

重点观察：

1. 是否只同步了正文没同步标题
2. 是否 Presence 正常但正文不同步
3. 是否只读用户错误地广播了更新

## 5. 快照 / 索引任务失败

排查顺序建议如下：

1. 先确认是否生成了目标 `snapshotVersion`
2. 再看 `rag_index_jobs` 是否创建成功
3. 再看 worker 是否成功领取任务
4. 再看失败发生在 `preprocessing / chunking / embedding / upserting / publishing` 哪一步

重点排查字段：

1. `documentId`
2. `targetSnapshotVersion`
3. `jobId`
4. `stage`
5. `errorCode`

## 6. 搜索 / 问答结果异常

先分清是哪类异常：

1. 完全搜不到
2. 能搜到但命中旧内容
3. 能搜到但混入无权限内容
4. Citation 无法回跳

排查顺序建议如下：

1. 确认 `latestIndexedVersion` 是否已更新
2. 确认当前检索是否只读该版本
3. 确认 Qdrant payload 是否含正确 `documentId / snapshotVersion / blockId`
4. 确认 API 权限过滤是否生效

## 7. 常见误判

排障时需要避免以下误判：

1. 浏览器里看到新内容，就误以为索引也已更新
2. Qdrant 里有向量，就误以为当前用户一定能搜到
3. 连接建立了，就误以为内容同步一定正常
4. AI 回答返回了，就误以为 Citation 一定对应正确版本

## 8. 推荐排障顺序总原则

遇到问题时，建议始终按下面顺序查：

```text
权限 / 准入
  -> 版本
  -> 任务状态
  -> 结果表现
```

不要反过来直接从结果猜权限和版本。

## 9. 升级为代码修复前应补的证据

在真正改代码前，建议至少先确认：

1. 问题发生在哪个运行时
2. 问题对应哪一个版本或哪一个任务
3. 是否能稳定复现
4. 是否已有结构化日志可以证明失败点

## 10. 当前结论

协同与索引问题往往不是“单点坏掉”，而是准入、版本、任务和结果链路中的某一段断了。本文档的目标，就是把排障顺序先固定下来，减少盲猜和误修。
