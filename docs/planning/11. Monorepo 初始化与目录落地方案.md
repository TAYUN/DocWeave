# 11. Monorepo 初始化与目录落地方案

阅读说明：

1. 本文档负责把当前技术方案转成可执行的工程初始化步骤和目录落地方案。
2. 本文档聚焦“目录怎么真正建出来、初始化顺序怎么排”。
3. 当前阶段顺序以 [00. 当前实施路线图](./00.%20当前实施路线图.md) 为准。

## 1. 第一版目标

第一版工程目标不是一次性把所有模块做完，而是先搭好后续不会反复推翻的边界。

第一版需要优先完成：

1. `apps/web`
2. `apps/api`
3. `apps/collab`
4. `apps/worker`
5. `packages/shared`
6. `packages/auth`
7. `packages/database`
8. `packages/editor`
9. `packages/ai`
10. `packages/rag`
11. `packages/collaboration`
12. `packages/ui`
13. `packages/config`

## 3. apps 落地方案

### 3.1 apps/web

技术基线：

1. `React SPA`
2. `Vite`
3. `TanStack Router`
4. `TanStack Query`
5. `Tailwind CSS v4`
6. `shadcn/ui`
7. `BlockNote`

建议优先实现：

1. 登录页。
2. 工作台首页。
3. 空间树 / 文档导航。
4. 文档页。
5. 搜索页。
6. 轻量聊天页。

### 3.2 apps/api

技术基线：

1. `AdonisJS v7`
2. `TypeScript`
3. `VineJS`

建议优先实现的 API：

1. `POST /api/auth/login`
2. `POST /api/auth/logout`
3. `GET /api/auth/me`
4. `GET /api/spaces`
5. `GET /api/spaces/:spaceId/tree`
6. `GET /api/documents/:documentId`
7. `PATCH /api/documents/:documentId`
8. `POST /api/collaboration/token`
9. `POST /api/documents/:documentId/snapshots`
10. `POST /api/ai/editor`
11. `POST /api/rag/search`
12. `POST /api/rag/chat`

### 3.3 apps/collab

技术基线：

1. `Hocuspocus`
2. `Yjs`

第一阶段优先能力：

1. 启动独立协同服务。
2. 文档房间连接。
3. Yjs update 转发。
4. Awareness 转发。
5. 协同 token 校验接入点。

### 3.4 apps/worker

第一阶段优先能力：

1. 轮询数据库任务记录。
2. 领取索引任务。
3. 执行 chunk / embedding / upsert qdrant。
4. 回写任务状态和 `latestIndexedVersion`。

## 4. packages 落地方案

### 4.1 packages/editor

职责：

1. `BlockNote` 统一封装。
2. 文档编辑器组件。
3. 初始化内容逻辑。
4. 协同接入点。
5. 编辑器 AI 结果应用逻辑。

### 4.2 packages/ai

职责：

1. `Vercel AI SDK` provider 封装。
2. 流式输出封装。
3. Prompt builder。
4. 编辑器 AI 与问答 AI 的统一调用入口。

### 4.3 packages/rag

职责：

1. chunk 切分。
2. `Qdrant` backend。
3. 检索、上下文构建、问答流程。
4. Citation 结构。

### 4.4 packages/collaboration

职责：

1. 房间命名规则。
2. 协同 token contract。
3. Awareness / Presence 结构。

## 5. 初始化顺序

推荐按以下顺序执行：

1. 初始化 Monorepo 外壳。
2. 初始化 `apps/web`。
3. 初始化 `apps/api`。
4. 初始化 `apps/collab`。
5. 初始化 `apps/worker`。
6. 初始化共享包。
7. 打通主业务最小闭环。

## 6. 第一阶段不建议提前做的事

当前不要在初始化阶段提前做：

1. `BullMQ` 完整接入。
2. `assistant-ui` 产品化聊天壳。
3. 复杂导出任务。
4. 复杂多 Agent 能力。
5. 过度设计的微服务拆分。

## 7. 当前结论

当前项目应直接按本文档落地第一版 Monorepo 工程，不再摇摆以下问题：

1. 前端是否使用 SPA。
2. 主业务后端是否使用 `AdonisJS v7`。
3. 协同服务是否独立。
4. RAG 是否直接采用 `Qdrant`。
5. 第一阶段是否先不上 `BullMQ`。
