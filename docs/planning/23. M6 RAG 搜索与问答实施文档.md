# 23. M6 RAG 搜索与问答实施文档

阅读说明：

1. 本文档负责把 RAG 搜索与问答最小闭环转成可执行实施范围。
2. RAG 输入与 Citation 设计以 [26. RAG 与 Citation 设计](../architecture/26.%20RAG%20与%20Citation%20设计.md) 为准。
3. 当前阶段顺序以 [00. 当前实施路线图](./00.%20当前实施路线图.md) 和 [52. 后续阶段实施规范](./52.%20后续阶段实施规范.md) 为准。
4. 跨边界数据、统一 envelope、adapter 和 view-model 以 [05. 数据契约与适配层设计](../architecture/05.%20数据契约与适配层设计.md) 为准。

## 1. 阶段目标

让用户可以在知识库中搜索、发起问答，并回跳到原文块。

## 2. 本阶段包含

1. `POST /api/rag/search`。
2. `POST /api/rag/chat`。
3. `packages/rag` 的 `Qdrant` backend。
4. 最小 Citation 回跳。
5. 轻量搜索页与聊天页。
6. 仅对稳定快照产生的已索引内容进行检索和问答。
7. 检索前由 API 计算用户可见文档范围。
8. 复用 M4 已有的数据库任务记录和 worker，不在 M6 引入 `BullMQ`。
9. 成功响应使用 `ApiSuccessResponse<T>`，HTTP 错误使用 `ApiErrorResponse`；页面通过 `apps/web/src/lib/api.ts` 和 feature view-model 消费数据。

接口约定补充：

1. `POST /api/rag/search` 使用 `{ searchText: string, spaceId?: string, limit?: number }`。
2. `POST /api/rag/chat` 使用 `{ message: string, spaceId?: string, topK?: number }`。
3. 两个接口的 `spaceId` 都只是体验筛选输入；API 必须先计算可见文档范围，再在授权范围内缩小。

## 3. 本阶段不包含

1. 导入、导出、OCR、PDF、Office 和复杂文件后处理。
2. 文档发布、分享链接和公开访问。
3. 复杂 rerank 流水线。
4. 高级知识图谱式检索。
5. 完整聊天产品化壳。
6. `BullMQ`、多租户调度控制台和复杂后台任务治理。

## 4. 最小任务

1. API 在检索前计算用户可见文档范围。
2. `packages/rag` 基于范围检索 `Qdrant`。
3. `packages/ai` 流式生成回答。
4. 回答返回 `documentId + snapshotVersion + blockId`。
5. 前端支持跳回原文位置。
6. 对无 `latestIndexedVersion` 的文档不返回可用检索结果。
7. Citation 至少能校验 `documentId`、`snapshotVersion` 和 `blockId` 的对应关系。

## 5. 完成标准

1. 已索引文档可被搜索命中。
2. 问答可流式返回。
3. Citation 可追溯并回到原文。

## 6. M6 前置条件

M6 不要求先完成完整导入导出能力，但必须依赖以下已有基础：

1. M4 已生成稳定快照和索引任务。
2. `latestIndexedVersion` 只在完整索引成功后更新。
3. Qdrant payload 至少能定位到文档和快照版本。
4. API 已具备当前用户和文档访问权限判断。

如果 Citation 需要精确回到块级位置，M6 可以在 `packages/document` 中补充最小的 BlockNote block 元数据预处理，但不得因此扩展为完整导入导出项目。

## 7. M6 交付边界

M6 的唯一主结果是：

```text
已索引稳定快照
  -> 权限过滤
  -> Qdrant 检索
  -> packages/ai 流式回答
  -> Citation 回到原文块
```

导入导出、发布分享和文件 RAG 分别进入 M8、M9 及其后续 change。
