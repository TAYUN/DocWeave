# Cross-Service Change Checklist

本文档用于约束 DocWeave 中“跨多个进程 / 包 / 外部服务”的 change，减少反复亏时间的非业务成本。

适用范围：

1. 同时改到 `apps/api`、`apps/worker`、`apps/collab`、`packages/*` 中两处及以上。
2. 接入新的外部服务，例如 embedding、vector store、对象存储、队列、搜索服务。
3. 需要新增 worker、任务状态、版本状态、快照或持久化协议。

## 1. 先过协议检查，不要直接开写

进入实现前，至少先回答下面 6 个问题：

1. 谁是真相源。
2. 谁分配版本。
3. 谁创建任务。
4. 谁执行任务。
5. 谁更新最终状态。
6. 失败后哪个状态绝不能被回滚或覆盖。

如果这 6 个问题还说不清，优先补 planning / design / contract，而不是继续堆实现。

## 2. 环境变量模板和代码读取必须同批收口

新增运行时配置时，默认同批完成下面 3 件事：

1. 更新代码里的 `config.ts` / 读取逻辑。
2. 更新对应服务的 `.env.example`。
3. 完成一次最小启动验证。

额外要求：

1. 本地开发常见的默认值，能给默认值就不要无意义强制必填。
2. `.env.example` 不是附属文档，而是实现的一部分。
3. 不允许代码读取口径和示例模板长期漂移。

## 3. 每个外部服务都要有一次最小真实联调

不要只依赖单测或类型检查。

每个外部系统至少做一次最小真实验证，例如：

1. embedding 服务：真实发一次请求。
2. Qdrant：真实检查 collection 并 upsert 一条数据。
3. 对象存储：真实上传 / 读取一个最小对象。
4. 队列：真实入队 / 出队一个最小任务。

目标不是做完整自动化，而是尽早暴露权限、资源预置、格式约束和兼容层差异。

## 4. 把第三方约束当成主线输入，而不是事后异常

实现时默认检查这些约束：

1. 权限是否已开通。
2. 模型或资源是否在当前 workspace / region 可用。
3. ID 形态是否满足第三方约束。
4. payload / vector dimensions / metadata 形态是否匹配。
5. OpenAI 兼容层是否真的支持计划使用的参数。

只要这些约束还没确认，就不要假设“SDK 能调通等于功能可用”。

## 5. 收口脚本和流程状态要当成正式工作面

对于使用 `spec-superflow` 的 change，默认把下面这些内容视为实现的一部分：

1. `tasks.md` 勾选状态。
2. `.spec-superflow.yaml` 关键字段。
3. `ssf state rebuild` / `state check`。
4. `closing` 脚本与 `audit` 产物。
5. 生成产物与手写文件的提交边界。

不要把“代码写完”误当成“change 已完成”。

## 6. 新踩坑要尽快沉淀成默认规则

一旦出现真实联调踩坑，优先问的不是“这次怎么修”，而是：

1. 这是不是以后会重复出现。
2. 这是不是说明缺默认规则、模板、脚本或校验。
3. 这条结论应该沉淀到哪里：
   - `docs/planning/`
   - `docs/workflow/`
   - `.env.example`
   - 启动脚本
   - 测试
   - checklist

例如：

1. Qdrant point id 统一用 UUID 或第三方要求的标准格式。
2. 新资源主键优先 UUID。
3. embedding 维度显式配置。
4. OpenAI 兼容层能力先查实，不按官方 OpenAI 经验想当然。

## 7. 当前最值得坚持的一条默认动作

如果只能先执行一条，优先执行这条：

**任何新增服务或外部依赖，都要求“代码读取逻辑 + `.env.example` + 最小启动验证”同批完成。**

这是当前投入最低、但最能减少后续反复排查时间的一条默认机制。
