# UI Review Checklist

这份清单用于评审 DocWeave 的页面方案、线框说明和前端实现。

它的目标不是替代设计，而是帮助主 agent、实现 agent 和 review agent 用同一套口径判断：

- 这个页面是不是在解决明确任务
- 页面结构是否稳定、清晰、可用
- AI 生成结果有没有偏离 DocWeave 的产品气质和实现边界

相关基线：

- 设计基线见 [`DESIGN.md`](../../DESIGN.md)
- 前端实现边界见 [`frontend-mantine-implementation-guide.md`](./frontend-mantine-implementation-guide.md)

## 1. 页面目标

- 这个页面是否有一句话说得清的主任务
- 页面标题、主内容和主操作是否都围绕同一个任务
- 页面是否在“展示信息”和“完成任务”之间有明确重心
- 是否存在页面目的不清，导致区块越加越多的问题

## 2. 信息架构

- 页面是否有明确的主区和次区
- 主区是否承载最重要的信息和最频繁的操作
- 次级信息是否被放在不会打断主任务的位置
- 页面是否避免了多个同权重卡片并列，导致用户看不出先看哪里
- 信息分组是否基于业务语义，而不是只为了版面均衡

## 3. 操作设计

- 主操作是否只有 1 到 2 个，并且足够明确
- 次级操作是否明显弱于主操作，但仍然可发现
- 危险操作是否和常规操作拉开了语义与视觉距离
- 页面是否给出了合理的下一步，而不是只把组件堆出来
- 用户完成一次核心任务时，路径是否顺畅且闭环

## 4. 状态完整性

- `loading` 时页面是否仍保持基本结构，不让布局大幅跳动
- `empty` 时是否说明“这里还没有内容”，并给出下一步动作
- `error` 时是否说明失败原因，并在可能时提供重试或回退动作
- `not found` 时是否明确对象不存在，而不是模糊显示空白
- `disabled`、无权限、受限状态是否清楚说明“为什么不能做”
- 页面是否覆盖了真实业务里的关键边界，而不只是 happy path

## 5. 视觉一致性

- 页面是否符合 DocWeave “稳定、清晰、低噪音”的工作台气质
- 视觉重心是否来自结构和内容，而不是大面积装饰
- 是否误用了营销页式 Hero、重渐变、玻璃感、夸张阴影或强品牌化背景
- 蓝色是否仅用于交互强调和活动状态，而不是铺满大面积区域
- 卡片、表单、状态标签是否保持克制，不抢正文和任务焦点

## 6. Mantine 实现边界

- 是否优先使用 Mantine 原生组件表达结构和状态
- 是否遵循默认主题优先，而不是无必要地到处显式传 `color`、`radius`、`shadow`、`variant`
- 页面级 CSS 是否主要处理布局与节奏，而不是重塑视觉体系
- 是否出现“看起来用了 Mantine，但实际又手写了一套设计系统”的情况
- 如果偏离 Mantine 默认行为，是否有明确理由支撑

## 7. 响应式与可用性

- 小屏下是否优先保留主任务，而不是机械堆叠桌面布局
- 辅助信息在移动端是否后置、折叠或降权
- 触发频繁的核心操作是否仍然容易点击和定位
- 文本层级和间距在小屏下是否仍易读
- 焦点、选中、禁用等交互状态是否保持可辨认

## 8. AI 生成页面常见问题

出现以下情况时，默认应提高警惕：

- 页面很“满”但没有清晰主任务
- 每块内容都长得一样重要，用户无法快速扫描
- 用大量装饰来掩盖信息架构薄弱
- 只实现静态展示，没有考虑真实数据状态和失败分支
- 为了“现代感”加入与 DocWeave 基线不一致的视觉语言
- 组件很多，但没有形成稳定的阅读和操作节奏

## 9. 评审输出建议

做 UI review 时，优先按下面顺序输出：

1. 页面主任务是否成立
2. 信息层级或操作路径是否有问题
3. 状态与边界是否缺失
4. 是否偏离 `DESIGN.md` 与 Mantine 实现边界
5. 剩余风险、验证盲区或建议补充的场景

如果未发现明确缺陷，也建议补一句：

> 未发现明确缺陷，但建议在真实数据、窄屏和空态下再做一轮验证。
