# 项目开发问题记录

本文档用于持续记录 DocWeave 开发过程中遇到的高价值问题。

记录目标：

1. 方便项目做完后统一复盘，而不是只记零散结论。
2. 方便后续写总结、写文章、准备面试时回看真实问题。
3. 让“当时为什么难、后来怎么解、还剩什么尾巴”都有出处。

建议记录标准：

1. 优先记录会反复出现、会影响多人协作、或能展开成一篇完整总结的问题。
2. 不只记“怎么修”，还要记“为什么会发生”。
3. 如果问题还没彻底解决，明确写出当前状态和后续观察点。

---

## 架构与技术选型问题

### 1. 做项目前期，架构方案和技术路线应该如何判断，而不是只靠直觉拼装

- 背景：
  在项目早期，最难的问题往往不是“某个接口怎么写”，而是“这整个项目该用什么技术方案、为什么这样选、这样选的代价是什么”。
- 暴露方式：
  当项目逐步长出 `AdonisJS`、`Tuyau`、`Mantine`、`BlockNote`、`Yjs + Hocuspocus`、`Qdrant`、`spec-superflow` 这些组件后，会开始反过来追问：这些方案到底是不是合适的，选择标准是什么，未来怎么向别人解释。
- 影响：
  如果没有明确的技术选型判断框架，开发过程会更像“先做了再解释”，后面在复盘、面试、写文章时很难系统回答：
  - 为什么不用别的方案？
  - 为什么现在这样拆边界？
  - 哪些是主动取舍，哪些只是当时不知道？
- 当前结论：
  技术选型问题本身值得单独长期记录，不应该只零散落在 `decisions/` 文档里。开发问题台账里也要保留“当时为什么犹豫、判断依据是什么、后来是否证伪”的一手记录。
- 值得复盘的问题：
  做一个中大型项目时，怎样建立一套自己的技术选型判断方法，而不是靠流行度、熟悉度或当下直觉做决定。

### 2. 业务数据契约和 HTTP 响应协议的职责边界不够早明确

- 背景：
  项目已经较早建立了 `packages/contracts`、`packages/adapters`、`application`、`view-model` 的分层，但当接口逐渐变多后，团队才明显意识到“DTO contract”和“HTTP response envelope”不是同一层问题。
- 暴露方式：
  讨论接口统一响应风格时，发现 `packages/contracts` 里定义的是 `data` 内部业务 shape，而不是整个 `res` 外层协议。
- 影响：
  早期会默认以为“既然已经有 contracts，响应风格应该也已经统一了”，但实际上 controller、exception handler 和前端 API 层各自承担了一部分职责，容易出现认知错位。
- 当前结论：
  `packages/contracts` 继续负责 business contract；
  HTTP 响应协议单独由文档、controller / serializer、exception handler 和前端 `api.ts` 收口。
- 值得复盘的问题：
  在项目早期，哪些“看似只是 API 细节”的问题，其实应该单独定义职责边界，而不能默认挂在现有架构名词下面。

### 3. 要不要采用 `{ code, data, msg }` 三段式，不能只按习惯判断

- 背景：
  在统一接口响应风格时，直觉上很容易把“三段式”当成一种天然标准。
- 暴露方式：
  真正对照当前技术栈后才发现，`AdonisJS + Tuyau + TanStack Query` 更自然的模型其实是“HTTP 状态码 + JSON body + 前端统一抛错”。
- 影响：
  如果不先判断技术栈真实受益点，容易为了“看起来统一”再造一层重复协议。
- 当前结论：
  这次项目没有采用 `{ code, data, msg }`，而是保留 HTTP 状态码语义，统一 body 外形。
- 值得复盘的问题：
  在团队决策里，哪些“行业常见写法”其实不应该直接照搬，而要结合框架默认模型重新判断。

## 边界与职责问题

### 4. 统一响应风格这件事，真正的收口点不在业务层，而在入口层

- 背景：
  一开始很容易把“接口风格不统一”理解成 controller 里 return 写得不整齐。
- 暴露方式：
  追完整条链路后发现，真正导致不统一的是入口层职责分散：
  - success envelope 有的走 `serialize()`，有的手写 `{ data }`
  - framework exception 走默认 handler
  - frontend error extraction 主要只认 `message`
- 影响：
  如果只在单个 controller 局部修，会得到“修了一条链路，但同类入口继续漂”的结果。
- 当前结论：
  这类问题的最小正确收口点是：
  - `apps/api/providers/api_provider.ts`
  - `apps/api/app/exceptions/handler.ts`
  - `apps/web/src/lib/api.ts`
- 值得复盘的问题：
  哪些问题看起来像“业务接口问题”，其实根因在系统入口层没有统一。

### 5. scaffold 接口如果长期不补 validator，会在类型和错误协议上双重漂移

- 背景：
  `collaboration`、`rag`、`ai` 这些入口在早期是占位能力，最开始用了 `request.only()` / `request.input()` 直接读参数。
- 暴露方式：
  一旦开始统一接口错误风格，就会发现这些接口没有进入 Vine 校验链路，也拿不到统一 `422 + { message, errors }`。
- 影响：
  这不仅影响服务端校验，也会让 Tuyau registry 对 body 类型的帮助变弱。
- 当前结论：
  即使是 scaffold 能力，也应该尽早补最小 validator，而不是长期裸读请求。
- 值得复盘的问题：
  占位实现在哪个阶段必须从“能跑”升级到“进入正式边界”，否则后面统一成本会突然变高。

## 协议与数据流问题

### 6. 前端真正需要的不是“某种固定响应协议”，而是稳定的错误语义

- 背景：
  讨论时很容易把焦点放在返回体长什么样。
- 暴露方式：
  真正落到 TanStack Query 和登录页提示上，发现页面并不关心后端有没有 `code` 字段，它只关心：
  - 成功时能不能稳定拿到 `data`
  - 失败时能不能稳定拿到 `error.message`
- 影响：
  如果服务端和前端不围绕“错误语义稳定”设计，页面层就会被迫感知多种异常 shape。
- 当前结论：
  统一响应风格的设计重点应该是“前端如何稳定消费”，而不是“协议字段长得是否像模板”。
- 值得复盘的问题：
  在 API 设计里，哪些看起来是后端规范问题，实际上应该从前端消费体验倒推。

## 默认假设与环境基线问题

### 7. 开发环境基线如果不稳定，会制造“像 bug 的非代码问题”

- 背景：
  登录页默认展示了开发账号，但本地数据库状态不一定总是和 seed 一致。
- 暴露方式：
  本次会话里出现了 `POST /api/auth/login -> 400 Invalid user credentials`，最后确认不是接口实现坏了，而是本地数据基线漂了。
- 影响：
  这类问题非常浪费时间，因为表面上看像“功能坏了”，实际是环境初始化没收稳。
- 当前结论：
  需要在文档里明确开发账号依赖 seed，并在开发流程中尽量减少“页面默认账号存在，但数据库不一定有”的情况。
- 值得复盘的问题：
  开发环境里有哪些“默认假设”如果不写清楚，会让排查方向先错半小时。

## 流程与协作问题

### 8. spec-superflow 流程本身也会成为项目复杂度的一部分

- 背景：
  这次改动不仅要改代码，还要起 change、补 proposal/spec/design/tasks/execution-contract、推进状态、做 closing。
- 暴露方式：
  当 `skill` 本体在当前环境不可用时，仍需要靠仓库内 SOP、`ssf` CLI 和 `close-change.mjs` 脚本手动把流程跑完。
- 影响：
  工作流本身带来了治理收益，但也增加了执行成本和状态维护成本。
- 当前结论：
  spec-superflow 很适合中等复杂度以上改动，但团队需要接受：流程工具本身也需要维护、同步和文档化。
- 值得复盘的问题：
  一个项目引入流程治理工具后，什么时候它是在降低复杂度，什么时候它自己也成为需要治理的对象。

## 重复劳动与低效模式

### 9. “本次 change 范围”和“顺手修的小问题”需要更主动地分开

- 背景：
  在处理响应风格 change 的过程中，又顺带暴露了开发账号 seed、README 说明、spec-superflow skill 镜像同步等问题。
- 暴露方式：
  如果不刻意拆分，很容易把“主改动”和“顺手整理”混成一个提交，最后既不利于 review，也不利于回顾。
- 影响：
  提交历史和 change 历史都会变模糊，后续很难回答“这次改动到底解决了什么”。
- 当前结论：
  这次最终拆成了两个提交：
  - 响应风格规范
  - 杂项和开发说明
- 值得复盘的问题：
  在复杂开发会话里，怎样及时识别“主问题”和“伴生问题”，避免提交历史变成一锅粥。

### 10. 生成产物和真实手改文件混在一起时，提交边界很容易被污染

- 背景：
  这次改动里既有手写代码，也有 Tuyau / `ssf` 相关生成产物。
- 暴露方式：
  如果不在提交前单独核对 staged 文件，很容易把无关文件、生成文件、额外脚本或环境说明一起带进去。
- 影响：
  review 成本会升高，后续定位也会更难。
- 当前结论：
  在跨文档、跨后端、跨前端、跨流程工具的 change 中，提交前必须显式核对 staged 范围。
- 值得复盘的问题：
  单仓多工具项目里，怎样建立“提交边界感”，避免每次改动都顺带夹带别的东西。

## 认知盲区与决策误判

### 11. 以为“已有架构名词”已经覆盖了问题，实际上只是换了个名字还没真正定义边界

- 背景：
  项目已经有 `contracts`、`adapters`、`application`、`view-model`、`Tuyau` 等一整套名词，很容易产生“边界已经设计好了”的错觉。
- 暴露方式：
  真正落到响应风格、错误出口、Query 错误语义这些问题时，才发现很多关键点只是“有相近概念”，并没有被明确归属。
- 影响：
  这会让决策变慢，也会让人高估当前架构成熟度。
- 当前结论：
  名词存在不等于边界成立。以后只要遇到“这个不是已经有 X 层负责了吗”的疑问，都值得追问一次：它真的被定义了吗，还是只是看起来像。
- 值得复盘的问题：
  如何区分“项目已经有这层设计”与“项目只是有相关名词，但职责还没真正落地”。

## M4 稳定快照与索引执行问题

真正容易反复亏时间的，不是 M4 某一段代码难写，而是“协议、环境模板、第三方约束、流程工件”这四类非业务代码成本没有被当成主线交付的一部分。

### 12. M4 的真正难点不是建表或写 worker，而是跨进程协议在设计文档里长期留白

- 背景：
  本次推进 M4 时，表面任务看起来是新增 `document_snapshots`、`rag_index_jobs`、`apps/worker` 和 `packages/rag`，好像主要是代码量和依赖量问题。
- 暴露方式：
  真正进入 planning 和 execution 之后才发现，最影响落地速度的不是“少了哪几个包”，而是 `collab -> api -> worker` 之间原本没有现成框架垫层，导致很多关键动作都要先补协议语义：
  - 什么叫 stable snapshot truth
  - `latestSnapshotVersion` / `latestIndexedVersion` 何时更新
  - 同版本 job 如何去重
  - 旧 job 何时 `superseded`
  - API 只负责入队还是直接执行
- 影响：
  如果只把问题理解成“缺实现”，就会低估 planning 的必要性，高估直接开写的可行性。结果不是写慢一点，而是很容易把版本真相、任务状态、发布语义和协同持久化边界混成一团。
- 当前结论：
  M4 这类跨进程链路的核心风险是“协议空白”，不是“代码空白”。真正缩短实现周期的不是更快写代码，而是先把协议和边界钉死。
- 值得复盘的问题：
  以后凡是遇到“一个主线功能跨两个以上进程/包”的 change，是否应该默认先检查协议层是否已经足够落地，而不是先估算实现工作量。

### 13. 环境变量模板、真实本地 `.env` 和运行时读取口径一旦漂移，会反复制造伪 bug

- 背景：
  本次 M4 落地同时新增了 `apps/worker`、百炼 embedding、Qdrant collection 和新的状态接口，环境变量数量明显增加。
- 暴露方式：
  会话里连续出现了多类“像代码 bug，其实是环境基线漂移”的问题：
  - `apps/api/.env.example` 与真实实现仍存在 `QDRANT_HOST/QDRANT_PORT` 旧口径
  - `apps/worker` 读取的是 `QDRANT_URL`
  - `DB_PASSWORD` 在 API 侧允许空值，但 worker 侧最初强制非空
  - `127.0.0.1` 和 `localhost` 在本地启动检查里表现不一致
  - `DASHSCOPE_API_KEY`、`WorkspaceId`、模型权限问题会直接伪装成“worker 执行失败”
- 影响：
  这类问题非常消耗注意力，因为它们出现在“服务已经启动但链路没通”的阶段，看起来像真实功能缺陷，实际根因却是模板、默认值和真实运行口径没有统一。
- 当前结论：
  对新增进程或新增外部服务接入，`.env.example` 不是附属文档，而是主线实现的一部分。环境变量模板、代码读取逻辑和本地默认值必须同一轮收口，否则测试成本会被反复放大。
- 值得复盘的问题：
  对未来所有新增服务接入，是否应该默认要求“代码读取逻辑 + `.env.example` + 最小启动验证”三件事同批完成，而不是先写代码、后补模板。

### 14. 第三方服务的“格式约束”往往比 SDK 调用本身更容易漏掉

- 背景：
  本次 M4 使用 `openai` SDK 接阿里云百炼兼容层，并把向量写入 Qdrant。
- 暴露方式：
  链路打通后，真实联调里先后暴露了两类不是本地单测能自然覆盖到的问题：
  - 百炼 API key 权限没开时，job 在 `embedding` 阶段失败
  - Qdrant point id 不能直接使用自定义字符串，必须是 `uint64` 或 UUID，导致 job 在 `upserting` 阶段失败
- 影响：
  这说明“SDK 能调用”并不等于“链路已经真实可用”。很多高影响问题并不在 TypeScript 类型层，也不在包安装层，而在第三方系统的实际约束里。
- 当前结论：
  第三方集成里，最容易漏掉的不是 API 名字或参数，而是：
  - 权限模型
  - 资源预置状态
  - ID / 维度 / payload 形态约束
  - provider 与兼容层之间的能力差异
    这些内容如果不做一次真实联调，很难靠本地静态代码完全暴露。
- 值得复盘的问题：
  以后涉及向量库、模型服务、对象存储等外部系统时，是否应该把“最小真实联调”视为功能完成定义的一部分，而不是当成可选补充。

### 15. 生成产物、流程工件和实现代码一起推进时，流程本身会成为额外的工程面

- 背景：
  这次 change 不是单纯写代码，还同时推进了 `proposal/specs/design/tasks/execution-contract`、`.spec-superflow.yaml`、`decision-point-audit.md`、Adonis 生成产物和 migration schema。
- 暴露方式：
  实际执行过程中，除了功能代码本身，还需要不断处理这些伴随工作：
  - 恢复 stash、对齐最新主分支
  - 维护 `.spec-superflow.yaml` 的 closing 必需字段
  - 运行 `ssf state rebuild` 和本地 closing 脚本
  - 区分手写改动与 Adonis/Tuyau 自动生成文件
  - 最后再清理提交边界并按主题拆 commit
- 影响：
  这说明中等复杂度以上的 change，真正消耗精力的往往不止业务实现，还包括流程状态和生成产物治理。若团队低估这部分成本，就会频繁感觉“明明功能已经好了，为什么还没法干净收口”。
- 当前结论：
  流程和生成产物并不是附带噪音，而是当前仓库工程现实的一部分。需要把它们当作显式成本管理，而不是假设“代码写完就算完工”。
- 值得复盘的问题：
  对未来类似 change，哪些流程状态和生成产物可以进一步脚本化或自动校验，减少手工同步与末尾收口成本。

### 16. 当前最值得尽快固化的，不是更多提醒，而是几条能直接减少反复排查的默认机制

- 背景：
  这次会话里最消耗时间的部分，并不是某段复杂业务逻辑，而是多次往返于协议、环境变量、第三方权限、Qdrant 约束、流程收口之间。
- 暴露方式：
  这些问题看起来彼此独立，但背后的模式很一致：
  - 代码读取逻辑与 `.env.example` 漂移
  - 跨进程链路先实现、后补协议
  - 第三方服务权限或格式约束直到真实联调才暴露
  - `spec-superflow` 收口状态依赖手工同步
- 影响：
  如果只把这些问题记成“本次踩坑”，下一次同类 change 仍然会重复消耗同样的排查时间。真正缺的不是更多经验，而是默认机制。
- 当前结论：
  当前最值得推广的几条默认机制是：
  1. 跨进程 change 先过固定协议检查表。
  2. 新增运行时配置必须同批完成“代码读取逻辑 + `.env.example` + 最小启动验证”。
  3. 每个外部系统都要做一次最小真实联调，而不是只靠本地测试。
  4. 第三方服务的权限、ID 格式、维度和兼容层能力应在实现时前置确认。
  5. `tasks.md`、`.spec-superflow.yaml`、closing 脚本和生成产物边界要被视为正式工作面。
- 值得复盘的问题：
  团队后续如何把这些“建议动作”变成默认执行习惯，避免每次都靠个人记忆重新补一遍。

## 接口契约与类型系统问题

### 17. 运行时响应风格统一了，不等于数据契约层已经真正统一

- 背景：
  这次围绕接口响应风格继续推进时，项目表面上已经有了统一的 `{ data } / { message } / { message, errors? }` 运行时模式，也已经有 `packages/contracts`。
- 暴露方式：
  真正落到前端 API 层、functional tests 和具体接口返回值时，才发现很多地方仍在本地手写：
  - `{ data: ... }` 外层响应壳
  - 登录结果、协同会话结果等返回 DTO
  - 错误响应 `{ message, errors? }` 的局部类型
- 影响：
  这会制造一种很容易误判的假象：以为“项目已经有 contracts，所以响应契约应该也统一了”。但实际上只是运行时风格统一了，类型契约层还没把高复用的 envelope 基元收上去。
- 当前结论：
  `packages/contracts` 不应该承担所有 controller 级响应分支，但应该提供高复用的共享响应基元，例如：
  - `ApiSuccessResponse<T>`
  - `ApiErrorResponse`
  - `ApiMessageResponse`
    同时，`data` 内部业务字段仍应回到各自领域 DTO，而不是继续散落在调用侧本地定义。
- 值得复盘的问题：
  项目里哪些“已经有统一运行时行为”的地方，其实还缺少一层共享类型契约，导致统一只是停留在代码习惯而不是可复用边界。

### 18. 第三方框架的保留字段语义如果不显式识别，会把局部问题放大成全局类型坍塌

- 背景：
  `POST /api/rag/search` 原先的请求体字段使用 `query`，从业务语义看非常自然，运行时也完全正常。
- 暴露方式：
  一旦进入 Tuyau 的 registry 生成和 `createTuyau({ registry })` 推导链路，这个 `query` 会被当成 transport 元字段，而不是普通 body 字段。结果不是单个接口局部报错，而是整个 `tuyau.api` 退化成 `TransformApiDefinition<unknown>`，连 `spaces` / `documents` 这类无关路由的类型也一起消失。
- 影响：
  这类问题特别隐蔽，因为：
  1. 运行时不报错。
  2. 业务代码看起来没问题。
  3. 真正暴露错误的位置往往是在下游消费方，例如 `tuyau.api.spaces.index({})`。
  4. 如果不顺着生成文件和第三方类型定义往上追，很容易把它误判成“前端类型突然坏了”。
- 当前结论：
  项目接入强类型 transport/代码生成工具后，像 `query`、`params`、`headers`、`cookies` 这类带框架保留语义的字段名，不应该再被当作普通业务字段随意使用。  
  更重要的是，这类问题暴露出一个高影响盲点：第三方框架的类型系统缺陷会直接反向塑造我们自己的接口命名和契约设计。
- 值得复盘的问题：
  今后只要引入会“从代码推导代码”的工具链，是否应该默认建立一份保留字段/保留语义清单，避免局部命名把全局类型系统拖垮。

## 流程与收口问题

### 19. 改动已经完成后再补 spec-superflow change，会暴露出“实现先于工件”的流程债

- 背景：
  这次“响应契约统一 + Tuyau 类型修复”在代码层已经做完、验证也已经跑通之后，才意识到这批改动的体量和影响面已经值得单独补一个 spec-superflow change。
- 暴露方式：
  后续不得不再反向补齐一整套工件：
  - `proposal.md`
  - `design.md`
  - `tasks.md`
  - `spec.md`
  - `execution-contract.md`
  - `.spec-superflow.yaml`
  - `decision-point-audit.md`
    同时还要再次执行 closing 脚本，确保 `rebuild/check/audit` 和新工件保持同步。
- 影响：
  这不会直接导致功能失败，但会显著增加收口成本，也会让变更过程的可追溯性变弱。  
  真正消耗时间的不是“写这些文件本身”，而是要把已经发生过的实现、验证和决策重新映射回 spec-superflow 的状态机语义。
- 当前结论：
  当一次改动开始同时具备下面几个特征时，应该更早判断是否需要独立 change，而不是等实现结束后再补：
  1. 横跨前端、后端、contracts、文档。
  2. 会影响统一规范或公共边界。
  3. 需要专门的验证闭环和提交边界管理。
- 值得复盘的问题：
  团队以后如何更早识别“这已经不是顺手改一下了”，从而在实现前就进入 change 流，而不是把流程成本挪到最后统一偿还。

## 重复劳动与隐性信号

### 20. 本次反复出现的，不是单点 bug，而是“边界缺少默认归属”带来的连锁补丁

- 背景：
  从响应风格、contracts、Tuyau 类型、文档同步，到最后的 spec-superflow 补档，这次会话看起来像处理了很多不同问题。
- 暴露方式：
  仔细看会发现它们反复指向同一种模式：
  - 一个问题先在局部暴露
  - 继续追会发现没有统一归属
  - 然后需要在代码、类型、文档、流程几层同时补洞
- 影响：
  这说明当前仓库真正缺的往往不是“再多写一个判断分支”，而是“这个问题默认应该由哪一层负责”。  
  只要默认归属没立住，后面就会继续出现：
  - 本地补类型
  - 重复改文档
  - 重新解释职责边界
  - 收口时额外补 change 工件
- 当前结论：
  这次最值得记住的隐性关键点不是某个具体修复，而是：  
  **很多重复劳动，都是“默认归属未建立”发出的信号。**  
  一旦发现自己在不同文件、不同会话里反复解释同一件事，往往说明项目不是缺一次修复，而是缺一个正式边界。
- 值得复盘的问题：
  未来如何更早把“重复解释/重复补洞”识别成架构治理信号，而不是继续把它当成正常开发噪音。

## 协同链路与运行时恢复问题

### 21. 本地回环地址不是“理所当然等价”，跨进程内部调用默认值会把环境差异伪装成权限或协议失败

- 背景：
  本次推进 `feat-m4-collaboration-persistence-baseline` 时，`apps/collab` 需要在 `onLoadDocument` / `onStoreDocument` 里回调 `apps/api` 的内部运行态接口。
- 暴露方式：
  浏览器侧一开始看到的是 `permission-denied` 和“本地降级”，表面上很像 Hocuspocus 鉴权失败或 token 有问题。继续往下追后才发现，真正失败的是 `apps/collab -> apps/api` 的内部 HTTP 调用：
  - `apps/collab` 默认打的是 `http://127.0.0.1:3333`
  - 本地这次 `apps/api` 实际监听在 `localhost` / `::1`
  - 最终在 `onLoadDocument` 阶段触发 `ECONNREFUSED`
- 影响：
  这类问题特别容易误导排查方向，因为最终暴露在页面上的不是“连接地址错了”，而是一个被上层框架泛化后的协同失败状态。开发者会先怀疑 token、房间名、provider 配置甚至权限模型，而不是想到 loopback 地址策略。
- 当前结论：
  这不是 Adonis 本身的 bug，而是“本地监听地址”和“内部调用默认地址”没有统一造成的环境基线问题。对多进程本地联调来说，`localhost`、`127.0.0.1`、`::1` 不是可以默认混用的值。  
  当前已把 `apps/collab` 默认 `API_BASE_URL` 收口到 `http://localhost:3333`，但更重要的是：以后不能再把本地地址策略当成无关紧要的默认细节。
- 值得复盘的问题：
  对 DocWeave 这种多进程项目，是否应该统一规定本地服务一律用什么 host 绑定、进程间默认用什么 base URL，并把这件事写进 `.env.example`、启动说明和最小联调检查里，而不是每次靠人肉猜。

### 22. 第三方编辑器的默认初始内容会悄悄污染服务端恢复链路

- 背景：
  本次 M4 第二阶段需要让 `onLoadDocument` 从 `documents.content` 恢复 BlockNote 正文到 Yjs，而不是继续依赖浏览器首帧 seed。
- 暴露方式：
  自动持久化、双客户端同步和 `documents.content` 写回都已经通过后，重连恢复仍然表现异常：数据库里明明已经存了目标正文，但新协同连接恢复出来的却是“从这里开始编写你的文档。”这样的默认占位内容。
- 影响：
  这类问题说明“链路看起来都通了”不等于“真相已经被正确恢复”。如果只验证了写回成功，而没有再验证“新连接能否按数据库真相恢复”，就会误以为恢复链路也没问题。  
  更隐蔽的是，污染源并不在业务代码，而在 BlockNote 服务端 editor 初始化时带入了默认内容，导致恢复 helper 把第三方默认值当成了恢复基线。
- 当前结论：
  服务端恢复 helper 不能只“会调用第三方转换 API”，还必须显式控制第三方宿主的初始化语义。本次已修正为：恢复时用目标 blocks 初始化服务端 BlockNote editor，而不是先起一个默认空白/占位 editor 再做转换。
- 值得复盘的问题：
  以后只要在服务端使用富文本/编辑器类库做 schema 转换，都应该默认追问一次：这个库有没有隐式默认内容、默认节点或初始化副作用，会不会把“看起来像空白”的恢复链路偷偷改写。

## 流程、提交与收口问题

### 23. change 已经 closing，不代表 git 收口会自然干净；流程收口和版本收口是两条不同的工作面

- 背景：
  这次 `feat-m4-collaboration-persistence-baseline` 已经完成了 `tasks.md`、`.spec-superflow.yaml`、`decision-point-audit.md` 和 closing SOP，看起来 change 在流程上已经可以视为收尾。
- 暴露方式：
  真正开始整理 git 时，还是出现了几类额外成本：
  - staged 范围要重新确认，决定是否把额外文档一起纳入
  - 切回 `master` 之后，需要确认工作树是否真的干净
  - 分支删除前，还要额外处理一个 `.adonisjs` 自动生成文件的键顺序漂移
- 影响：
  这说明“流程收口完成”并不自动等于“版本控制层也已经自然收口”。如果团队没有把 git 整理本身当成正式工作面，就会在最后一步继续被生成产物、提交边界和分支状态消耗注意力。
- 当前结论：
  对 DocWeave 这种既有 spec-superflow、又有 Adonis/Tuyau 生成产物的仓库来说，closing 至少分成两层：
  1. 流程层：`tasks/state/audit` 同步完成。
  2. 版本层：工作区干净、提交边界确认、分支生命周期闭合。  
     这两层都做完，才是真的“可以放下这次 change”。
- 值得复盘的问题：
  后续是否要补一个更明确的“closing 后 git 收口清单”，把 staged 检查、生成产物漂移确认、分支合并和删除也纳入默认步骤，而不是每次靠临场经验处理。

## 编辑器 AI 与协同正文边界问题

### 24. AI SDK transport 不会自动继承项目业务 API 的认证钩子

- 背景：
  编辑器 AI 采用 `@blocknote/xl-ai` + Vercel AI SDK 的 `DefaultChatTransport`，普通业务 API 则通过 Tuyau client 统一注入 Bearer token。
- 暴露方式：
  用户已经登录，但 `/api/ai/editor` 仍返回 401。原因是 AI transport 使用独立的原生 `fetch` 链路，不会经过 Tuyau 的 `beforeRequest` hook。
- 影响：
  登录态看起来正常，但 AI 功能单独失效；如果只检查普通 API，会误以为后端认证配置错误。
- 当前结论：
  共享编辑器包不直接依赖 Web 应用认证模块，由 `apps/web` 将动态认证 headers 注入 `DocumentEditorAiProps`，每次 AI 请求读取最新 token。
- 值得复盘的问题：
  以后引入任何独立 transport、WebSocket 或 SDK 内置 fetch，都必须明确它是否复用项目统一 API client；认证边界应作为接入验收项，而不是默认假设。

### 25. 模型能力与 BlockNote 工具调用协议之间存在供应商级约束

- 背景：
  BlockNote AI 官方链路要求 `tool_choice: required`，项目当前通过 DashScope OpenAI-compatible endpoint 调用 `qwen3.6-plus`。
- 暴露方式：
  请求已经带上认证并成功到达后端，但前端只显示 `An error occurred`。直接验证 DashScope 响应后发现，qwen3.6-plus 的思考模式不接受 `tool_choice: required`。
- 影响：
  服务端默认错误脱敏隐藏了真正的供应商错误，模型、工具 schema、流式 runtime 三层问题容易被混为一谈。
- 当前结论：
  编辑器 AI runtime 对该场景注入 `enable_thinking: false`，保留 BlockNote 所需的强制工具调用；该设置不扩散到 RAG 或其他聊天场景。
- 值得复盘的问题：
  AI 接入不能只验证“文本能返回”，还必须用真实模型做一次“工具调用 + 流式 + 供应商特殊模式”的最小实验，并把结果写进模型兼容矩阵。

### 26. BlockNote 协同 AI 的结果合并会暴露无效 selection

- 背景：
  `@blocknote/xl-ai@0.51.4` 的 `acceptChanges()` 会全量替换 ProseMirror 文档，并重建 Yjs 同步插件。
- 暴露方式：
  AI 功能已经能够生成并应用结果，但浏览器出现 `TextSelection endpoint not pointing into a node with inline content` 和 `Position ... is not within a blockContainer node` 警告。
- 影响：
  结果表面上可以应用，但 selection 已经落在旧文档位置，后续继续输入、协同同步或下一次 AI 操作可能出现隐性异常。
- 当前结论：
  协同模式使用自定义 AI menu，在接受结果前把光标收敛到仍存在的 BlockNote 文本块末尾，避免旧位置进入第三方 Yjs merge。当前可用版本没有更高的 `@blocknote/xl-ai` 可直接升级。
- 值得复盘的问题：
  富文本库的“结果应用成功”不能只看正文是否变化，还要验证 selection、undo、协同同步和下一次编辑是否仍然有效。

### 27. 页面生命周期不应驱动协同 provider 重建

- 背景：
  文档页同时持有业务 DTO、React Query 状态和 Yjs provider。保存成功后，React Query 会生成新的 document 对象和 content 数组。
- 暴露方式：
  页面保存后内容短暂闪烁，或重新进入文档时切回初始化内容。排查发现协同 effect 依赖整个 `document` 对象和 `initialContent` 数组，保存后的普通 DTO 更新会销毁并重建协同连接。
- 影响：
  页面元数据更新被错误地当成协同会话变化，造成双编辑器交替挂载、草稿状态覆盖和恢复结果不稳定。
- 当前结论：
  协同连接生命周期只依赖文档 ID、协同 token 和降级状态；初始正文通过 ref 提供给首次 seed，不再因为保存后的内容对象变化而重建 provider。连接未稳定时只显示 loading，不提前挂载 standalone 编辑器。
- 值得复盘的问题：
  页面状态更新与协同会话状态必须分层；React 对象引用变化不能直接作为长连接生命周期信号。

### 28. 最后一个客户端断开必须被视为持久化事件，而不是普通清理事件

- 背景：
  Hocuspocus 默认通过 debounce 调用 `onStoreDocument`，页面路由切换时浏览器会销毁协同 provider。用户期望未点击业务保存按钮的正文草稿在切屏后仍然存在。
- 暴露方式：
  页面内保存时内容不再消失，但切换路由后重新打开文档又回到初始化内容；未保存的输入也随之消失。数据库检查确认正文仍是初始化内容，说明最后一段 Yjs 更新没有可靠落库。
- 影响：
  用户会把问题理解成“编辑器丢稿”，而实际是页面生命周期、WebSocket 断开、Hocuspocus debounce 和 API 持久化之间没有形成明确提交边界。单纯依赖定时保存无法覆盖快速切屏、刷新和 provider 销毁场景。
- 当前结论：
  保留正常变更的 debounce 持久化，同时在最后一个客户端断开时显式把当前 Y.Doc 回写 API；空的协同缓存则丢弃并从持久化正文恢复。该断开回写是草稿可靠性边界，不等同于业务层稳定快照。
- 值得复盘的问题：
  协同系统必须明确至少三个时刻：编辑变更、会话断开、业务保存/快照。最后一个客户端断开应当是强制 flush 点，不能把“内存里暂时有内容”误认为“内容已经持久化”。这类行为应纳入路由切换、刷新和快速输入后的自动化回归测试。

### 29. “前后端都要用”的东西，不等于都应该塞进 `contracts`

- 背景：
  本次处理 API 英文错误提示时，最初很容易把问题理解成“前后端各自定义了一份错误文案，所以把常量统一放到一个地方就行”。
- 暴露方式：
  真正落到实现时才发现，这里其实混着两类完全不同的东西：
  - `ApiErrorCode` 这类跨边界稳定语义
  - `错误码 -> 默认中文文案` 这类前后端都要运行时读取的共享常量
    如果不拆开，下一步就会自然滑向“既然前后端都用，那就都放 `packages/contracts`”。
- 影响：
  这会让 `contracts` 从“稳定数据契约层”逐渐膨胀成“共享实现杂物层”。短期看方便，长期会让契约层混入产品文案、运行时目录、展示策略，后续维护时越来越难判断什么属于跨边界承诺，什么只是共享实现细节。
- 当前结论：
  这次最终明确了边界：
  - `packages/contracts` 只承载稳定错误语义和 response shape
  - `packages/shared` 承载前后端共同需要的运行时常量目录
    以后再遇到“前后端都要用的值”，第一反应不应该是“塞进 contracts”，而应该先判断它到底是契约，还是共享运行时实现。
- 值得复盘的问题：
  项目里凡是“共享”诉求，都值得先追问一次：共享的是稳定语义，还是共享的是实现常量。如果这一步不做，架构层很快就会把“共享”误写成“混层”。

### 30. `--ff-only` 合并的关键不只是命令本身，还包括不要把 Git 关键步骤并行化

- 背景：
  本次在提交、推送 change 分支后，需要切回基线分支，用 `--ff-only` 合并，确保不产生额外 merge commit。
- 暴露方式：
  实操时如果把 `git pull --ff-only` 和 `git merge --ff-only <branch>` 并行触发，就会出现一个很迷惑的现象：
  - `merge` 已经成功把当前分支快进到了目标提交
  - `pull` 因为工作树在执行期间发生变化而报“local changes would be overwritten”之类的错误
    表面看像 Git 状态异常，实际是两个本该串行的关键步骤被并行化了。
- 影响：
  这类问题很容易制造错误判断。开发者会先怀疑工作区脏了、stash 没处理干净、甚至以为 merge 失败了，实际上版本结果可能已经是对的，只是命令执行时序把反馈信息搅乱了。
- 当前结论：
  Git 的关键状态切换步骤，尤其是 `checkout`、`pull --ff-only`、`merge --ff-only`、`push`，应该明确串行执行。并行适合读操作，不适合会改变 HEAD、索引或工作树的命令。
- 值得复盘的问题：
  在多工具、多自动化环境里，哪些命令虽然看起来独立，实际上共享同一份 Git 状态机，必须被当成串行动作处理。

### 31. M6 证明 RAG 的难点不在模型调用，而在端到端真相保持

- 背景：
  M6 同时落地了块级索引、`space_members` 权限真相、active indexed version 过滤、HTTP 流式问答、Citation 回跳、编辑器高亮和知识准备入口。
- 暴露方式：
  实现中反复出现的并非单个 SDK 调用失败，而是跨层身份与版本漂移：向量 payload 不能替代当前权限、已保存/已快照/已索引不是同一状态、模型会简写 Citation ID、BlockNote 会重绘 DOM、stream 开始后不能再使用普通 HTTP error，以及 closing 需要同步主规范和 Git 状态。
- 影响：
  若只按“接通检索 + 调模型”评估完成度，系统会得到看似可用但不可审计、可能越权、引用不可回跳或用户无法判断知识是否最新的 RAG 功能。
- 当前结论：
  M6 的正确交付单位是“可审计闭环”：member-derived scope -> active snapshot point -> query embedding -> active-version retrieval -> stream event -> Citation identity -> editor evidence -> 用户可理解的快照/索引状态。其中 RAG 采用 BlockNote block-level chunking、Cosine Qdrant 向量检索和 snapshot-driven 异步增量索引；AI runtime 通过 AI SDK + 阿里云适配收口 chat/embedding，`space_members` 以唯一 `(space_id, user_id)` 关系承载 owner/viewer 事实，旧空间必须经人工确认映射迁移，MCP 未有身份映射时 fail closed。详细复盘见 [M6 RAG、AI 与知识库闭环复盘](./M6%20RAG、AI%20与知识库闭环复盘.md)。
- 值得复盘的问题：
  下一阶段若增加 PostgreSQL FTS 兜底、rerank、Agent 或多轮会话，如何保持同样的权限先裁剪、版本可追溯、Citation 可验证和终止语义，而不是在新入口重新复制一套隐性规则。尤其要避免把 FTS 与向量分数直接相加，或用实时正文结果伪装成 stable-snapshot RAG Citation。

## 后续追加建议

后面继续开发时，可以优先记录下面几类问题：

1. 哪些问题一开始以为是功能 bug，最后发现是边界设计 bug。
2. 哪些问题一开始以为是代码问题，最后发现是流程或环境问题。
3. 哪些问题反复出现，说明需要沉淀成规范、脚本或检查项。
4. 哪些问题你后来已经能系统回答，适合扩展成一篇独立文章。

## 技术选型自问清单

这部分不是记录某一次具体问题，而是给后续选型时反复使用的判断模板。

### 1. 这个技术解决的是主问题，还是只是在优化次问题

- 要问自己：
  - 如果不用它，当前最痛的问题还在不在？
  - 它是在解决核心矛盾，还是只是在优化已经能工作的边角？
- 典型误区：
  为了“更先进”引入方案，但它解决的其实不是当前阶段最关键的问题。

### 2. 这个技术是在减少复杂度，还是只是在转移复杂度

- 要问自己：
  - 我是少写了复杂度，还是把复杂度转移到别的层？
  - 它会不会让我在调试、部署、培训、协作时付出更高成本？
- 典型误区：
  以为抽象层越多越专业，结果只是把复杂度藏起来，最后在 3am 才看见。

### 3. 这个技术和项目现有边界是否天然契合

- 要问自己：
  - 它和当前的分层、数据流、运行时模型是不是顺的？
  - 它会不会逼我改很多原本已经合理的边界？
- 典型误区：
  某个技术单独看很好，但和当前系统组合后，边界会开始拧巴。

### 4. 不选它的代价是什么，选它的代价又是什么

- 要问自己：
  - 不选它，我失去什么？
  - 选它，我必须长期承担什么？
- 典型误区：
  只看“选它有什么好处”，不看“以后要为它付出什么维护成本”。

### 5. 这个方案的替换成本高不高

- 要问自己：
  - 它是可替换件，还是会深度嵌进整个系统？
  - 一旦选错，未来改掉的代价大不大？
- 典型误区：
  对高耦合技术做了轻率决定，后面发现不是“换一个库”，而是“重切一层架构”。

### 6. 这是因为我真的判断过，还是因为我只熟悉它

- 要问自己：
  - 我选它是因为它更合适，还是因为我更熟？
  - 我排除其他方案时，有没有说得清理由？
- 典型误区：
  把“熟悉度”误当成“合理性”。

### 7. 这是当前阶段应该做的决定，还是过早优化

- 要问自己：
  - 这个决定是当前就必须做，还是以后有痛点再做也来得及？
  - 现在不定它，会不会阻塞主线？
- 典型误区：
  在还没被真实问题逼到的时候，先把未来十种可能性都设计好了。

### 8. 这个方案会不会制造新的默认假设

- 要问自己：
  - 它会不会要求团队记住新的隐含前提？
  - 如果这些前提没人写清楚，未来会不会反复踩坑？
- 典型误区：
  技术方案本身没错，但它带来的默认假设没人显式记录，最后变成环境和流程问题。

### 9. 这个决定以后怎么向别人解释

- 要问自己：
  - 如果我要在面试、复盘、设计评审里解释这次选择，我能不能说清：
    - 为什么选它
    - 为什么不选别的
    - 当时的约束是什么
    - 后来验证结果如何
- 典型误区：
  做决定时没有留下判断过程，最后只能说“当时就这么做了”。

### 10. 选型之后，最早应该验证什么

- 要问自己：
  - 这个方案最值得尽早验证的风险点是什么？
  - 我应该用什么最小实验确认它不是纸上谈兵？
- 典型误区：
  方案定了就一路往下写，直到很后面才发现最关键的前提根本不成立。
