## 评估结果：官方资料对照后的完整结论

我查阅了 BlockNote 官方文档、Hocuspocus 官方 Hooks/React 绑定、和 AdonisJS 当前认证配置。以下是逐项对照：

### 缺口 1 — Token 签名机制 🔧 **可以直接定**

**官方资料支撑：**

- **Hocuspocus Server** 的 `onAuthenticate` hook 接收 `{ token, documentName, connection }`，返回的 `context` 会透传到下游所有 hook（`onLoadDocument`、`onChange` 等）。抛异常即拒绝连接。
- **Hocuspocus Provider (React)** 的 `<HocuspocusRoom>` 组件接受 `token`（string 或 async resolver），提供 `onAuthenticationFailed` 回调。
- **AdonisJS** 当前使用 `tokensGuard`（`DbAccessTokensProvider`）+ `sessionGuard`，**没有内置 JWT**，需要额外引入 `jsonwebtoken`。
  参考文档：https://docs.adonisjs.com/guides/auth/custom-auth-guard

**结论**：可以直接签。方案：

1. `apps/api` 用 `jsonwebtoken` 签发短期 JWT（HS256），payload 按文档 16 的 `CollaborationTokenPayload`
2. 签名密钥存在环境变量 `COLLAB_SECRET`，`apps/api` 和 `apps/collab` 共享
3. `apps/collab` 的 `onAuthenticate` hook 用同一密钥验证 JWT，解析 payload 放入 `context`
4. 注意：文档 16 的 `workspaceId: number` / `documentId: number` 与当前 DB（UUID string）不符，签发时按实际类型改 `string`

### 缺口 2 — Yjs + BlockNote 绑定 ✅ **官方有明确 API**

BlockNote 官方协同文档确认：

```tsx
import { withCollaboration } from '@blocknote/core/yjs'
import { HocuspocusProvider } from '@hocuspocus/provider'

const doc = new Y.Doc()
const provider = new HocuspocusProvider({
  url: 'ws://localhost:1234',
  name: roomName,
  token: [redacted],
  document: doc,
})

const editor = useCreateBlockNote(
  withCollaboration({
    collaboration: {
      provider,
      fragment: doc.getXmlFragment('document-store'),
      user: { name: user.name, color: user.color },
    },
  })
)
```

`@hocuspocus/provider-react` 还提供了更 React 原生的方式——`<HocuspocusProviderWebsocketComponent>` + `<HocuspocusRoom>` + `useHocuspocusProvider()` hooks（带 `useHocuspocusConnectionStatus`、`useHocuspocusAwareness` 等）。

**这条链路可以写出完整代码，无需再探索。**

### 缺口 3 — apps/collab 工程化配置 ✅ **有标准模式**

Hocuspocus Server 4.x 的配置模式：

```ts
import { Server } from '@hocuspocus/server'

const server = Server.configure({
  port: 1234,
  async onAuthenticate({ token, documentName }) {
    /* JWT verify */
  },
  async onLoadDocument({ documentName }) {
    /* load from DB */
  },
  async onStoreDocument({ documentName, document }) {
    /* persist */
  },
})
server.listen()
```

依赖清单：`@hocuspocus/server`、`@hocuspocus/provider`、`yjs`、`jsonwebtoken`。TypeScript 配置可参考 `apps/api`。

### 缺口 4 — 共享协议代码落点

当前存在 `packages/contracts`（已有 document/space/auth 类型）和 `packages/collaboration`（空壳）。

**选择**：扩展现有 `packages/contracts`，新增 `packages/contracts/src/collaboration.ts`。`packages/collaboration` 可留给更厚的前端协同工具函数（如 provider 工厂、awareness 转换），而非纯类型定义。

### 缺口 5 — workspaceId / documentId 类型 ✅ **确认即可修正**

文档 16 写的是 `number`，但当前 DB 中 `spaces.id` 和 `documents.id` 都是 `string`（UUID）。修正为 `string` 即可，不影响整体设计。

## 方案C：

### 缺口 1 — Token 签名机制

当前项目已落地，且方案很轻量：

维度 实现
算法 HMAC-SHA256，自定义格式 base64url(payload).base64url(signature)，非 JWT
密钥共享 两端（apps/api + apps/collab）共读同一个环境变量 COLLAB_SECRET，各自从 .env 加载
防时序攻击 用 node:crypto 的 timingSafeEqual 做签名对比
载荷校验 版本号（version: 1）防混用、expiresAt 防过期、roomName 防跨房间复用
代码位置 packages/collaboration/src/server-token.ts — signCollaborationToken / verifyCollaborationToken
导出入口 @ai-doc/collaboration/server-token（package.json exports 子路径），API 端和 collab 端各自 import
关键设计决策：不依赖 JWT 库，直接用 crypto.createHmac + base64url 做的自定义格式。原因注释写了"方便后续可平滑替换成 JWT"但当前不需要额外依赖。

签名 → 验证链路：

API 端: signCollaborationToken(payload, COLLAB_SECRET) → token string → 返回给前端
↓
前端: 拿 token 传给 HocuspocusProvider → WebSocket 连接 collab 服务
↓
Collab 端: onAuthenticate 钩子 → verifyCollaborationToken(token, COLLAB_SECRET) → 校验通过才允许连接 + 额外校验 roomName === documentName（防止 token 跨房间复用）+ 映射 canEdit → connectionConfig.readOnly（只读用户连上也不能提交更新）

## 最终推荐方案

综合当前仓库现状、官方文档支持度、实现复杂度与第一阶段目标，M3 最终推荐采用：

**以方案 B 为主线，吸收方案 C 的“少依赖、最小闭环”做法，不采用方案 A 的 JWT 依赖方案。**

换句话说：

1. 协议结构、工程拆分、前端接线方式，采用方案 B。
2. Token 实现风格，采用方案 C 的自定义 HMAC 签名短期 token。
3. 不引入 `jsonwebtoken`，不把 JWT 标准化能力提前带进第一阶段。

### 为什么不选方案 A

方案 A 的 `JWT HS256 + jsonwebtoken` 可以工作，但对当前阶段来说偏重：

1. 当前 M3 只需要一张可离线验签、短期有效、绑定房间的协同准入票据。
2. `apps/collab` 需要的是“本地验签后准入”，而不是完整 JWT 生态能力。
3. 为这一阶段额外引入 `jsonwebtoken`，收益不高，后续维护面反而更大。

### 为什么最终采用 B + C

这个组合最符合 DocWeave 当前状态：

1. BlockNote 官方已经提供 `withCollaboration(...)` 路线，Hocuspocus 也有标准 provider / server hooks，可以直接实现。
2. 当前 `apps/collab` 仍是空壳，适合先做最小 Node + TypeScript 服务，而不是一开始就设计过厚抽象。
3. 当前 `packages/contracts` 已经是共享 DTO 的既有落点，协同协议继续放这里最顺手。
4. 当前第一阶段重点是“两个浏览器协同编辑 + presence + 受控准入”，不是 token 标准化平台建设。

## 定稿决策

### 1. Token 方案

采用短期自定义 HMAC token，不复用 Adonis access token，不采用 JWT。

格式：

```text
token = base64url(payloadJson) + "." + base64url(hmacSha256(payloadJson, COLLAB_SECRET))
```

其中：

1. `payloadJson = JSON.stringify(payload)`
2. `COLLAB_SECRET` 由 `apps/api` 与 `apps/collab` 共用
3. `signCollaborationToken(payload, COLLAB_SECRET)` 放在 `apps/api`
4. `verifyCollaborationToken(token, COLLAB_SECRET)` 放在 `apps/collab`

最小 payload 结构：

```ts
interface CollaborationTokenPayload {
  version: 1
  workspaceId: string
  documentId: string
  roomName: string
  capabilities: {
    canRead: boolean
    canEdit: boolean
    canComment: boolean
    canUseEditorAI: boolean
  }
  user: {
    id: number
    name: string
    avatarUrl: string | null
    color: string
    role: 'owner' | 'editor' | 'commenter' | 'viewer'
  }
  issuedAt: string
  expiresAt: string
}
```

校验要求：

1. 验签必须使用 `timingSafeEqual` 或等价安全比较。
2. 必须校验 `expiresAt` 是否过期。
3. 必须校验 `roomName === documentName`，防止 token 跨房间复用。
4. 成功校验后的 payload 要挂入 `apps/collab` 的连接 `context`，供后续 hooks 复用。

### 2. 前端集成方案

文档页协同初始化顺序固定为：

```text
metadata
  -> collaboration token
  -> Y.Doc / provider
  -> editor
  -> awareness / presence
```

BlockNote 集成采用官方 `withCollaboration(...)` 路线：

```ts
const doc = new Y.Doc()
const provider = new HocuspocusProvider({
  url,
  name: roomName,
  token,
  document: doc,
})

const editor = useCreateBlockNote(
  withCollaboration({
    collaboration: {
      provider,
      fragment: doc.getXmlFragment('document-store'),
      user,
    },
  })
)
```

约束如下：

1. `withCollaboration` 是主路径，不自造 editor binding。
2. `initialContent` 不再与 Yjs 长期并存为双正文真相。
3. REST 文档内容仅用于空房间初始化兜底，或协同启动失败后的本地 fallback。
4. 可保留 `2.5s` 超时降级到本地快照，但它只应作为失败兜底，不应成为常态双写路径。

### 3. apps/collab 最小工程化方案

第一阶段只实现最小可运行协同服务，不提前做持久化系统。

推荐最小文件：

```text
apps/collab/
├── package.json
├── tsconfig.json
└── src/
    ├── index.ts
    ├── config.ts
    ├── server.ts
    ├── auth.ts
    └── rooms.ts
```

服务端主结构：

```ts
Server.configure({
  port,
  async onAuthenticate(data) {},
  async onLoadDocument(data) {},
  async onStoreDocument(data) {},
})
```

阶段要求：

1. `onAuthenticate`：验签、校验房间名、生成连接上下文。
2. `onLoadDocument`：先返回内存 `Y.Doc` 即可。
3. `onStoreDocument`：预留钩子，但 M3 阶段不做真实持久化。
4. `onConnect` / `onDisconnect` / `onAwarenessUpdate` 可按需要补日志和 presence 同步观察。

### 4. 共享协议落点

协同协议统一放到：

```text
packages/contracts/src/collaboration.ts
```

这里承载：

1. `CollaborationTokenPayload`
2. `CollaborationCapabilities`
3. `CollaborationAwarenessState`
4. `CollaborationPresenceEntry`
5. `CollaborationConnectionStatus`
6. `buildDocumentRoomName(workspaceId, documentId)`
7. `parseDocumentRoomName(roomName)`

约束：

1. `packages/contracts` 只放协议与类型。
2. `sign/verify` 不放到 `contracts`。
3. `packages/collaboration` 当前阶段可以继续留空，后续只有在出现共享运行时代码重复时再引入。

### 5. ID 类型统一

M3 协同协议中的：

1. `workspaceId`
2. `documentId`

统一使用 `string`，与当前数据库和 DTO 现状保持一致。

`user.id` 继续保持 `number`，无需强行统一成字符串。

### 6. API 准入要求

`POST /api/collaboration/token` 必须挂认证，不允许继续公开暴露。

当前要求：

1. 先读当前会话。
2. 校验目标文档是否可访问。
3. 计算 `capabilities`。
4. 签发短期 collab token。

`apps/collab` 只消费这个结果，不重新成为权限真相源。

## 最终一句话结论

DocWeave 的 M3 协同最小闭环，最终采用：

**`packages/contracts` 统一协议，`apps/api` 签短期 HMAC token，`apps/collab` 本地验签并控制房间准入，`apps/web` 用 BlockNote + Yjs + Hocuspocus 走单一协同正文主链路。**
