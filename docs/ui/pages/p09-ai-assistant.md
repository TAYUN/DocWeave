# P09 AI 助手页

## 路由

`/ai`

## 用途

为用户提供统一的 AI 能力入口与状态说明，帮助用户理解“哪些 AI 能力已经可用、应该去哪里使用、当前为什么不可用”。

这个页面不是知识库问答主场景，也不是文档正文编辑主场景。它的职责是：

1. 说明当前 AI 能力范围
2. 给出进入可执行场景的明确路径
3. 承载最小的 AI 状态反馈与最近动作入口

## 当前阶段定位

- 当前在页面总览中属于 `△ 后续阶段 / M5 编辑器 AI`
- 不建议把 `/ai` 作为 M5 的主闭环页面
- M5 的真实主闭环仍应优先落在 `/documents/:documentId` 的编辑器上下文里
- 当前代码库中没有对应已落地页面，本文件不代表当前 UI 已存在

因此，`/ai` 在当前规格体系里的合理定位是：

- M2 - M4：占位页，只说明能力未启用与未来入口边界
- M5：可作为“AI 能力入口页 / 最近动作页 / 选择文档开始页”，但不承担真正的改写执行面板
- M7：可补充跳转到 `/chat` 的入口说明，但仍不吞并 RAG 问答职责

## 主任务

1. 用户查看当前 AI 能力是否可用
2. 用户理解“AI 编辑”与“RAG 问答”的区别
3. 用户快速进入一个可执行 AI 编辑的文档
4. 用户查看最近 AI 动作或最近使用的文档入口（M5 之后才成立）

## 内容分区

| 区块             | 内容                                                | 说明                                |
| ---------------- | --------------------------------------------------- | ----------------------------------- |
| 页面引导区       | 标题 + 一句用途说明                                 | 只说明功能边界，不做营销式 Hero     |
| 能力状态面板     | AI 编辑 / RAG 问答 / 最近动作 的可用状态            | 用于回答“现在能不能用”              |
| 边界说明面板     | `/ai`、`/chat`、`/documents/:documentId` 的职责区别 | 必须显式存在，避免入口混乱          |
| 从文档开始面板   | 最近文档入口 + 前往工作台按钮                       | 主区，M5 真正可闭环的入口仍是文档页 |
| 最近 AI 动作面板 | 最近使用过的 AI 动作记录                            | 次区，仅在 M5 后端有记录能力时成立  |

## 设计决策

- `/ai` 不做“万能聊天页”，避免与 `/chat` 重叠
- `/ai` 不直接承载正文大编辑区，避免与 `/documents/:documentId` 重叠
- 当前阶段优先把它定义为入口页 + 状态页 + 边界说明页
- 页面主操作不是“在这里直接提问”，而是“进入可执行的文档场景”
- 没有能力支撑时，应明确写成未启用或占位，不伪装成可用功能

## 状态规格

| 状态                   | 表现                                                                                                                                 |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| 默认                   | 展示能力状态、边界说明、文档入口                                                                                                     |
| 加载中                 | 页面骨架或局部 Loader，优先保护主结构，不做整页品牌化 loading                                                                        |
| 空态：无最近文档       | 显示“还没有最近文档” + “前往工作台选择文档”按钮                                                                                      |
| 空态：无 AI 动作记录   | 显示“还没有 AI 使用记录” + 说明“请先在文档页使用 AI 编辑”                                                                            |
| disabled               | 某项能力当前不可操作，但控件和入口保持可见，并明确说明原因，例如“需先选择文档”“当前阶段未启用”“文档为只读”                           |
| restricted / 无权限    | 用户或工作区无权使用 AI 能力时，显示明确说明，例如“当前工作区未开通 AI 能力”或“你没有权限使用 AI 编辑”，并提供返回工作台或设置页路径 |
| 能力未启用（阶段未到） | 显示状态 Badge + 说明“当前版本仅保留入口，不可执行真实 AI 操作”                                                                      |
| 数据加载失败           | 普通错误卡片 / Alert，说明是“能力状态加载失败”或“最近记录加载失败”，并给出重试                                                       |
| 文档入口不可用         | 最近文档已删除或无权访问时，显示“文档不可用”，提供返回工作台路径，不静默消失                                                         |

## 与其他页面的边界

### 与 `/chat` 的边界

- `/chat` 是 RAG 问答页
- 它的主任务是：提问、查看答案、查看引用来源、回到原文
- `/ai` 不负责知识库问答结果展示，不负责 citation 列表，也不承载问答历史主链路
- 在 M7 之后，`/ai` 最多只提供“前往 `/chat`”的能力入口说明

### 与 `/documents/:documentId` 的边界

- `/documents/:documentId` 是正文编辑主场景
- 标题、摘要、正文编辑、保存、未保存离开提示，都应留在文档页
- M5 的改写、扩写、缩写、翻译、总结动作，真正执行位置应优先在文档页编辑器上下文中
- `/ai` 不提供大块正文编辑区域，不在这里直接替代文档页完成内容改写

## 导航关系

- 从哪来
  - AppHeader 或侧边导航中的 AI 入口
  - 文档页中的“查看 AI 能力说明”辅助入口
- 到哪去
  - 点击最近文档 / 推荐文档 → `/documents/:documentId`
  - 点击知识问答入口（M7 后）→ `/chat`
  - 点击返回工作台 → `/`
- 错误回退
  - 文档不可访问 → 返回工作台
  - AI 能力受限 → 返回工作台或设置页

## 实现约束

- 使用普通工作台结构，不切成独立品牌页
- 遵循 Mantine 默认主题与 `default-first` 原则，不额外制造 AI 专属视觉体系
- 页面级 CSS 只负责结构、留白和分区，不做大渐变、重阴影、营销化 Hero
- M5 前：
  - `/ai` 只能作为占位或能力说明页
  - 不应假装已经具备真实 AI 编辑闭环
- M5：
  - 可以闭环的能力是“进入文档页并在编辑器上下文中使用 AI”
  - `/ai` 本页最多承载入口、状态、最近动作，不应承担 BlockNote AI 主交互面板
- M7：
  - 可增加到 `/chat` 的清晰跳转
  - 但不把 `/chat` 的答案区、来源区复制进 `/ai`
- 数据锚点：
  - M5 前，能力状态与最近动作可先使用本地占位数据或静态配置驱动
  - M5 后，再接入真实 AI 能力状态、最近动作记录和文档入口数据
