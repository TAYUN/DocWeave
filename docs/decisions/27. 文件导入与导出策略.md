# 27. 文件导入与导出策略

阅读说明：

1. 本文档负责定义 DocWeave 后续阶段的文档导入、导出与格式转换策略。
2. 本文档聚焦“哪些格式优先支持、谁负责转换、什么时候进入 worker”。
3. 文件存储边界以 [15. 文件存储与上传策略](./15.%20文件存储与上传策略.md) 为准。
4. BlockNote 服务端处理基线以 [22. BlockNote 服务端能力接入设计](../architecture/22.%20BlockNote%20服务端能力接入设计.md) 为准。

## 1. 文档目的

本文档用于明确 DocWeave 在导入导出链路上的优先级、系统边界和后续演进方式，避免过早把格式转换做成一套巨大流水线，或者让浏览器承担不稳定的重型文档转换。

## 2. 核心结论

DocWeave 在文件导入与导出上的正式路线如下：

1. 导入与导出都属于服务端能力，不由前端直接承担重型转换。
2. 文档结构处理优先复用 `@blocknote/server-util`，避免过早自写基础转换链路。
3. 第一版优先支持与 BlockNote / HTML / Markdown 更接近的格式。
4. 重型转换、批量任务和后处理应进入 `apps/worker`，不要塞进普通 API 请求。
5. 导出产物统一走对象存储与文件元数据模型，不把临时磁盘路径当成长期真相。

## 3. 为什么这样设计

采用这条路线的原因如下：

1. BlockNote 天然适合围绕结构化 JSON、HTML、Markdown 等格式构建转换链路。
2. 浏览器端做复杂导出转换稳定性差，且会放大设备差异和性能问题。
3. 导出通常与稳定快照、发布版本和后续分享能力强相关，更适合服务端收口。
4. 文档转换一旦做复杂，极易演化成异步任务，不应一开始就绑死在同步 API 中。

## 4. 导入与导出的边界

### 4.1 `apps/web`

负责：

1. 发起导入或导出请求。
2. 展示任务状态、下载入口和失败提示。
3. 在表单层做最小文件类型与大小提示。

不负责：

1. 承担真实格式转换。
2. 直接访问对象存储临时文件。

### 4.2 `apps/api`

负责：

1. 校验导入导出权限。
2. 创建导入导出任务。
3. 管理文档、快照与导出文件元数据关系。
4. 返回任务状态与下载入口。

### 4.3 `apps/worker`

负责：

1. 执行重型导入解析。
2. 执行导出转换与文件生成。
3. 处理失败重试与中间产物清理。

## 5. 第一版优先支持的能力

第一版建议优先支持：

1. 基于稳定快照导出 `Markdown`
2. 基于稳定快照导出 `HTML`
3. BlockNote JSON 的导入 / 恢复

如果阶段内需要再多做一步，可按下面顺序扩展：

1. `PDF` 导出
2. Markdown 导入
3. HTML 导入

## 6. 导出内容真相

导出不应直接读取浏览器临时态，也不应默认读取协同运行态。

推荐原则如下：

1. 导出优先基于稳定快照。
2. 若未来支持“发布版本导出”，则优先允许用户显式选择发布版本。
3. 导出产物应保留版本关联，便于后续审计、回溯和重新生成。

## 7. 导入内容真相

导入本质上是“将外部内容转成 DocWeave 可管理的结构化文档”。

建议原则如下：

1. 外部文件先进入受控上传链路。
2. 转换成功后生成新的文档初始内容或新的稳定快照。
3. 原始导入文件可作为附件或原始来源保留。
4. 不把用户上传的原始格式直接当作编辑器运行时真相。

## 8. 任务化原则

以下场景建议直接任务化，不走同步 API：

1. 大文件导入
2. 复杂格式转换
3. `PDF` 导出
4. 批量导出
5. 需要图片、附件、字体或样式预处理的导出

第一版即使先支持同步轻量导出，也应保留“未来迁到 worker”不破接口的语义。

## 9. 与对象存储的关系

导入导出产物建议统一走文件元数据模型：

1. 原始上传文件进入对象存储。
2. 中间产物可按策略临时保留或清理。
3. 导出结果作为独立文件记录写入数据库。
4. 下载仍通过 `apps/api` 权限校验后签发短期 URL。

## 10. 不建议的做法

需要避免以下问题：

1. 在浏览器端做复杂导出主流程。
2. 直接基于协同实时态导出最终文件。
3. 将导出文件散落写到本地磁盘路径并长期引用。
4. 一开始就承诺支持过多 Office / PDF 双向完美转换。

## 11. 第一阶段与后续阶段的关系

当前阶段最重要的不是“支持所有格式”，而是先建立以下抽象：

1. 稳定快照是导出与格式转换的基础输入。
2. `apps/api` 管权限与任务入口。
3. `apps/worker` 管重型转换。
4. 文件结果统一纳入对象存储与元数据模型。

## 12. 当前结论

DocWeave 的导入导出策略应先围绕“稳定快照 + 服务端转换 + 任务化演进”建立，而不是围绕“浏览器即时导出一切格式”建立。

只要这个方向先固定，后续无论补 PDF、Markdown、HTML 还是更复杂的 Office 场景，都不会把系统边界做乱。
