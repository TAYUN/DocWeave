# 21. 聊天与 Copilot 交互层策略

阅读说明：

1. 本文档负责定义聊天页与文档 Copilot 的前端交互层取舍。
2. 本文档聚焦“第一阶段聊天 / Copilot 用什么 UI 形态，后续怎么演进”。
3. 底层模型接入仍以 [20. AI SDK 与模型接入策略](./20.%20AI%20SDK%20与模型接入策略.md) 为准。
4. 编辑器内 AI 能力边界以 [22. BlockNote 服务端能力接入设计](../architecture/22.%20BlockNote%20服务端能力接入设计.md) 和 [21. M5 编辑器 AI 实施文档](../planning/21.%20M5%20编辑器%20AI%20实施文档.md) 为准。

## 1. 文档目的

本文档用于明确 DocWeave 在“聊天 / Copilot 交互层”上的当前决策，避免团队在第一阶段就把 AI 交互壳做得过重，或者反过来在聊天场景里重复造轮子。

## 2. 核心结论

当前项目正式采用如下路线：

1. 底层 AI 调用与流式协议统一基于 `Vercel AI SDK`。
2. 第一阶段聊天页与文档 Copilot 优先使用项目自定义轻量 UI。
3. `assistant-ui` 暂不作为第一阶段必选依赖，但保留为后续增强选项。
4. 编辑器内 AI 不纳入本决策的主范围，它优先按 `@blocknote/xl-ai` 路线评估与接入。

## 3. 为什么现在不直接上 `assistant-ui`

第一阶段暂不直接引入 `assistant-ui` 的原因如下：

1. 当前更重要的是先打通业务接口、流式返回、权限校验和来源展示闭环。
2. 聊天页与 Copilot 的实际产品交互还在探索期，过早绑定完整 UI 框架会抬高调整成本。
3. 第一阶段页面数量有限，自定义轻量壳足以支撑基础消息列表、输入区、停止生成和来源展示。
4. 编辑器内 AI 已有 `@blocknote/xl-ai` 方向，聊天层没有必要同时再引入过重依赖扩大集成面。

## 4. 为什么仍然保留 `assistant-ui` 作为后续选项

虽然第一阶段先不上，但 `assistant-ui` 仍值得保留为后续演进方向，原因如下：

1. 当聊天 / Copilot 进入稳定产品形态后，成熟消息 UI 组件会减少重复建设。
2. 后续如果需要历史会话、附件、建议问题、更多工具交互，`assistant-ui` 的收益会明显增加。
3. 只要当前先把底层 runtime、消息协议和来源渲染边界设计清楚，后续替换前端壳的成本可控。

## 5. 第一阶段推荐形态

第一阶段推荐如下分工：

1. 聊天页
   - 自定义轻量 UI
   - 聚焦消息列表、输入框、流式状态、来源展示
2. 文档 Copilot
   - 自定义轻量侧边面板或抽屉
   - 聚焦当前文档上下文问答、总结、解释与引用
3. 编辑器内 AI
   - 优先评估 `@blocknote/xl-ai`
   - 不与聊天 / Copilot 共用重聊天式 UI 壳

## 6. 需要先稳定的底层边界

无论 UI 壳是否变化，第一阶段都应先稳定以下边界：

1. `apps/api` 的 AI / RAG 流式接口契约。
2. 前端消息结构与来源结构。
3. 流式中断、错误和重试状态。
4. 文档上下文与权限校验链路。

只有这些边界先稳定，后续才有条件平滑替换 UI 壳。

## 7. 不建议的做法

需要避免以下做法：

1. 第一阶段为了“看起来完整”而过早引入重交互框架。
2. 聊天页、Copilot、编辑器 AI 三套界面各自发明完全不同的消息协议。
3. 把前端 UI 组件选择误当成底层 AI runtime 决策。

## 8. 后续触发升级的条件

若后续出现以下情况，可再正式评估接入 `assistant-ui`：

1. 聊天页进入稳定产品形态。
2. 文档 Copilot 需要更成熟的会话历史与附件能力。
3. 多个 AI 入口开始共享较多消息交互组件。
4. 自定义 UI 的维护成本明显高于引入成熟交互层。

## 9. 当前结论

DocWeave 当前在聊天 / Copilot 交互层上的合理路线不是“永远自定义”，也不是“现在立刻上完整框架”，而是：

1. 第一阶段先用轻量自定义 UI 打通闭环。
2. 底层统一收口到 `Vercel AI SDK` 与稳定消息协议。
3. 后续在产品交互稳定后，再按需引入 `assistant-ui`。
