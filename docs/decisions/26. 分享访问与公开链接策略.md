# 26. 分享访问与公开链接策略

阅读说明：

1. 本文档负责定义 DocWeave 后续阶段的分享访问、公开链接与只读公开页策略。
2. 本文档聚焦“谁可以分享、分享链接暴露什么、公开访问如何受控”。
3. 认证与权限基线以 [14. 认证与会话实现决策](./14.%20认证与会话实现决策.md) 为准。
4. 文件访问与对象存储边界以 [15. 文件存储与上传策略](./15.%20文件存储与上传策略.md) 为准。

## 1. 文档目的

本文档用于明确 DocWeave 后续引入“文档分享”时的最小策略边界，避免后续出现把业务文档直接暴露成公开对象地址、分享权限绕过工作区权限、分享链接与文档版本语义混乱等问题。

## 2. 核心结论

DocWeave 在分享访问上的正式路线如下：

1. 分享能力作为业务层能力，由 `apps/api` 统一收口。
2. 公开分享页与工作台页面分离，走独立公开路由与公开布局。
3. 第一版分享以“只读公开链接”优先，不直接开放匿名协同编辑。
4. 分享链接访问仍由 `apps/api` 判定分享状态、有效期和目标版本。
5. 分享页引用的文件资源仍通过受控访问链路获取，不直接泄露底层对象存储地址。

## 3. 为什么需要单独定策略

如果不提前收口，后续很容易出现：

1. 直接把文档详情页 URL 当成“分享链接”。
2. 公开分享绕过文档版本与权限边界。
3. 分享页附件、图片、导出文件暴露为长期可猜测地址。
4. 后续一旦需要撤销分享或设置过期时间，现有链接模型难以兼容。

## 4. 分享能力的系统边界

推荐边界如下：

1. `apps/api`
   - 创建、撤销、校验分享链接
   - 返回公开页所需 DTO
2. `apps/web`
   - 渲染公开分享页
   - 不直接承担分享权限真相
3. `apps/collab`
   - 第一阶段默认不参与匿名公开分享
4. `apps/worker`
   - 后续可负责导出分享副本或公开内容预处理

## 5. 第一阶段与后续阶段定位

当前项目第一阶段不强推完整分享体系，但应先固定方向：

1. 第一阶段主闭环仍聚焦登录态内的文档编辑、索引和问答。
2. 分享访问作为后续阶段能力设计，不应反向影响当前认证与协同基线。
3. 现在先把分享能力边界讲清楚，后续落地时实现成本会低很多。

## 6. 推荐分享模型

后续阶段建议围绕独立 `shareId` 建立分享对象，而不是直接公开 `documentId`。

推荐至少包含：

1. `shareId`
2. `documentId`
3. `workspaceId`
4. `createdBy`
5. `accessLevel`
6. `status`
7. `expiresAt`
8. `targetSnapshotVersion` 或等价版本引用

这样做的原因如下：

1. 便于独立撤销、过期、审计和统计。
2. 可以让分享访问与主业务文档 ID 解耦。
3. 可以明确公开访问到底查看哪一版内容。

## 7. 公开链接访问级别

后续建议至少区分以下访问级别：

1. `public_readonly`
2. `workspace_members_readonly`
3. `workspace_members_comment`

第一版推荐只开放：

1. `public_readonly`
2. `workspace_members_readonly`

当前不建议一开始就开放：

1. 匿名编辑
2. 匿名评论
3. 匿名协同 Presence

## 8. 分享页读取的内容真相

公开分享页不应直接消费协同运行态。

推荐原则如下：

1. 分享页优先读取稳定快照或显式发布版本。
2. 不把未稳定的协同实时内容直接暴露给匿名访问。
3. 如果文档后续支持“发布”，分享链接应默认指向已发布版本。

这条原则非常重要，因为它决定：

1. 分享内容是否稳定。
2. 引用和导出是否可追溯。
3. 分享页和 RAG / Citation 是否围绕同一版本语义。

## 9. 与文件资源访问的关系

分享页中的图片、附件和导出文件仍应遵守受控访问策略。

建议如下：

1. 公开文件访问通过 `apps/api` 的分享上下文校验后再签发短期 URL。
2. 不直接把对象存储桶改成公开读。
3. 不把内部 `objectKey` 当作分享主标识暴露给前端。

## 10. 与路由和页面结构的关系

根据当前页面设计，公开分享页应走独立路由与公开布局。

推荐方向如下：

1. 公开分享页挂 `ShareLayout`。
2. 分享访问不复用工作台 Sidebar 和登录态导航。
3. 分享详情页可保留品牌头部、文档标题、作者与更新时间等只读展示信息。

## 11. 审计与撤销要求

分享能力后续至少应支持：

1. 创建分享链接。
2. 查看当前文档已有分享链接。
3. 主动撤销分享链接。
4. 设置过期时间。
5. 标记分享是否启用密码保护或后续附加校验。

第一版即使先不上全部能力，也应保证数据模型与 API 语义允许后续平滑扩展。

## 12. 不建议的做法

需要避免以下高风险方案：

1. 直接公开内部文档页 URL。
2. 直接把对象存储设为公开读来实现分享。
3. 让前端本地决定分享权限与过期判断。
4. 把分享访问和匿名协同编辑捆成同一个能力。

## 13. 当前结论

DocWeave 的分享访问应被理解为一条独立的受控公开链路：

```text
share object
  -> apps/api verify
  -> stable snapshot / published version
  -> public readonly page
```

只要这条链路成立，后续再扩展密码分享、成员限定分享、公开附件访问和分享分析都会顺很多。
