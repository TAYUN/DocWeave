# 11. 状态管理策略

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段前端状态分层与边界。
2. 本文档聚焦“什么状态放哪里”。
3. 当前最终执行口径以 [01. 最终技术方案确认](./01.%20最终技术方案确认.md) 为准。
4. 若需看 API 请求与缓存组织，继续读 [10. 前端数据层与 API 客户端策略](./10.%20前端数据层与%20API%20客户端策略.md)。

## 1. 文档目的

本文档用于明确当前项目第一阶段的前端状态管理边界，避免把协同、编辑器、聊天和普通服务端状态混成一团。

## 2. 核心结论

当前项目状态管理采用分层策略：

1. `TanStack Query` 管服务端状态。
2. `TanStack Router` 管路由与 URL 状态。
3. 轻量客户端共享状态使用本地 store 方案，默认只承载 UI 级共享状态。
4. `packages/editor` 内聚编辑器运行态。
5. 协同连接态与 Presence 靠近协同接入层。
6. 聊天消息运行态由页面局部状态与专用 runtime 承担。

## 3. 为什么必须分层

DocWeave 前端至少同时存在：

1. 当前用户与会话展示态。
2. 空间树与文档元数据。
3. 编辑器局部运行态。
4. 协同连接态。
5. Presence 在线成员。
6. AI 流式消息。
7. 搜索筛选与来源联动。

如果不做分层，后续最容易出现：

1. 全局 store 过重。
2. 缓存失效难以控制。
3. 页面状态和业务状态互相污染。

## 4. TanStack Query 负责什么

负责：

1. 当前用户。
2. 空间列表。
3. 文档树。
4. 文档元数据。
5. 文档成员。
6. 快照与索引状态。
7. 搜索结果。

不负责：

1. 编辑器内部选区状态。
2. 文本输入草稿。
3. 协同运行态正文。
4. 聊天流式 token 追加细节。

## 5. TanStack Router 负责什么

负责：

1. 页面路由。
2. path params。
3. search params。
4. 页面级 filter 与 URL 同步。

建议用路由表达的状态：

1. `documentId`
2. `spaceId`
3. `blockId`
4. 搜索 query
5. 搜索筛选条件

## 6. 轻量客户端共享状态负责什么

建议承载：

1. 侧边栏折叠状态。
2. 面板开关状态。
3. 当前选中的 UI tab。
4. 非 URL 化的页面局部模式状态。

不建议承载：

1. 服务端数据。
2. 聊天消息主状态。
3. 编辑器正文。
4. 协同文档内容。

## 7. 编辑器状态边界

编辑器状态应优先封装在 `packages/editor` 内部。

包括：

1. BlockNote 实例。
2. 当前文档内容绑定。
3. 协同 provider 绑定。
4. 选区与命令状态。

业务页面不应直接把大量编辑器临时状态丢到全局 store。

## 8. 协同状态边界

协同相关运行态应尽量靠近协同接入层：

1. 连接状态。
2. 当前房间信息。
3. 在线成员 awareness。
4. 同步状态。

说明：

1. 这部分更接近实时会话状态。
2. 不应和普通业务 Query 混写。

## 9. 聊天状态边界

聊天消息运行态优先交给页面局部逻辑与后续专用 runtime。

建议：

1. 当前会话消息。
2. 正在生成状态。
3. 停止生成控制器。
4. 来源卡片展示。

不要：

1. 把整段聊天状态塞进全局共享 store。
2. 强行用 Query 承接所有流式 token 状态。

## 10. 第一阶段的状态管理红线

第一阶段必须避免：

1. 一个大 store 管所有状态。
2. 页面里裸写大量共享 mutable state。
3. 把协同正文当普通全局状态存。
4. 把聊天运行态混进业务缓存层。

## 11. 当前结论

当前项目第一阶段状态管理正式确认为：

1. Query 归 Query。
2. Router 归 Router。
3. 客户端共享 store 只做轻量 UI 状态。
4. 编辑器与协同状态内聚。
5. 聊天状态交给专用运行时或页面局部状态。
