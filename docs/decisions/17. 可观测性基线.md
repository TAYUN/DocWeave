# 17. 可观测性基线

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段的可观测性最低基线。
2. 本文档聚焦“日志怎么记、错误怎么分、哪些关键事件必须打点”。
3. 若需看总体架构边界，优先读 [01. 最终技术方案确认](./01.%20最终技术方案确认.md) 与 [01. 整体架构设计](../architecture/01.%20整体架构设计.md)。

## 1. 文档目的

本文档用于把第一阶段的可观测性要求提前收口成统一口径，避免后续出现日志结构混乱、跨服务问题难排查、AI 与索引链路失败后无法定位等问题。

## 2. 核心结论

DocWeave 第一阶段正式采用如下可观测性基线：

1. 统一使用结构化日志，而不是随意 `console.log`。
2. `apps/api`、`apps/collab`、`apps/worker` 输出同一组基础上下文字段。
3. 错误必须带稳定错误码。
4. AI 调用、协同连接、快照/索引任务、权限拒绝必须有明确事件记录。
5. 第一阶段不强求完整分布式追踪，但要保留 `requestId` / `traceId` 语义。

## 3. 统一日志字段

建议所有关键日志至少包含以下字段：

1. `timestamp`
2. `level`
3. `service`
4. `env`
5. `event`
6. `requestId`
7. `traceId`
8. `userId`
9. `workspaceId`
10. `documentId`
11. `jobId`
12. `durationMs`
13. `errorCode`
14. `message`

说明如下：

1. 并非每条日志都必须填满全部业务字段。
2. 但同一条关键链路上的日志应尽量保留可串联上下文。

## 4. 推荐日志形态

第一阶段建议统一采用：

1. 结构化 JSON 日志。
2. 输出到标准输出。
3. 由 Docker / 部署环境统一收集。

不建议第一阶段同时混用多套文本日志风格。

## 5. 错误分级与错误码

### 5.1 推荐级别

建议统一以下级别：

1. `debug`
2. `info`
3. `warn`
4. `error`
5. `fatal`

### 5.2 推荐最小错误码

第一阶段建议至少统一：

1. `validation_failed`
2. `unauthenticated`
3. `forbidden`
4. `rate_limited`
5. `not_found`
6. `conflict`
7. `dependency_failed`
8. `timeout`
9. `internal_error`

## 6. 必须打点的关键链路

### 6.1 `apps/api`

至少记录：

1. 请求入口与耗时。
2. 登录成功与失败。
3. 协同 token 签发成功与拒绝。
4. 编辑器 AI 请求开始、结束、失败。
5. 搜索 / 问答请求开始、结束、失败。

### 6.2 `apps/collab`

至少记录：

1. 房间连接建立。
2. 协同 token 校验失败。
3. 房间解析失败。
4. 连接断开。
5. 文档加载或持久化失败。

### 6.3 `apps/worker`

至少记录：

1. 索引任务创建。
2. 任务开始。
3. 切块完成。
4. embedding 完成。
5. 写入 `Qdrant` 完成。
6. 任务失败或过期。

## 7. AI 与索引专项字段

### 7.1 AI 调用

建议附带：

1. `provider`
2. `model`
3. `action`
4. `promptType`
5. `inputTokens`
6. `outputTokens`
7. `durationMs`

### 7.2 索引任务

建议附带：

1. `snapshotVersion`
2. `targetSnapshotVersion`
3. `jobId`
4. `stage`
5. `chunkCount`
6. `durationMs`

## 8. 协同专项字段

协同日志建议至少带上：

1. `roomName`
2. `workspaceId`
3. `documentId`
4. `userId`
5. `reason`
6. `errorCode`

这样可以保证：

1. 鉴权失败可定位到具体房间和用户。
2. 协同连接断开可关联到具体文档。
3. 后续恢复与持久化问题更容易排查。

## 9. 第一阶段不急于做的内容

当前不建议一开始就重投入：

1. 完整 OpenTelemetry 全链路追踪。
2. 复杂指标平台建模。
3. 统一审计后台。
4. 过细的业务看板。

第一阶段更重要的是：

1. 日志统一。
2. 错误统一。
3. 关键链路失败能查。

## 10. 当前结论

DocWeave 第一阶段的可观测性目标不是“平台多完整”，而是先保证：

1. 关键服务说同一种日志语言。
2. 关键失败有稳定错误码。
3. AI、协同、快照、索引问题都能快速定位。
