# 01. 最终技术方案确认

阅读说明：

1. 本文档负责回答“当前最终拍板是什么”。
2. 若后续新增讨论性或过程性文档，与本文冲突时以本文为准。
3. 若需看模块边界和数据流，继续读 [01. 整体架构设计](../architecture/01.%20整体架构设计.md)。
4. 若需看阶段推进顺序，继续读 [00. 当前实施路线图](../planning/00.%20当前实施路线图.md)。

## 1. 文档目的

本文档用于正式确认 DocWeave 当前阶段的主技术栈、运行时拆分和关键实现路线，作为后续搭建工程骨架、编写代码和扩写专项设计时的统一基线。

本文档优先解决以下问题：

1. 前端、后端、协同、AI、RAG 分别采用什么技术。
2. 哪些方案已经确认，哪些能力要延后。
3. 第一阶段必须坚持哪些边界。
4. 第一阶段要不要一次性把队列、聊天 UI 等基础设施做满。

## 2. 最终结论

当前项目的正式技术方案确定如下。

### 2.1 前端

1. `React SPA`
2. `Vite`
3. `TanStack Router`
4. `TanStack Query`
5. `Tailwind CSS v4`
6. `@mantine/core`
7. `@mantine/hooks`
8. `@mantine/notifications`
9. `lucide-react`

补充说明：

1. 前端采用纯 SPA，不使用 `Next.js` 作为主产品前端形态。
2. `React SPA` 更适合承接 `BlockNote`、`Yjs`、AI 流式交互和长生命周期编辑态。
3. 产品整体 UI 统一采用 `Mantine` 组件体系，`Tailwind CSS v4` 保留用于布局与细节样式微调。

### 2.2 编辑器与协同

1. `@blocknote/core`
2. `@blocknote/react`
3. `@blocknote/mantine`
4. `Yjs`
5. `Hocuspocus`

补充说明：

1. 协同服务必须独立运行，不塞进主业务 API。
2. 编辑器正文协同态与元数据、索引态必须分离。
3. 当产品前端已经全局启用 `Mantine` 时，BlockNote 样式入口应优先采用 `@blocknote/mantine/blocknoteStyles.css`，避免重复引入 `Mantine` 核心样式。

### 2.3 编辑器 AI 与聊天交互

1. 编辑器 AI 前端采用 `@blocknote/xl-ai`
2. 编辑器 AI 后端采用 `@blocknote/xl-ai/server`
3. 模型接入统一采用 `Vercel AI SDK`
4. 聊天 / Copilot 第一阶段先采用轻量自定义 UI
5. 后续按需接入 `assistant-ui`

补充说明：

1. 编辑器 AI 只服务当前文档和当前选区，不等于知识库聊天。
2. 聊天 / Copilot 的 UI 方案不在第一阶段强制落地。
3. `AdonisJS` 在这里承担的是宿主 API 和权限编排职责，不替代 `BlockNote` 官方推荐的 AI backend 生态。
4. 可以保留项目自己的适配边界，但默认实现应优先贴近 `BlockNote` 官方包，而不是先自建同类协议。

### 2.4 后端与业务服务

1. 主业务后端采用 `AdonisJS v7`
2. 协同服务采用独立 `Hocuspocus + Yjs` 进程
3. Worker 采用独立 Node.js 进程
4. 文档服务端处理、快照 / 导出 / RAG 前处理优先采用 `@blocknote/server-util`

补充说明：

1. `apps/api` 负责认证、权限、文档元数据、文件、AI / RAG 编排，并承载 `@blocknote/xl-ai/server` 接入点。
2. `apps/collab` 负责协同房间、Awareness、token 校验接入点。
3. `apps/worker` 负责 embedding、indexing、file tasks 和后续导出任务。
4. 只要场景属于 BlockNote 文档结构处理，应优先复用 `@blocknote/server-util`，避免过早自写快照转换与导出流水线。

### 2.5 数据与基础设施

1. 主业务数据库采用 `PostgreSQL`
2. 向量检索采用 `Qdrant`
3. 缓存与后续队列基线采用 `Redis`
4. 对象存储统一采用 `S3-compatible` 抽象
5. 本地开发默认使用 `MinIO`

补充说明：

1. 明确不采用 `pgvector` 作为当前主向量方案。
2. `Qdrant` 更适合当前项目的 RAG 演进路线和后续检索能力扩展。
3. 生产环境可在 `S3-compatible` 抽象下切换 `S3 / COS / R2 / MinIO`，业务层不感知底层提供商差异。

### 2.6 RAG 与任务调度

1. RAG 由项目内 `packages/rag` 自建
2. `packages/rag` 负责检索、上下文构建和问答流程
3. 第一阶段先保留任务抽象，不强制接入 `BullMQ`
4. 第二阶段再把调度实现替换为 `BullMQ`

补充说明：

1. 第一阶段先写数据库 job 记录和 worker 轮询实现。
2. 任务接口要先稳定，再替换底层调度方式。
3. 这样可以避免项目过早被队列基础设施拖慢主闭环。

## 3. 系统分层确认

当前系统正式采用以下分层：

```text
apps/web
  -> React SPA
  -> 工作台、文档页、聊天页、设置页

apps/api
  -> AdonisJS v7
  -> auth / workspace / space / document / file / ai / rag
  -> host @blocknote/xl-ai/server endpoints

apps/collab
  -> Hocuspocus + Yjs
  -> collaboration room / awareness / collab token verify

apps/worker
  -> snapshot -> chunk -> embedding -> qdrant
  -> file post-process / export / rebuild index

packages/ai
  -> Vercel AI SDK provider 统一接入层
  -> 为 @blocknote/xl-ai/server 与 chat / rag 提供模型能力

packages/rag
  -> 检索、上下文构建、问答流程

packages/document
  -> 基于 @blocknote/server-util 的快照、导出、前处理封装
```

## 4. 第一阶段必须坚持的边界

第一阶段实现时必须遵守以下边界：

1. `apps/web` 不直接访问数据库。
2. `apps/api` 不承载协同 WebSocket。
3. `apps/collab` 不承载普通业务 CRUD。
4. `packages/ai` 统一模型接入，不在业务 controller 里散落 provider SDK 调用。
5. `packages/rag` 只做检索、上下文构建和问答流程，不承担通用模型接入职责。
6. BlockNote AI 后端协议优先复用 `@blocknote/xl-ai/server`，不先自定义一套等价协议。
7. BlockNote 文档服务端处理优先复用 `@blocknote/server-util`，不先自写快照 / 导出基础能力。
8. RAG 只吃稳定快照，不直接吃协同运行态。
9. 聊天 / Copilot 不抢跑第一阶段主闭环。

## 5. 第一阶段策略确认

第一阶段不是“一次把所有基础设施做满”，而是先建立最小闭环。

当前明确采用如下策略：

1. 先打通文档编辑、协同、稳定快照、索引和最小编辑器 AI。
2. `BullMQ` 延后到第二阶段。
3. 聊天 / Copilot UI 延后到主链路稳定后再接。
4. 先保证抽象接口和运行时拆分正确，再逐步扩展体验层。

## 6. 当前结论

当前项目最终正式技术方案确认为：

1. `React SPA + Vite + TanStack Router + TanStack Query`
2. `Tailwind CSS v4 + @mantine/core + @mantine/hooks + @mantine/notifications + lucide-react`
3. `@blocknote/core + @blocknote/react + @blocknote/mantine`
4. 编辑器 AI 前端采用 `@blocknote/xl-ai`
5. `AdonisJS v7`
6. `Hocuspocus + Yjs` 独立协同服务
7. `@blocknote/xl-ai/server + Vercel AI SDK`
8. 自定义轻量聊天 / Copilot UI，后续按需接入 `assistant-ui`
9. `@blocknote/server-util`
10. `packages/rag`
11. `Qdrant`
12. `PostgreSQL`
13. `Redis`
14. `MinIO + S3-compatible abstraction`
15. 独立 `apps/worker`

该方案作为后续工程初始化和第一阶段建设的默认基线。
