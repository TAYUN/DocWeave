# 14. 认证与会话实现决策

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段的认证、会话与权限接入基线。
2. 总体技术路线以 [01. 最终技术方案确认](./01.%20最终技术方案确认.md) 为准。
3. 系统边界以 [01. 整体架构设计](../architecture/01.%20整体架构设计.md) 为准。

## 1. 文档目的

本文档用于明确以下问题：

1. 第一阶段采用什么登录方式。
2. Web、API、Collab 之间如何传递认证与权限语义。
3. 会话存在什么位置，哪些服务信任什么令牌。
4. 编辑器 AI、RAG、协同 token 如何复用统一权限基线。

## 2. 核心结论

当前项目第一阶段认证与会话路线正式确认为：

1. 主认证方式采用账号密码登录。
2. 主业务会话由 `apps/api` 基于 `AdonisJS v7` 统一管理。
3. 前端 `apps/web` 通过带凭据请求复用 API 会话，不自行维护另一套长期可信登录态。
4. `apps/collab` 不直接信任浏览器登录态，而只信任 `apps/api` 签发的短期协同 token。
5. 编辑器 AI、RAG 搜索与问答的权限校验统一落在 `apps/api`。

## 3. 为什么采用这条路线

采用该路线的原因如下：

1. DocWeave 是重业务后端产品，认证、权限、成员、审计都应由主业务 API 收口。
2. `React SPA` 更适合通过带凭据请求复用服务端会话，而不是在多个运行时之间散落长期 token。
3. 协同服务天然是独立进程，必须通过短期 token 做边界隔离，不能直接共享业务会话 cookie。
4. 第一阶段应优先保证权限链路清晰，而不是急于扩展社交登录、多端 SSO 或复杂外部身份体系。

## 4. 当前不采用的主路线

第一阶段明确不采用以下主路线：

1. 社交登录作为主入口。
2. 前端长期保存高权限 bearer token 作为唯一真相。
3. `apps/collab` 直接读取主业务 session 作为连接凭据。
4. 让编辑器 AI 或 RAG 服务自己绕过 API 直接判定业务权限。

## 5. 主认证流程

第一阶段推荐采用以下主流程：

```text
用户在 Web 登录
  -> apps/web 调用 apps/api 登录接口
  -> apps/api 创建会话
  -> 浏览器后续以带凭据请求访问业务 API
  -> apps/api 负责识别当前用户与权限
```

补充约束：

1. `apps/web` 不自行判定真实业务权限。
2. `apps/web` 只持有“当前用户展示态”和必要的前端状态，不把前端状态视为权限真相。

## 6. 协同认证流程

协同链路采用如下模式：

```text
用户打开文档页
  -> apps/web 调用 apps/api 申请 collaboration token
  -> apps/api 校验当前用户是否有目标文档访问权限
  -> apps/api 签发短期 collab token
  -> apps/web 使用 token 连接 apps/collab
  -> apps/collab 校验 token 并建立房间连接
```

约束如下：

1. `apps/collab` 只接受短期协同 token。
2. 协同 token 必须绑定 `workspaceId`、`documentId`、`userId`、`role` 或 `capabilities` 以及过期时间。
3. 协同 token 只作为短期连接凭据，不作为长期登录凭据。

## 7. 编辑器 AI 与 RAG 权限流程

### 7.1 编辑器 AI

流程如下：

```text
用户在文档页触发 AI
  -> apps/web 调用 apps/api
  -> apps/api 读取当前会话
  -> apps/api 校验目标文档权限
  -> apps/api 组装 AI 请求并调用 packages/ai
```

### 7.2 RAG 搜索与问答

流程如下：

```text
用户发起搜索或问答
  -> apps/web 调用 apps/api
  -> apps/api 根据会话计算可见文档范围
  -> packages/rag 只在该范围内检索
  -> packages/ai 生成回答
```

约束如下：

1. `packages/rag` 不直接信任浏览器传来的“可访问文档列表”。
2. 权限范围必须由 `apps/api` 基于业务数据重新计算。

## 8. 权限模型建议

第一阶段建议采用三层权限结构：

1. `workspace role`
2. `space role`
3. `document role`

用途如下：

1. `workspace role` 用于团队成员边界和管理能力。
2. `space role` 用于知识空间范围控制。
3. `document role` 用于例外共享、协同编辑和 AI / RAG 精细控制。

## 9. 会话与安全基线

第一阶段建议同时补齐以下基线：

1. `@adonisjs/session`
2. `@adonisjs/auth`
3. `@adonisjs/shield`
4. `@adonisjs/limiter`
5. `VineJS` 请求校验

建议重点保护以下接口：

1. 登录接口。
2. 协同 token 接口。
3. 编辑器 AI 接口。
4. RAG 搜索与问答接口。
5. 文件上传与签名 URL 接口。

## 10. 第一阶段不做事项

第一阶段明确不做：

1. 社交登录。
2. 复杂组织级 SSO。
3. 多身份提供商切换。
4. 独立 API key 平台化管理。
5. AI 调用额度计费体系。

## 11. 当前结论

DocWeave 第一阶段认证与会话的关键不是“登录方式多不多”，而是先建立一条清晰的可信链路：

```text
apps/api session
  -> business permission truth
  -> collab short-lived token
  -> editor AI / RAG permission gate
```

只要这条链路稳定，后续再扩展社交登录、SSO、分享访问和更复杂的权限矩阵都会顺很多。
