# 23. RAG 检索方案选型

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段 RAG 检索底座与检索分工。
2. 若需看索引任务编排，继续读 [25. 搜索与索引任务编排设计](../architecture/25.%20搜索与索引任务编排设计.md)。
3. 若需看全文兜底策略，继续读 [24. 全文搜索兜底方案](./24.%20全文搜索兜底方案.md)。

## 1. 文档目的

本文档用于明确：

1. 第一阶段语义检索为什么采用 `Qdrant`。
2. RAG 检索与搜索页全文检索如何分工。
3. 为什么第一阶段不把多后端混检做成必选项。

## 2. 核心结论

当前项目第一阶段检索路线正式确认为：

1. 语义检索底座采用 `Qdrant`。
2. 搜索页全文检索兜底由 `PostgreSQL` 承担。
3. RAG 问答与语义召回围绕 `Qdrant` 组织。
4. `packages/rag` 保持 backend 抽象，但第一阶段只落 `Qdrant` 正式实现。

## 3. 为什么选择 Qdrant

原因如下：

1. 更适合当前项目的 RAG 演进路线。
2. payload 过滤和检索模型更适合文档级语义召回。
3. 与独立 worker 和版本化索引链路更容易配合。

## 4. 为什么不让搜索页完全依赖 Qdrant

因为：

1. 标题精确命中不稳定。
2. 术语、编号、路径类搜索结果需要更可解释的全文检索。
3. 向量索引延迟或失败时需要稳定兜底。

## 5. 当前结论

第一阶段 RAG 检索选型正式确认为：

1. `Qdrant` 承担语义检索与问答召回。
2. `PostgreSQL` 承担搜索页全文检索兜底。
3. `packages/rag` 保持后端抽象，但先只稳定一套 `Qdrant` 闭环。
