# 12. ORM 与数据库访问策略

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段数据库访问层的实现风格。
2. 本文档聚焦“什么时候用 ORM，什么时候用 Query Builder / SQL，访问层放在哪里”。
3. 当前最终执行口径以 [01. 最终技术方案确认](./01.%20最终技术方案确认.md) 为准。
4. 若需看数据库领域模型和表设计，继续读 [04. 数据库设计](../architecture/04.%20数据库设计.md)。

## 1. 文档目的

本文档用于明确当前项目第一阶段的数据库访问风格，避免后续在 `AdonisJS`、ORM、原生 SQL 之间反复摇摆。

## 2. 核心结论

当前项目正式采用：

1. `PostgreSQL` 作为主业务数据库。
2. `AdonisJS v7 + Lucid ORM` 作为默认业务访问方式。
3. 复杂查询允许使用 query builder 或原生 SQL。
4. `packages/database` 统一收口数据库访问层。
5. 核心资源默认按软删除语义设计。

## 3. 为什么不纯 ORM

纯 ORM 在以下场景里容易变笨重：

1. 文档树查询。
2. 权限范围查询。
3. 索引状态联表查询。
4. 批量任务查询。

因此，必须允许在复杂查询中使用更直接的 SQL 或 query builder。

## 4. 为什么也不纯 SQL

纯 SQL 会带来：

1. 重复样板。
2. 模型语义丢失。
3. 常规 CRUD 成本升高。
4. 新成员理解成本变高。

因此，常规业务实体仍应优先使用 `Lucid ORM`。

## 5. 推荐分工

### 5.1 Lucid ORM 适用场景

1. 用户。
2. 工作区。
3. 空间。
4. 文档元数据。
5. 成员关系。
6. 配置类表。

### 5.2 Query Builder / SQL 适用场景

1. 文档树。
2. 递归关系查询。
3. 权限范围查询。
4. 索引状态聚合查询。
5. 批量任务查询。

## 6. packages/database 的职责

`packages/database` 负责：

1. 连接配置。
2. migration。
3. seed。
4. transaction 工具。
5. repository / query 访问层。

不建议：

1. 让 `apps/api` 到处直接写数据库细节。
2. 让 controller 里塞大量 SQL。

## 7. 事务策略

推荐原则：

1. 跨多个业务实体写入时明确使用事务。
2. transaction 封装统一放在 `packages/database`。
3. controller 不直接管理复杂事务边界。

## 8. 当前结论

当前项目数据库访问策略正式确认为：

1. 默认 `Lucid ORM`。
2. 复杂查询允许 query builder / SQL。
3. 数据访问统一收口到 `packages/database`。
4. 核心资源默认按软删除语义设计。
