# 22. Embedding 与模型配置策略

阅读说明：

1. 本文档负责定义 embedding 模型选择、配置归属、切换规则与索引兼容性的专项决策。
2. 本文档聚焦“embedding 怎么定、怎么管、变更后系统怎么响应”。
3. 当前最终执行口径以 [01. 最终技术方案确认](./01.%20最终技术方案确认.md) 为准。
4. 若需看 RAG 检索底座和索引流程，继续读 [23. RAG 检索方案选型](./23.%20RAG%20检索方案选型.md)。

## 1. 文档目的

本文档用于明确当前项目中 embedding 模型的选择原则、配置范围、切换策略和索引兼容性约束。

## 2. 核心结论

当前项目第一版对 embedding 的策略如下：

1. 采用“服务端统一配置 + 单知识范围内单一生效 embedding 模型”的策略。
2. embedding 配置由服务端保存，不信任客户端上送的模型参数。
3. embedding 配置一旦变更，对应范围内的索引必须视为待重建。
4. 第一版先保证一套稳定 embedding 闭环，不引入同一知识范围内多模型混用。

## 3. 为什么要单独定义 embedding 策略

如果不单独定义，后续容易出现：

1. 索引数据维度不一致。
2. 同一批文档跨模型生成的向量不可直接比较。
3. 检索效果波动后难以归因。
4. 模型切换后旧索引被错误复用。

## 4. 配置边界

推荐 embedding 配置归属到服务端共享 RAG 设置，而不是前端会话。

建议至少支持：

1. `provider`
2. `model`
3. `baseUrl`
4. `apiKey`
5. `dimensions`
6. `enabled`

## 5. 当前结论

当前项目 embedding 的正确口径不是“前端想传什么模型就传什么”，而是：

```text
server-owned embedding config
  -> single active model per scope
  -> reindex when changed
```
