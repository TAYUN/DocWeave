# 24. 全文搜索兜底方案

阅读说明：

1. 本文档负责定义第一阶段搜索页的全文搜索兜底策略。
2. 本文档聚焦“搜索页全文检索放哪、和 RAG 语义检索怎么分工”。
3. 若需看 RAG 检索底座与快照索引链路，继续读 [23. RAG 检索方案选型](./23.%20RAG%20检索方案选型.md) 和 [25. 搜索与索引任务编排设计](../architecture/25.%20搜索与索引任务编排设计.md)。

## 1. 文档目的

本文档用于补齐当前项目“搜索页到底怎么搜”的明确口径，避免后续出现：

1. 把 `Qdrant` 误当成唯一搜索入口。
2. 搜索页和 RAG 问答共用一条模糊检索链路。
3. 标题词命中和精确术语搜索体验不稳定。
4. 向量索引延迟或失败时，搜索页没有兜底能力。

## 2. 核心结论

第一阶段正式采用如下分工：

1. 搜索页全文搜索兜底由 `PostgreSQL` 承担。
2. RAG / AI 搜索语义召回由 `Qdrant` 承担。
3. 搜索页不依赖 `Qdrant` 才能工作。
4. 所有搜索都必须先经过主业务权限范围计算。

## 3. 为什么全文兜底放在 PostgreSQL

原因如下：

1. `PostgreSQL` 本来就是业务真相源，天然持有标题、稳定正文快照和元数据。
2. 全文检索、高亮摘要、标题加权排序更适合在数据库层稳定实现。
3. 即使 `Qdrant` 尚未完成索引，用户仍然可以进行基础搜索。

## 4. 与 Qdrant 的分工

### 4.1 PostgreSQL 负责什么

1. 标题命中。
2. 正文关键词命中。
3. 高亮摘要。
4. 搜索页基础结果。

### 4.2 Qdrant 负责什么

1. 语义召回。
2. RAG 搜索候选。
3. 问答上下文构建。
4. Citation 对应 chunk 检索。

## 5. 当前结论

当前项目第一阶段全文搜索兜底方案正式确认为：

1. 搜索页全文搜索兜底由 `PostgreSQL` 承担。
2. `Qdrant` 继续承担语义检索与 RAG 问答链路。
3. 搜索页和 RAG 问答不强行混成单一路径。
