# 13. 队列与异步任务策略

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段后台任务与队列执行层的专项决策。
2. 本文档聚焦“异步任务怎么跑、放哪跑、哪些先纳入标准队列”。
3. 当前最终执行口径以 [01. 最终技术方案确认](./01.%20最终技术方案确认.md) 为准。
4. 若需看索引任务链路和快照关系，结合 [25. 搜索与索引任务编排设计](../architecture/25.%20搜索与索引任务编排设计.md) 一起阅读。

## 1. 文档目的

本文档用于明确当前项目第一阶段异步任务的处理方式，重点覆盖 RAG 索引、快照相关任务和后续可扩展的后台任务能力。

## 2. 核心结论

当前项目第一阶段正式采用：

1. 独立 `apps/worker` 作为后台任务执行进程。
2. `PostgreSQL` 中的任务记录作为第一阶段业务任务真相。
3. 数据库轮询作为第一阶段默认调度实现。
4. `Redis + BullMQ` 作为第二阶段标准队列方案。
5. `@adonisjs/limiter` 作为请求入口限流基线。

## 3. 为什么第一阶段不直接上 BullMQ

原因如下：

1. 当前项目第一阶段重点是先打通快照、索引、RAG 和编辑器 AI 主链路。
2. 若过早接入完整队列基础设施，会拖慢主闭环推进。
3. 先稳定任务抽象，再替换调度实现，能降低返工成本。

## 4. 第一阶段任务执行方式

第一阶段推荐如下模式：

1. `apps/api` 创建任务记录。
2. `apps/worker` 轮询读取 `pending` 任务。
3. `apps/worker` 占用任务后执行。
4. 执行完成后回写状态。

## 5. 任务抽象

建议至少统一保留：

```ts
interface IndexJobScheduler {
  enqueueDocumentIndex(documentId: string, snapshotVersion: number): Promise<void>
}
```

要求：

1. API 不直接依赖具体队列实现。
2. Worker 不要求上层必须知道是 DB polling 还是 BullMQ。

## 6. 第一阶段纳入异步执行的任务

必须纳入：

1. `rag.index.document`

建议预留：

1. `document.export`
2. `file.post-process`

## 7. 后续演进路线

第二阶段再替换为：

1. `Redis + BullMQ`
2. 更明确的 retry / concurrency / consumer 策略

但前提是：

1. 任务抽象已经稳定。
2. 主链路已经闭环。

## 8. 当前结论

当前项目第一阶段队列与异步任务策略正式确认为：

1. 先用数据库任务记录 + 独立 worker 轮询。
2. 后续再切 `BullMQ`。
3. 任务抽象先稳定，再替换底层调度。
