# 20. AI SDK 与模型接入策略

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段 AI SDK 与统一模型接入层的专项决策。
2. 若需看编辑器 AI 接入，继续读 [22. BlockNote 服务端能力接入设计](../architecture/22.%20BlockNote%20服务端能力接入设计.md)。

## 1. 核心结论

当前项目第一阶段 AI SDK 路线正式确认为：

1. 标准 AI SDK 采用 `Vercel AI SDK`。
2. 不在第一阶段引入重型 Agent 编排框架作为基础依赖。
3. `packages/ai` 作为统一模型调用封装层。
4. `@blocknote/xl-ai/server` 与 RAG 问答共用统一模型层。

## 2. 为什么采用这条路线

原因如下：

1. 编辑器 AI 和知识库问答都需要流式输出。
2. `Vercel AI SDK` 更适合现代前端 AI 交互与多 provider 抽象。
3. 项目需要统一模型配置、错误处理、重试和流式协议能力。

## 3. 边界

### 3.1 apps/api

负责业务入口、权限校验和请求编排。

### 3.2 packages/ai

负责：

1. provider 配置。
2. runtime 封装。
3. prompt helper。
4. 流式输出能力。

### 3.3 packages/rag

负责检索和上下文构建，不直接承担通用模型接入职责。

## 4. 当前不采用的主路线

第一阶段明确不采用：

1. 各 controller 各自裸调 provider SDK。
2. 让 `packages/rag` 自己承担通用模型接入。
3. 为编辑器 AI 单独再维护一套与聊天完全不同的模型基础设施。

## 5. 当前结论

DocWeave 第一阶段 AI 模型层采用：

```text
Vercel AI SDK
  -> packages/ai
  -> @blocknote/xl-ai/server / rag / chat
```

这样既贴近当前产品需求，也保留了后续扩展 provider 和能力层的空间。
