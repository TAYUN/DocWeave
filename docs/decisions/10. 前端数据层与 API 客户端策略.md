# 10. 前端数据层与 API 客户端策略

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段前端数据获取、缓存、contract 与 API client 的专项决策。
2. 本文档聚焦“前端如何请求 API、如何缓存、如何管理 contract 和 adapter、什么时候再引入更重的生成链路”。
3. 当前最终执行口径以 [01. 最终技术方案确认](./01.%20最终技术方案确认.md) 为准。
4. 若需看前端状态边界，继续读 [11. 状态管理策略](./11.%20状态管理策略.md)。

## 1. 文档目的

本文档用于明确当前项目第一阶段前端数据层的正式方案，解决以下问题：

1. `apps/web` 如何消费 `apps/api`。
2. 数据 cache、失效与重取如何组织。
3. `contract / adapter / view-model` 在前端数据链路中如何分工。
4. 为什么当前采用 `Tuyau + TanStack Query`，而不急于再叠加更复杂生成链路。

## 2. 核心结论

当前项目第一阶段前端数据层正式采用：

1. `TanStack Query` 作为运行时数据获取与缓存层。
2. `Tuyau` 作为 `apps/web` 到 `apps/api` 的类型安全 transport client。
3. `packages/contracts` 作为跨边界 DTO 与输入输出 contract 承载层。
4. `packages/adapters` 作为 `Model -> DTO`、兼容转换与前端 view-model 适配层。
5. 页面层默认优先消费 `view-model`，而不是直接消费底层数据库结构。
6. `apps/api` 请求体验证由 `VineJS` 负责，不与前端共享。

## 3. 为什么采用这套方案

第一阶段的主目标是快速打通业务闭环，而不是先搭建复杂生成链路。

当前 API 仍处于持续演进阶段，因此更适合：

1. 先通过 `Tuyau` 建立类型安全的 transport 调用。
2. 先用 `TanStack Query` 保证缓存、失效和请求状态管理清晰。
3. 先通过 `contract + adapter + view-model` 稳定边界，再考虑后续是否引入更自动化的生成方案。

## 4. Tuyau、contract、adapter、view-model 的分工

### 4.1 Tuyau 负责什么

负责：

1. 前后端类型安全 transport 调用。
2. 让 `apps/web` 消费 `apps/api` 的 route contract 更直接。

不负责：

1. 业务层数据整形。
2. 页面展示模型构建。
3. 替代缓存策略。

### 4.2 packages/contracts 负责什么

负责：

1. 跨边界 DTO。
2. 输入 payload 与响应 payload。
3. 任务 payload 与事件结构。

### 4.3 packages/adapters 负责什么

负责：

1. `Model -> DTO` 适配。
2. 历史兼容结构转换。
3. parse / serialize。
4. 前端 `DTO -> view-model` 过渡封装。

### 4.4 页面层负责什么

页面层负责：

1. 消费 query。
2. 使用 view-model 渲染。
3. 触发 mutation 与 UI 反馈。

## 5. Query key 规范

建议采用稳定、可组合的 query key 结构：

1. `['auth', 'me']`
2. `['workspace', 'list']`
3. `['space', 'tree', { spaceId }]`
4. `['document', { documentId }]`
5. `['document', 'snapshots', { documentId }]`
6. `['document', 'index-status', { documentId }]`
7. `['rag', 'search', { workspaceId, query, filters }]`

要求：

1. 同一资源必须集中定义。
2. 页面内不要随手拼字符串 key。
3. mutation 成功后的失效范围要可推导。

## 6. 请求包装规范

推荐前端数据调用统一处理：

1. 认证凭据。
2. 基础错误包装。
3. 401 / 403 基础处理。
4. `Tuyau` transport 调用边界。
5. DTO 到 view-model 的转换。

不建议：

1. 在页面组件里直接写裸请求。
2. 在页面组件里分散处理 HTTP 错误和结构兼容。

## 7. TanStack Query 的使用边界

`TanStack Query` 负责：

1. 当前用户。
2. 空间列表。
3. 文档树。
4. 文档元数据。
5. 快照与索引状态。
6. 搜索结果。

不负责：

1. 编辑器内部运行态。
2. 协同正文。
3. 瞬时 UI 开关状态。
4. 聊天 token 流拼接过程。

## 8. VineJS 与前端数据层关系

当前项目采用如下分工：

1. `VineJS` 负责 `apps/api` 的 route params、query、body 校验。
2. 前端不直接依赖 `VineJS` 作为请求体验证层。
3. 前端信任的是 API contract 与返回 DTO，而不是后端内部 validator 实现细节。

## 9. 当前结论

当前项目第一阶段前端数据层路线正式确认为：

1. `TanStack Query`
2. `Tuyau`
3. `packages/contracts`
4. `packages/adapters`
5. 页面优先消费 `view-model`
6. 后端请求体验证由 `VineJS` 承担
