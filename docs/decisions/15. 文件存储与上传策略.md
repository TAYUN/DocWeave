# 15. 文件存储与上传策略

阅读说明：

1. 本文档负责定义 DocWeave 第一阶段的文件存储、上传和访问策略。
2. 总体技术路线以 [01. 最终技术方案确认](./01.%20最终技术方案确认.md) 为准。
3. 认证与权限基线以 [14. 认证与会话实现决策](./14.%20认证与会话实现决策.md) 为准。

## 1. 文档目的

本文档用于明确以下问题：

1. 第一阶段文件存储采用什么抽象和提供商。
2. 上传、访问、权限校验分别由哪个运行时负责。
3. 文件在编辑器、导出、RAG 和普通附件场景中如何被引用。
4. 后续切换对象存储提供商时应遵守什么边界。

## 2. 核心结论

当前项目第一阶段文件存储路线正式确认为：

1. 业务文件统一采用 `S3-compatible` 抽象。
2. 本地开发默认使用 `MinIO`。
3. 生产环境在同一抽象下可切换 `S3 / COS / R2 / MinIO`。
4. 文件元数据真相由 `PostgreSQL` 管理。
5. 文件对象内容由对象存储管理。
6. 文件访问优先通过 `apps/api` 校验权限后签发短期 URL。

## 3. 为什么采用这条路线

采用该路线的原因如下：

1. 编辑器图片、附件、导出文件和后续 AI / OCR 中间产物都适合统一走对象存储。
2. `S3-compatible` 抽象最有利于本地开发与云上切换。
3. `MinIO` 非常适合作为本地默认存储服务。
4. 文件权限属于业务系统问题，应由 `apps/api` 统一判定，而不是让前端直接拼对象地址。

## 4. 存储边界

### 4.1 apps/api

负责：

1. 文件上传凭据与元数据管理。
2. 文件归属与权限校验。
3. 签名 URL 签发。
4. 删除、替换、回收和引用校验。

不负责：

1. 实时协同同步。
2. 长时间文件后处理。
3. 向量索引。

### 4.2 apps/worker

负责：

1. 图片后处理。
2. OCR / 文本提取预留。
3. 导出文件生成。
4. 引用文件的后续索引前处理。

### 4.3 apps/web

负责：

1. 发起上传。
2. 展示文件。
3. 在文档和页面中引用文件。

不负责：

1. 直接判定文件可访问性。
2. 长期信任对象地址。

## 5. 文件标识策略

第一阶段建议明确区分：

1. `fileId`
2. `objectKey`

约束如下：

1. 对外主标识使用 `fileId`。
2. `objectKey` 只作为存储层内部定位字段。
3. 前端与业务接口不直接以 `objectKey` 作为公开主标识。

## 6. 文件访问策略

第一阶段默认采用“后端校验权限后签发短期 URL”的模式。

流程如下：

```text
前端请求访问某文件
  -> apps/api 根据当前会话校验文件所属 workspace / document / permission
  -> apps/api 生成短期 signed URL
  -> 前端使用该 URL 访问对象存储
```

这样做的原因：

1. 不需要把对象存储暴露为公开资源。
2. 更容易控制文档附件和私有文件权限。
3. 后续切换存储提供商时，前端不需要重写访问策略。

## 7. 上传策略

第一阶段建议采用“由 API 控制元数据，上传流程可逐步增强”的路线。

推荐顺序如下：

### 7.1 第一阶段最小可用

1. 前端把文件发给 `apps/api`。
2. `apps/api` 完成基本校验。
3. `apps/api` 写入对象存储。
4. `apps/api` 记录文件元数据。

### 7.2 后续可增强

当文件量、体积或体验要求提升后，可演进为：

1. API 先创建上传会话。
2. API 签发直传凭据。
3. 前端直传对象存储。
4. API 再确认文件元数据和归属关系。

## 8. 文件分类建议

第一阶段建议至少区分以下文件类型：

1. 文档内图片。
2. 普通附件。
3. 导出文件。
4. 系统中间产物。

这样做的原因：

1. 不同类型文件的访问策略和生命周期不同。
2. 后续导出、OCR、RAG 附件处理都需要更清晰的文件分类。

## 9. 与编辑器和 RAG 的关系

### 9.1 编辑器

编辑器中的图片或附件引用，建议通过 `fileId` 建立业务关联，而不是把对象地址直接写成长期真相。

### 9.2 RAG

第一阶段 RAG 主输入仍然是稳定快照正文。

补充约束：

1. 文件本身不自动进入 RAG 主链路。
2. 若后续要接 PDF、图片或 Office 文件解析，应由 `apps/worker` 负责提取，再进入专门的索引流程。

## 10. 第一阶段不做事项

第一阶段明确不做：

1. 复杂多段分片上传。
2. 公网匿名大文件分享体系。
3. 多层冷热分级存储。
4. 完整媒体处理流水线。
5. 所有文件类型自动 OCR / 索引。

## 11. 当前结论

DocWeave 第一阶段文件策略的关键不在于“支持多少存储厂商”，而在于先建立一条稳定抽象：

```text
PostgreSQL file metadata
  + S3-compatible object storage
  + apps/api permission-gated signed URL
  + apps/worker post-process
```

只要这条抽象成立，后续切换 `MinIO / S3 / COS / R2`，扩展直传、导出、OCR 和文件索引都会容易很多。
