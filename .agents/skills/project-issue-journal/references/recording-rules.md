# Recording Rules

这份参考用于约束 `项目开发问题记录` 的写法，避免它退化成零散吐槽或流水账。

## 1. 什么值得记录

优先记录下面几类问题：

1. 会影响后续架构、决策、进度或协作方式的问题。
2. 会重复出现的问题。
3. 一开始看起来像局部 bug，后来发现根因更深的问题。
4. 能明显体现个人成长、项目治理演进或团队边界认知变化的问题。

不优先记录下面这些内容：

1. 纯一次性的微小拼写错误。
2. 没有复盘价值的临时操作失误。
3. 只靠一句“已经修了”就说完、没有长期意义的问题。

## 2. 怎样识别“用户没完全意识到的关键点”

重点看这些信号：

1. 用户在重复问同一类边界问题，只是换了说法。
2. 用户在解决表面问题，但真正的根因出在更上层的默认约束、环境基线、流程状态或提交边界。
3. 某个问题虽然暂时能工作，但会持续制造后续成本。
4. 某类问题说明项目缺的是默认规则、自动化脚本或统一入口，而不只是一次性修复。

## 3. 怎样识别“重复性工作”

不要只盯具体动作，要看动作背后的模式。常见模式包括：

1. 重复手动同步说明文档、流程状态和脚本。
2. 重复解释同一套职责边界。
3. 重复排查“不是代码本身坏了，而是环境/数据/流程漂移”的问题。
4. 重复清理 staged 范围、生成产物和杂项改动。
5. 重复补最小校验、兜底错误处理或桥接层逻辑。

## 4. 如何写得对未来有用

每条记录至少回答五件事：

1. 当时发生了什么。
2. 为什么这件事会发生。
3. 它真正影响了什么。
4. 当前是如何理解或处理它的。
5. 未来回头看，这个问题最值得追问的是什么。

## 5. 文风要求

1. 写具体，不写空话。
2. 写问题的“意义”，不只写问题的“表面”。
3. 写成未来可以扩展成复盘文章的素材，而不是即时聊天摘要。
